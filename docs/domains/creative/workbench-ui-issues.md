# 前端界面问题分析与改进方向

> 范围：当前 `frontend/` 运行中的 Creative-first 界面  
> 结论性质：现状审视 + 可执行改进建议  
> 更新日期：2026-04-18

## 1. 总结结论

当前前端已经完成了从 **Task-first** 向 **Creative-first** 的主路径迁移，但整体界面仍然带有明显的“重构过渡期 / 诊断台”痕迹。

核心判断：

> **现在的问题不是缺页面，而是信息架构、页面分层、文案一致性、错误反馈和工作台可用性还没有完全收口。**

也就是说，这一版前端已经“能跑通”，但还没有完全收敛成稳定、清晰、可扩展的产品界面。

---

## 2. 当前界面的主要问题

## 2.1 信息架构仍然依赖解释，而不是结构本身

当前很多页面都在通过 `Alert` 解释：

- 哪个页面是主入口
- 哪个页面是诊断入口
- 哪个页面不该承担主业务语义

这说明用户仍然需要“被教育”才能理解系统，而不是通过导航和页面结构自然理解。

典型位置：

- `frontend/src/pages/Dashboard.tsx`
- `frontend/src/pages/TaskList.tsx`
- `frontend/src/features/creative/pages/CreativeWorkbench.tsx`
- `frontend/src/features/creative/pages/CreativeDetail.tsx`

### 改进方向

- 继续收口导航分层，让结构本身表达职责：
  - 创作工作台
  - 发布与运行
  - 素材
  - 账号
  - 设置
- 减少说明型 `Alert`
- 用更清晰的页面标题、分区和主按钮层级替代“解释文案”

---

## 2.2 业务页混入了太多运行时 / 切换诊断信息

`CreativeDetail` 当前同时承担：

- 作品详情
- 当前版本审核
- 发布池投影
- scheduler cutover 诊断
- shadow diff 原始信息
- 任务诊断跳转

`CreativeWorkbench` 也直接暴露了：

- scheduler mode
- effective mode
- shadow read
- kill switch

### 问题本质

业务视图和工程诊断视图没有彻底分层。

### 改进方向

将页面拆成两层：

### 默认业务层

默认只展示：

- 作品状态
- 当前版本
- 当前审核结论
- 是否可发布
- AIClip / 审核 / 查看详情等主操作

### 高级诊断层

通过独立区域、Drawer、Tab 或“查看运行诊断”入口再展示：

- pool item
- shadow diff
- scheduler mode
- kill switch
- 原始 diff JSON

目标是把“给用户干活的信息”和“给研发排障的信息”分开。

---

## 2.3 Creative Workbench 还不够像真正可用的工作台

当前 Workbench 主要还是：

- 卡片列表
- pool 维度的简单筛选

但还缺少真正支撑日常工作的能力：

- 搜索
- 状态筛选
- 排序
- 分页
- 快速视图

### 风险

作品数量一多，当前界面会很快变成“能浏览，但不方便管理”。

### 改进方向

优先补齐：

- 标题 / 编号搜索
- 状态筛选（待审核 / 已通过 / 需返工 / 已拒绝）
- 排序（最近更新 / 最近失败 / 待处理优先）
- 分页或虚拟滚动
- 常用预设筛选：
  - 待我审核
  - 可发布
  - 生成失败
  - 池中版本不一致

进一步可扩展：

- 表格 / 卡片双视图
- 批量操作
- 保存筛选视图

---

## 2.4 文案语言不统一，且已出现坏文案

当前界面存在明显的中英混搭：

- Layout / Dashboard / Workbench 偏英文
- Detail / Review / Drawer 偏中文
- Auth 页面也偏英文

同时存在至少一处明显坏文案：

- `frontend/src/features/creative/components/VersionPanel.tsx`

其中“审核当前版本”按钮位置出现了异常占位字符，说明文案系统还不够稳定。

### 改进方向

- 冻结一套统一语言策略
- 若主要面向中文用户，前台界面应以中文为主
- 英文仅保留必要技术术语
- 做一次全量文案清洗：
  - 页面标题
  - 按钮
  - 空态
  - 错误提示
  - 导航命名

---

## 2.5 响应式和布局密度控制较弱

当前界面存在多处固定布局：

- 左侧导航固定宽度
- Dashboard 统计卡在窄屏下弹性不足
- Detail 页 Drawer 宽度固定

这在 Electron 非最大化窗口、小屏幕或窄窗口中会降低可用性。

### 改进方向

- 给 Sider 增加 breakpoint / collapse 策略
- 统计卡改成更稳妥的响应式栅格
- 详情页改成 Tabs 或双栏可折叠结构
- Drawer 宽度改成响应式规则，而不是固定值

---

## 2.6 错误态、空态、默认态存在混淆

当前部分查询在失败时会退化成默认值或空数据投影，用户可能无法区分：

- 接口真的返回“空”
- 还是接口失败了

例如：

- Dashboard 中 publish status 有默认 `idle`
- Workbench 对 error 态处理不充分

### 问题本质

错误有时被伪装成“空数据”或“默认状态”。

### 改进方向

所有核心页面统一采用四态：

- loading
- success
- empty
- error

并为关键页面提供：

- 明确错误提示
- 重试按钮
- 必要的局部降级说明

重点页面包括：

- CreativeWorkbench
- CreativeDetail
- Dashboard
- 发布诊断相关区域

---

## 2.7 AIClip workflow 仍偏工具台，而不是产品工作流

当前 AIClip 面板仍保留很多“工具层参数心智”：

- 手工输入路径
- 直接操作底层 clip / audio / cover / pipeline
- 输出路径和目录仍较偏底层

虽然功能上已经接入 Creative 工作流，但使用体验仍然更像“工程工具面板”。

### 改进方向

把 AIClip workflow 继续产品化：

- 用素材选择器替代路径输入
- 增加模板 / 预设
- 用 Stepper 组织流程
- 增加结果预览
- 将“提交为新版本”明确为主 CTA
- 把高级参数折叠进次级区域

---

## 2.8 登录 / 授权界面偏技术系统风格

登录页和授权状态页当前更像内部系统页面：

- `Remote Auth Login`
- 直接展示 `Device ID`
- 多数提示偏技术语义

如果目标用户包括运营或非研发角色，这套界面会显得偏“系统后台”而不是产品入口。

### 改进方向

- 默认隐藏 Device ID 等技术信息
- 把授权失败、失效、grace 模式等文案改成用户可理解的话术
- 把高级诊断信息折叠收起，而不是默认展示

---

## 3. 建议的改进优先级

## P0：应优先处理

1. 统一界面文案语言
2. 把业务页里的过量诊断信息下沉到高级诊断层
3. 补齐 Workbench 的搜索 / 筛选 / 排序 / 分页能力
4. 统一 loading / success / empty / error 四态
5. 修复坏文案与过渡期 Phase 文案

## P1：第二批处理

1. 优化响应式布局
2. 把 Detail 页做成更清晰的分区 / Tab 结构
3. 降低 Dashboard 信息噪音
4. 优化 Auth 页面产品化表达

## P2：体验升级项

1. AIClip workflow 进一步产品化
2. 批量审核 / 批量操作
3. 自定义视图 / 保存筛选器
4. 快捷操作和更顺畅的主路径流转

---

## 4. 一句话结论

如果用一句话总结当前前端状态：

> **当前前端已经完成了 Creative-first 的方向迁移，但还没有彻底从“重构验收台 / 诊断台”收敛成“稳定、清晰、可扩展的产品界面”。**

因此，下一阶段更适合进入：

> **UI / UX 收口阶段**，而不是继续扩页面数量。
