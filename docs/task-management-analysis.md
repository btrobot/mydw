## 一、业务场景重新定义

根据实际业务需求，整个业务流程应该是：

```
原始素材 → 视频合成（扣子工作流） → 成品视频 → 上传到得物
```

**核心认知修正**：
- ❌ 旧理解：任务 = 直接上传现有视频到得物
- ✅ 新理解：任务 = 编排素材 → 调用外部工作流合成视频 → 上传成品到得物

---

## 二、完整业务流程拆解

### 阶段 1: 素材准备（已实现）

**输入**：
- 原始视频片段（videos 表）
- 文案素材（copywritings 表）
- 音频素材（audios 表）
- 封面图片（covers 表）
- 话题标签（topics 表）
- 商品信息（products 表）

**输出**：
- 可用的素材库

**当前状态**：✅ 已实现

---

### 阶段 2: 任务编排（部分实现）

**业务动作**：
1. 选择素材组合（视频 + 文案 + 音频 + 封面 + 话题）
2. 指定上传账号
3. 设置发布策略（时间、优先级）
4. 创建任务记录

**当前实现**：
- ✅ 组装任务（选择视频 + 账号，轮询分配）
- ✅ 关联素材（video_id, copywriting_id, audio_id, cover_id）
- ✅ 关联话题（task_topics 多对多）
- ⚠️ **缺失**：没有"视频合成"这个环节的建模

**问题识别**：
- 当前 Task 表的 `video_id` 指向的是什么？
  - 如果是原始素材 → 需要增加"合成视频"环节
  - 如果是成品视频 → 那原始素材在哪里？

---

### 阶段 3: 视频合成调度（**缺失**）

**业务动作**：
1. 将素材组合发送给扣子工作流
2. 扣子执行视频合成（剪辑、配音、加字幕、加封面）
3. 返回成品视频文件路径
4. 更新任务状态

**需要建模的实体**：
- **CompositionJob（合成任务）**
  - 输入：原始素材 ID 列表
  - 输出：成品视频路径
  - 状态：pending/processing/completed/failed
  - 外部工作流 ID（扣子 workflow_id）
  - 执行日志

**状态流转**：
```
Task.status = draft（草稿）
  ↓ 提交合成
CompositionJob.status = pending → processing → completed
  ↓ 合成完成
Task.status = ready（待上传）
  ↓ 调度上传
Task.status = uploading → uploaded/failed
```

---

### 阶段 4: 上传调度（部分实现）

**业务动作**：
1. 从待上传队列取任务（按优先级、时间窗口、账号限额）
2. 调用 Patchright 自动化上传
3. 记录上传日志
4. 更新任务状态

**当前实现**：
- ✅ 调度器（scheduler.py）
- ✅ 发布服务（publish_service.py）
- ✅ 发布日志（publish_logs 表）
- ✅ 发布配置（publish_config 表）

**问题识别**：
- 当前调度器直接从 `pending` 状态取任务上传
- 没有区分"等待合成"和"等待上传"两个阶段

## 三、业务流程图

```
┌─────────────────────────────────────────────────────────────┐
│                     任务生命周期                              │
└─────────────────────────────────────────────────────────────┘

用户操作                系统状态                  外部系统
    │
    ├─ 创建任务 ────────► draft（草稿）
    │                      │
    │                      │ 编辑素材、配置
    │                      │
    ├─ 提交合成 ────────► composing（合成中）
    │                      │
    │                      ├─ 创建 CompositionJob
    │                      │
    │                      ├─ 调用扣子 API ──────► 扣子工作流
    │                      │                        │
    │                      │                        │ 视频合成
    │                      │                        │
    │                      │ ◄─ Webhook 回调 ──────┤
    │                      │
    │                      ├─ 下载成品视频
    │                      │
    │                      ▼
    │                   ready（待上传）
    │                      │
    │                      │ 进入上传队列
    │                      │
    ├─ 调度器轮询 ────────► uploading（上传中）
    │                      │
    │                      ├─ 检查时间窗口
    │                      │
    │                      ├─ 检查账号限额
    │                      │
    │                      ├─ 调用 Patchright ────► 得物平台
    │                      │                        │
    │                      │                        │ 自动化上传
    │                      │                        │
    │                      │ ◄─ 上传结果 ──────────┤
    │                      │
    │                      ▼
    │                   uploaded（已上传）
    │
    └─ 查看结果 ────────► 完成

失败处理：
    任何阶段 ──► failed（失败）──► 重试 ──► 回到对应阶段
    任何阶段 ──► cancelled（取消）──► 结束
```

## 四、关键问题待确认

### 问题 1: 扣子工作流集成方式

**需要了解**：
1. 扣子 API 文档在哪里？
2. 如何提交任务？（HTTP API / SDK）
3. 如何接收回调？（Webhook / 轮询）
4. 视频文件如何传递？（上传到扣子 / 提供下载链接）
5. 如何查询任务进度？
6. 如何取消正在执行的任务？

**影响范围**：
- CompositionJob 表设计
- CozeClient 实现
- Webhook 端点设计

---

### 问题 2: 当前 Task 表的 video_id 语义

**需要确认**：
- 当前 `Task.video_id` 指向的是：
  - A. 原始视频素材（需要合成）
  - B. 成品视频（已合成完毕）
  - C. 两者都可能（需要增加字段区分）

**建议方案**：
- 新增 `source_video_ids` 字段（JSON array）存储原始素材
- 新增 `final_video_path` 字段存储成品视频路径
- 保留 `video_id` 用于向后兼容（可废弃）

---

### 问题 3: FFmpeg 本地合成

**需要确认**：
- 是否需要支持本地 FFmpeg 合成作为备选方案？
- 还是完全依赖扣子？

**影响**：
- 如果支持本地合成，需要在 WorkflowTemplate 中增加 `workflow_type` 字段
- 本地合成可以复用现有的 `backend/services/ai_clip_service.py`

---

### 问题 4: 素材组合规则

**需要确认**：
1. 一个任务可以使用多个视频片段吗？（如 3 个 10 秒片段合成 30 秒视频）
2. 文案如何应用？（字幕 / 配音 / 描述文字）
3. 音频如何应用？（背景音乐 / 替换原音）
4. 封面如何应用？（视频首帧 / 独立图片）

**影响**：
- Task 表字段设计
- CompositionJob 参数设计
- 前端表单设计
