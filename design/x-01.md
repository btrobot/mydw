    └── PM (任务分配、跨域协调)

  5. 禁止跨域 - 除非获得授权

  - 领导层: PM, Tech Lead
  - 执行层: Frontend Lead, Backend Lead, QA Lead, DevOps
  - 专家层: UI Developer, API Developer, Automation Developer, Test Engineer, Security Expert

  协作流程

  Question → Options → Decision → Draft → Approval

 框架设计

 组织架构

 用户 (Product Owner)
   └── Project Manager (PM)
         └── Tech Lead
               ├── Frontend Lead
               │     └── UI Developer
               ├── Backend Lead
               │     ├── API Developer
               │     └── Automation Developer
               ├── QA Lead
               │     └── Test Engineer
               └── DevOps Engineer

 协作协议

 1. 垂直委托规则

 用户
   └── PM (任务分配、优先级、跨域协调)
         └── Tech Lead (架构、技术决策、代码审查)
               └── Developer (具体实现)

 规则: 复杂决策不能跳过层级，必须层层传达。

 2. 水平协作规则

 ┌────────────────┬──────────────┬────────────────────┐
 │     Agent      │   协作对象   │      协作内容      │
 ├────────────────┼──────────────┼────────────────────┤
 │ Frontend Lead  │ Backend Lead │ API 契约、类型定义 │
 ├────────────────┼──────────────┼────────────────────┤
 │ Backend Lead   │ QA Lead      │ 测试用例、验收标准 │
 ├────────────────┼──────────────┼────────────────────┤
 │ Automation Dev │ Backend Lead │ API 端点、Webhook  │
 ├────────────────┼──────────────┼────────────────────┤
 │ QA Lead        │ DevOps       │ CI/CD 集成         │
 └────────────────┴──────────────┴────────────────────┘

 规则: 水平协作只能协商，不能做绑定决定。

 3. 冲突升级规则

 ┌──────────────┬─────────────────┐
 │   冲突类型   │    升级目标     │
 ├──────────────┼─────────────────┤
 │ 技术架构     │ Tech Lead       │
 ├──────────────┼─────────────────┤
 │ 设计/产品    │ Project Manager │
 ├──────────────┼─────────────────┤
 │ 安全问题     │ Security Expert │
 ├──────────────┼─────────────────┤
 │ 测试标准     │ QA Lead         │
 ├──────────────┼─────────────────┤
 │ 无法达成共识 │ Project Manager │
 └──────────────┴─────────────────┘

 4. 变更传播规则

 当某个变更影响多个域时：
 1. 发起变更的 Agent 通知 PM
 2. PM 协调受影响方
 3. 每方确认变更后实施

 5. 跨域变更禁止

 除非获得 PM 授权，否则：
 - ❌ Frontend 不能修改 Backend 代码
 - ❌ Backend 不能修改 Frontend 代码
 - ❌ Developer 不能修改其他 Developer 负责的文件

 工作流协议

 标准开发流程

 用户请求
     │
     ▼
 ┌─────────────┐
 │ Project     │
 │ Manager     │ ──► 任务分解
 │             │ ──► 分配负责人
 └─────────────┘
     │
     ▼
 ┌─────────────┐
 │ Tech Lead   │ ──► 架构设计
 │             │ ──► API 契约
 └─────────────┘
     │
     ├──► Frontend Lead ──► UI Developer
     │
     ├──► Backend Lead ──► API Developer
     │                 ──► Automation Developer
     │
     └──► QA Lead ──► Test Engineer

 每个任务的协作流程

 1. Question (问询)
    └─► "这个功能需要实现什么？"

 2. Options (选项)
    └─► "方案 A vs 方案 B，各有优缺点"

 3. Decision (决策)
    └─► 用户选择方案

 4. Draft (草案)
    └─► Agent 编写代码草案

 5. Approval (批准)
    └─► "我可以写入这个文件吗？"
    └─► 用户: "可以" / "修改后再给我看"

 冲突解决流程

 冲突发生
     │
     ▼
 ┌─────────────┐
 │ 水平协商    │ ──► 尝试达成共识
 └─────────────┘
     │ 无法解决
     ▼
 ┌─────────────┐
 │ 升级到      │ ──► Tech Lead (技术)
 │ 共同上级    │ ──► PM (非技术)
 └─────────────┘
     │
     ▼
 ┌─────────────┐
 │ 决策文档化  │ ──► 写入 ADR 或决策日志
 └─────────────┘
