# ralplan 新会话启动词

> 用途：在一个全新、干净上下文中启动 `ralplan` 任务  
> 更新时间：2026-04-16

---

## 推荐版本（可直接复制）

```text
$ralplan 基于 refactor-docs 下现有设计资产，制定“实施阶段 1 / 2 / 3”的开发路线图，明确开发顺序、模块拆分、依赖关系、测试策略、风险控制和验收标准。

请以以下文档为输入：
- refactor-docs/短视频作品自动生成与发布系统-新版PRD-基于MVP现状.md
- refactor-docs/作品实体与状态机详细设计.md
- refactor-docs/人工检查流与版本管理设计.md
- refactor-docs/作品工作台页面原型与交互流转.md
- refactor-docs/发布池与调度器重构说明.md
- refactor-docs/AIClip 工作流化改造设计.md
- refactor-docs/架构与工程约束.md
- refactor-docs/渐进式重建实施方案.md
- docs/coding-standards.md

请注意以下约束：
1. 这里的“实施阶段 1 / 2 / 3”是新的实施分期，不是历史阶段编号。
2. 总体路线采用“新建目录 + 渐进替换 + 复用已有底层能力”，不是纯原地重构，也不是纯推倒重写。
3. 本地业务后端负责业务执行，产品控制后台逐步承接关键商业控制点。
4. 产品主视图逐步从 Task 迁移到 Creative / 作品工作台，但 Task 继续保留为执行承载对象。
5. 需要给出分阶段交付建议，并明确每阶段的目标、依赖、风险、验收标准。

输出要求：
- 给出“实施阶段 1 / 2 / 3”的建议范围
- 每阶段列出：目标、核心改动、依赖前置、主要风险、测试重点、完成标准
- 标出哪些模块必须复用，哪些模块建议新建
- 标出哪些工作适合先做，哪些必须后做
- 最后给出推荐的实际开发顺序
```

## 极简版本（如果你想要最小上下文）

```text
$ralplan 基于 refactor-docs 下的 PRD、详细设计文档、架构与工程约束、渐进式重建实施方案，制定“实施阶段 1 / 2 / 3”的开发路线图。

重点约束：
- 采用“新建目录 + 渐进替换 + 复用已有底层能力”路线
- 本地业务后端负责业务执行，产品控制后台逐步承接关键商业控制点
- 产品主视图迁移到 Creative / 作品工作台
- Task 保留为执行承载对象
- 输出每阶段的目标、依赖、风险、测试重点和完成标准
```

## 使用建议

### 什么时候用这个版本

适合：
- 你想在全新会话里启动 `ralplan`
- 你不想带入太大的历史上下文
- 你已经有完整设计包，希望直接进入实施规划

### 推荐做法

- 优先使用“推荐版本”
- 如果你只想要最小上下文，使用“极简版本”
- 如需更高确定性，可在启动后补一句：

```text
如果文档之间存在冲突，请以新版 PRD 和《架构与工程约束》为高优先级基线，并明确指出冲突点。
```

---

## 一句话说明

> 这个启动词的目标不是重新定义产品，而是让 `ralplan` 在干净上下文里，基于现有设计资产输出可执行的实施路线图。
