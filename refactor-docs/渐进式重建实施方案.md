## 1. 文档目标

本文档用于回答一个关键实施问题：

> 当前系统下一阶段应采用“原地重构”还是“新建目录重建”的路线？

本文档给出结论、判断依据、实施边界和阶段计划，作为后续开发排期与任务拆解的执行基线。

---

## 2. 结论先行

### 最终建议

> **采用“新建目录 + 渐进替换 + 复用已有底层能力”的渐进式重建路线。**

这不是纯原地重构，也不是推倒重写，而是：

- **在现有系统内新建“作品域”与新页面**；
- **保留现有 MVP 继续可用**；
- **复用现有 auth、调度、发布、解析、FFmpeg、自动化等底层能力**；
- **逐步把业务主入口从 Task 迁移到 Creative / 作品工作台**；
- **最后再收缩旧页面和旧语义。**

### 一句话解释

> **不是重做一个新系统，而是在现有系统里重建新的业务主域，并逐步替换旧主域。**

---

## 3. 为什么不建议纯原地重构

纯原地重构的问题在于：

1. **旧语义和新语义会纠缠**
   - 当前 `Task` 既承担执行，又被当成业务主对象使用。
   - 如果继续在旧结构上叠加，很容易越改越乱。

2. **回归风险高**
   - 当前系统已经有：
     - 授权门禁
     - 商品解析
     - 任务组装
     - 合成提交
     - 调度发布
   - 大量原地改动容易把已有可用链路改坏。

3. **难以控制迁移边界**
   - UI、API、状态机、执行器同时原地改，容易导致阶段目标不清。

---

## 4. 为什么不建议纯重写

纯重写的问题在于：

1. **会浪费现有可用能力**
   - 本地 auth 接入
   - 产品控制后台对接
   - Scheduler
   - PublishService
   - ProductParseService
   - AIClipService

2. **交付周期太长**
   - 纯重写意味着重新打通从素材到发布的完整闭环。

3. **容易脱离真实业务约束**
   - 当前 MVP 已经暴露出很多真实边界，比如：
     - direct publish 约束
     - topic 注入边界
     - auth gate
     - ScheduleConfig 真相源
   - 纯重写很容易在纸面上“更优雅”，但忽略已有运行事实。

---

## 5. 为什么选择“渐进式重建”

这种路线的核心优势是：

### 5.1 保留可运行能力

现有系统仍可继续：

- 创建任务
- 解析商品
- 合成视频
- 调度发布

不会因为新设计阶段而失去可用性。

### 5.2 为新业务域腾出干净空间

可以直接新建：

- 作品实体
- 版本管理
- 人工检查流
- 发布池
- 作品工作台

避免继续污染旧 Task 主视图。

### 5.3 复用底层能力，重建上层语义

复用已有：

- auth
- scheduler
- publish
- product parse
- ai clip 底层能力

重建新的：

- Creative 域模型
- 作品页面
- 检查流
- 发布池

这是一条投入产出比最高的路线。

---

## 6. 实施总原则

## 6.1 原则一：新业务语义走新目录

新增“作品域”相关能力时，不要继续往旧 Task-only 结构上叠。

建议：

- 新 API 走新 route
- 新 service 走新模块
- 新页面走新页面路径
- 新状态机走新领域对象

## 6.2 原则二：底层执行能力优先复用

以下能力原则上不重写：

- `AuthService` / 本地授权门禁
- `PublishService`
- `TaskScheduler`
- `ProductParseService`
- `AIClipService`
- `MediaStorageService`

## 6.3 原则三：旧入口先保留，后下沉

旧入口在迁移期仍保留：

- `TaskList`
- `TaskDetail`
- `TaskCreate`

但定位逐步改成：

- 执行视图
- 诊断视图
- 兼容入口

## 6.4 原则四：每一阶段都必须可运行

每个阶段完成后，系统都必须满足：

- 能启动
- 关键链路不回退
- 新旧页面能共存
- 数据可以解释

---

## 7. 目录级改造建议

## 7.1 本地后端建议新增目录/模块

建议在现有 `backend/` 中新增“作品域”模块，而不是另起一个新 backend 项目。

### 推荐新增

```text
backend/
  api/
    creative.py
    creative_review.py
    creative_publish_pool.py
  services/
    creative_service.py
    creative_version_service.py
    creative_review_service.py
    publish_pool_service.py
    ai_clip_workflow_service.py
```

### 原则

- 新业务域用新模块
- 旧执行能力尽量复用
- 兼容层尽量薄

## 7.2 前端建议新增目录

建议在现有 `frontend/src/` 中新增功能域，而不是另起一个新前端。

### 推荐新增

```text
frontend/src/
  features/
    creative/
      pages/
      components/
      hooks/
      types/
```

### 推荐新增页面

- `CreativeWorkbench.tsx`
- `CreativeDetail.tsx`
- `CreativeVersionPanel.tsx`
- `CreativeCheckDrawer.tsx`
- `PublishPoolView.tsx`（后续可选）

---

## 8. 哪些必须复用，哪些建议新建

## 8.1 必须复用的能力

这些属于底层执行或基础设施，不建议重写：

| 模块 | 原因 |
|------|------|
| 本地授权门禁 | 已与产品控制后台联动，改动风险高 |
| Scheduler | 已有 runtime truth 和 auth gate |
| PublishService | 已打通发布链路 |
| ProductParseService | 已打通商品解析到素材库 |
| AIClipService | 已有稳定的媒体处理底层能力 |
| MediaStorageService | 已承担文件存储语义 |

## 8.2 建议新建的能力

这些属于新的业务语义层，建议新建：

| 模块 | 原因 |
|------|------|
| CreativeItem / Version / CheckRecord / PackageRecord | 新主域模型 |
| 作品工作台 | 新主业务入口 |
| 人工检查流 | 当前系统缺失 |
| 发布池 | 当前调度模型缺失业务候选层 |
| AIClipWorkflowService | 需要把工具能力升级为工作流能力 |

## 9. 分阶段实施方案

## Phase 1：建立作品域最小骨架

### 目标

在不改坏现有系统的前提下，把“作品”作为新业务主域引入。

### 建议交付

1. 新增 `creative_items`
2. 新增 `creative_versions`
3. 新增最小作品 API
4. Task 创建时可选映射到 Creative
5. 新增作品工作台路由与空壳页面

### 完成标准

- 新旧任务系统共存
- 至少可以从任务映射到作品
- 作品有最小列表和详情入口

---

## Phase 2：接入人工检查与版本管理

### 目标

让“作品”具备真实业务判断能力。

### 建议交付

1. 新增 `check_records`
2. 接入 `APPROVED / REWORK_REQUIRED / REJECTED`
3. 作品详情页支持人工检查
4. 影响最终内容的修改创建新版本
5. 版本历史面板可查看

### 完成标准

- 作品检查流可跑通
- 版本变更可追踪
- 当前版本切换逻辑明确

---

## Phase 3：接入 AIClip 工作流化能力

### 目标

让 AIClip 不再只是工具页，而是作品生成链一部分。

### 建议交付

1. 新增 `AIClipWorkflowService`
2. 作品详情页支持“重新合成视频”
3. AIClip 生成成功后创建新版本
4. 生成成功后自动进入检查流

### 完成标准

- AIClip 可以围绕作品运行
- 输出能回写版本与成品包

---

## Phase 4：引入发布池

### 目标

把“可发布候选集”从 Task.ready 中解耦出来。

### 建议交付

1. 新增 `publish_pool_items`
2. 检查通过后自动入池
3. 返工/新版本后旧池项失效
4. 工作台可筛选“可发布作品池”

### 完成标准

- 发布池和作品状态联动
- 业务上“能发什么”已不再直接等于 Task.ready

---

## Phase 5：调度器切换到发布池驱动

### 目标

完成从“选 Task”到“选作品版本”的关键升级。

### 建议交付

1. 调度器先查发布池
2. 选中池项后再映射为 publish Task
3. 发布结果回写到：
   - pool item
   - creative item/version
   - task

### 完成标准

- 调度器主候选集来自发布池
- Task 下沉为执行承载对象

---

## Phase 6：旧页面与旧语义下沉

### 目标

让新作品流成为默认主入口，旧 Task-only 视图退居次要角色。

### 建议交付

1. 任务列表改名为执行列表（可后续）
2. 作品工作台成为主业务入口
3. TaskDetail 明确为执行诊断页
4. 逐步减少旧字段和旧入口的业务含义

### 完成标准

- 操作者主要围绕作品工作台工作
- Task 主要服务执行和诊断

---

## 10. 风险清单

## 风险 1：新旧语义同时存在导致混乱

### 应对

- 文档先行
- 页面命名明确
- 新页面一律围绕“作品”命名
- 旧页面明确标注为“任务/执行视图”

## 风险 2：重复造轮子

### 应对

- 明确“必须复用清单”
- 新 service 只做上层编排，不重写底层能力

## 风险 3：迁移太慢，长期双轨

### 应对

- 每阶段都有明确退出条件
- 到 Phase 5 后必须切换主入口

## 风险 4：调度器切换时回归风险高

### 应对

- 先引入发布池，再改调度器
- 调度器切换前保留兼容开关/观察期

---

## 11. 不变量

实施过程中必须坚持以下不变量：

1. 本地后端 = 业务执行内核
2. 产品控制后台 = 商业控制与治理内核
3. 新业务主对象是作品，不是 Task
4. Task 继续保留，但主要承担执行职责
5. 每阶段都必须可运行、可回退、可验证
6. 不允许因为“重建”而丢失现有可用底层能力

---

## 12. 推荐实施方式

按 OMX 最佳实践，建议现在采用：

> **设计先行 + 渐进实施 + 每阶段验证**

也就是说：

- 先有 PRD 和详细设计包
- 再按阶段实施
- 每完成一阶段都做一次验证和收口
- 不要一口气全量切换

---

## 13. 一句话定义

> 这条路线不是“重写一套新系统”，而是“在现有系统内重建新的作品主域，并用渐进替换的方式接管旧任务主域”，从而以最低风险完成产品升级。
