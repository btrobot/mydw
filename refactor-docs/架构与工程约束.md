## 1. 文档定位

本文档回答的不是“做什么”，而是：

- 架构必须怎么分层；
- 技术栈允许用什么、不允许随意换什么；
- 模块边界如何划分；
- 本地与远端谁是哪个领域的真相源；
- 工程实现必须遵守哪些共识。

### 与其他文档的分工

- **PRD**：定义产品目标和业务流程
- **详细设计文档**：定义模块、实体、状态机和页面流转
- **本文件**：定义跨文档、跨模块都必须遵守的硬约束
- **AGENTS.md**：对 agent 的执行约束
- **ADR**：记录关键架构决策及原因

### 术语映射

为避免实现术语污染产品与架构表达，本文档统一采用以下称呼：

| 推荐术语 | 当前实现映射 |
|----------|--------------|
| 产品控制后台 | `remote/remote-backend` |
| 产品管理后台 | `remote/remote-admin` |
| 本地业务后端 | `backend/` |
| 桌面工作台 | `frontend/` + Electron |

说明：

- `remote` 只保留为当前实现目录和历史术语，不作为本文档的主表达。
- 在产品、架构、PRD 与详细设计中，默认使用“产品控制后台 / 产品管理后台”。

## 2. 核心架构原则

## 2.1 双后端原则

本系统采用双后端架构：

- **本地后端** = 业务执行内核
- **产品控制后台** = 产品控制内核

### 本地后端职责

- 商品/素材/作品/任务/发布池数据管理
- 素材下载与存储
- 本地 SQLite 持久化
- FFmpeg 媒体处理
- 浏览器自动化发布
- 调度器运行
- 本地运行时状态与日志

### 产品控制后台职责

- 用户登录与授权
- License / entitlement / device / session
- 最低版本门禁
- 后台管理与审计
- 后续产品控制与商业化扩展能力

### 禁止事项

- 禁止把本地强依赖文件/执行环境的重业务流程直接迁移为产品控制后台主执行流程
- 禁止把产品控制后台控制状态在本地重新发明成另一套产品语义

---

## 2.2 Electron 优先原则

本项目是 **Desktop-first connected product**，不是纯 Web SaaS。

### 必须遵守

- Electron Renderer 只负责 UI 与交互，不直接承担业务执行逻辑
- Electron 前台统一通过本地 API 访问业务能力
- Electron Main 负责应用生命周期、窗口、托盘、启动协议，不承载业务编排

### 禁止事项

- 禁止在 Renderer 中直接拼接文件系统、FFmpeg、浏览器自动化逻辑
- 禁止在 Renderer 中直接依赖产品控制后台 API 作为主业务入口，除非该能力本来就属于 control plane

---

## 2.3 真相源单一原则（Single Source of Truth）

每类核心状态必须有唯一真相源。

### 当前必须坚持的真相边界

| 领域 | 真相源 |
|------|--------|
| 调度配置 | `ScheduleConfig` |
| 调度运行状态 | `TaskScheduler runtime state` |
| Task 资源集合语义 | `task_videos / task_copywritings / task_covers / task_audios / task_topics` |
| 本地机器授权运行态 | local machine-session truth |
| 用户/License/设备/版本门禁 | 产品控制后台 |
| profile-level default topics | `publish_profile_topics` |

### 禁止事项

- 禁止为同一概念同时维护两套平级真相
- 禁止新页面绕开 canonical API 读旧字段
- 禁止把兼容字段重新提升为主语义来源

---

## 2.4 作品优先原则

从下一阶段开始，产品主视图以 **作品（CreativeItem）** 为中心，Task 下沉为执行对象。

### 必须遵守

- 新增业务页面优先围绕作品层设计
- 人工检查、版本管理、发布池都围绕作品和版本展开
- Task 主要用于执行过程、失败原因、调试与运行记录

### 禁止事项

- 禁止继续把 Task 当成唯一产品主对象扩展业务语义
- 禁止把版本管理、人工检查结果直接塞进 Task 临时字段替代作品层设计

---

## 2.5 工作流闭环原则

每条作品必须能走完以下闭环：

```text
来源准备 → 生成 → 人工检查 → 可发布 → 发布执行 → 结果回写
```

### 必须遵守

- 影响最终内容的修改必须生成新版本
- 检查永远针对具体版本
- 发布永远针对具体版本
- 发布前必须有完整性校验

## 3. 技术栈约束

## 3.1 前端技术栈

### 当前受约束技术栈

- Electron
- React
- TypeScript（strict）
- Ant Design
- React Query
- Vite

### 约束

- 前端页面继续使用 React 函数组件
- 数据请求统一通过 hooks + API client
- 不新增平行状态管理体系，保持轻量，优先 React Query 与现有模式

### 禁止事项

- 禁止无明确理由引入第二套 UI 框架
- 禁止新代码大量使用 `any`
- 禁止跳过生成的 API client 直接随意拼接请求，除非明确处于兼容层或工具层

---

## 3.2 本地后端技术栈

### 当前受约束技术栈

- FastAPI
- SQLAlchemy
- SQLite
- Pydantic v2
- Loguru
- FFmpeg
- Patchright / Playwright

### 约束

- 路由层保持 thin controller
- 业务逻辑进入 services
- DTO/请求响应结构进入 schemas
- 数据模型统一走 SQLAlchemy models

### 禁止事项

- 禁止在 API route 中堆砌复杂业务逻辑
- 禁止在 service 内部直接返回未约束的任意结构替代 schema 契约
- 禁止随意引入第二套 ORM / 第二套 Web 框架

---

## 3.3 产品控制后台技术栈

### 当前受约束技术栈

- FastAPI
- SQLAlchemy
- 产品管理后台前后端分离

### 约束

- 产品控制后台保持 control plane 职责，不承接本地业务执行
- auth / admin / self-service 路由边界清晰
- 错误码、契约、版本门禁语义保持稳定

### 禁止事项

- 禁止把 local business workflow 直接塞入产品控制后台
- 禁止产品控制后台接口错误码语义漂移

---

## 4. 模块边界约束

## 4.1 页面层

页面层负责：

- 展示状态
- 收集输入
- 触发操作
- 组合现有 hooks

页面层不负责：

- 执行业务规则
- 持久化逻辑
- 直接访问底层执行器

## 4.2 Hook / Client 层

负责：

- 封装 API 调用
- 管理查询与 mutation
- 统一错误处理入口

不负责：

- 复杂业务编排
- 规则决策

## 4.3 Service 层

负责：

- 业务编排
- 状态流转
- 真相源读写
- 兼容边界收口

不负责：

- 页面展示逻辑
- UI 文案耦合

## 4.4 Model / Schema 层

- Model 负责持久化结构
- Schema 负责 API contract
- 禁止用 ORM 模型直接替代 API contract

## 5. 数据与状态约束

## 5.1 Task 语义约束

- `Task` authoritative 模型为资源集合模型
- `video_ids / copywriting_ids / cover_ids / audio_ids / topic_ids` 为主语义
- 旧 FK 和 legacy JSON 字段只能用于兼容，不得重新定义主行为

## 5.2 Topic 语义约束

当前自动注入 task assembly 的默认话题来源只有：

- `PublishProfile` 默认话题

当前不会自动注入：

- global topics
- topic groups

## 5.3 Direct Publish 约束

当前 direct publish 必须坚持保守语义：

- 1 个最终视频
- 0/1 个文案
- 0/1 个封面
- 独立音频不得直接发布
- 多视频、多文案、多封面不能静默降级

## 5.4 作品版本约束

- 每次影响最终内容的修改都必须生成新版本
- 人工检查永远针对版本
- 发布永远绑定版本
- 当前版本变化后，旧版本的“可发布结论”不自动继承

---

## 6. 发布与调度约束

## 6.1 调度配置约束

- 调度配置 canonical source 必须是 `ScheduleConfig`
- 页面与调度器都必须围绕 canonical API / model 读写

## 6.2 调度运行约束

- 调度器运行状态以 `TaskScheduler runtime state` 为准
- 新增发布池后，调度器优先基于“可发布作品版本”而不是直接扫 ready task

## 6.3 发布池约束

- 只有经过人工检查批准的当前版本才能进入发布池
- 发布池是业务候选集，不是执行日志表
- 调度器负责从池中选择，发布服务负责执行上传

---

## 7. 授权与商业控制约束

## 7.1 基本原则

- 本地后端负责业务执行
- 产品控制后台逐步承接关键商业控制点

### 当前必须由产品控制后台控制的能力

- 登录 / refresh / logout / me
- 用户 / 设备 / session / audit
- 最低版本门禁

### 后续建议逐步上移的能力

- VIP / entitlement 能力门禁
- 批量能力上限
- 高级模板 / 工作流 / 配置下发
- License 套餐与灰度控制

## 7.2 禁止事项

- 禁止把所有关键商业控制都只放在本地
- 禁止让桌面端在无约束情况下长期脱离产品控制后台完整运行
- 禁止为商业控制写死不可演进的本地分支逻辑

---

## 8. 工程实现约束

## 8.1 依赖约束

- 无明确必要，不新增依赖
- 引入新依赖必须说明用途、替代方案、影响范围
- 优先复用现有工具与服务

## 8.2 日志与错误处理约束

- 后端统一使用 loguru，不使用随意 `print`
- 错误码与错误语义保持稳定
- 对用户可见错误与内部日志错误要分层

## 8.3 配置约束

- 配置统一进入 config / settings 系统
- 禁止散落硬编码路径、端口、密钥、产品控制后台地址

## 8.4 文件与目录约束

- 页面在 `frontend/src/pages`
- 共享组件在 `frontend/src/components`
- hooks 在 `frontend/src/hooks`
- 后端路由在 `backend/api`
- 后端业务逻辑在 `backend/services`
- 数据模型在 `backend/models`
- schema 在 `backend/schemas`

### 禁止事项

- 禁止页面直接调用 service
- 禁止 service 反向依赖页面/前端类型
- 禁止在多个目录重复定义相同领域概念
