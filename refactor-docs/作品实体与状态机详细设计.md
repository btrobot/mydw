## 1. 文档目标

本文档用于把新版 PRD 中提出的“作品层”落成可实现的领域设计，回答以下问题：

1. 为什么当前 `Task` 不足以承担产品主对象。
2. 新增的 `CreativeItem / CreativeVersion / PackageRecord / CheckRecord` 分别负责什么。
3. 作品状态机如何与当前 `Task.status` 执行状态机协同。
4. 本地后端应该新增哪些表、字段、接口和约束。
5. 如何在不推翻现有 MVP 的前提下平滑演进。

---

## 2. 设计结论先行

### 2.1 核心结论

建议新增一层 **作品域模型（Creative Domain）**，并明确分层：

- **CreativeItem**：业务主实体，面向桌面工作台展示。
- **CreativeVersion**：作品某一版本的具体内容快照。
- **PackageRecord**：该版本对应的标准成品包记录。
- **CheckRecord**：人工检查动作和结论。
- **Task**：执行承载对象，负责合成/发布等后台执行状态。

### 2.2 一句话解释

- **作品** 决定“这条内容现在是否可发、是否满意、是否还要改”。
- **任务** 决定“后台现在有没有在跑、跑到了哪、为什么失败”。

## 3. 为什么必须新增作品实体

当前 `Task` 已经能承担：

- 素材集合关联
- 合成状态
- 发布状态
- 调度执行

但仍然不适合作为产品主对象，原因有三点：

### 3.1 `Task` 是执行对象，不是内容对象

当前 `TaskResponse` 以资源集合和执行状态为中心，适合：

- 创建任务
- 提交合成
- 调度上传
- 记录失败

但不适合：

- 预览作品
- 做版本切换
- 记录人工检查意见
- 明确“哪个版本最终被确认可发布”

### 3.2 一个作品可能经历多次执行

同一条作品在真实使用中会经历：

- 第一次生成
- 重生成文案
- 重合成视频
- 替换封面
- 再次发布

如果全部都压在一个 `Task` 上，会导致：

- 业务语义和执行语义纠缠；
- 很难看清“当前最终版本”；
- 很难做版本追溯和回退。

### 3.3 桌面工作台真正面向的是作品

操作者在桌面端想看的是：

- 作品预览
- 当前文案
- 当前封面
- 当前状态
- 是否可发布
- 为什么失败

而不是一堆执行字段。

因此必须引入“作品层”。

## 4. 目标领域模型

```text
Product / Material Pack
   ↓
素材 Asset（video / cover / copywriting / audio / topic）
   ↓
CreativeItem（作品主实体）
   ↓
CreativeVersion（版本快照）
   ↓
PackageRecord（成品包）
   ↓
Task（合成/发布执行）
   ↓
PublishResult / Logs
```

## 5. 实体设计

## 5.1 CreativeItem（作品主实体）

### 定位

表示“一个待处理/待检查/待发布的作品单元”。

### 建议字段

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | 主键 |
| creative_no | varchar | 作品编号 |
| source_type | varchar | 来源：product/manual/mixed |
| source_product_id | bigint nullable | 来源商品 |
| current_version_id | bigint nullable | 当前生效版本 |
| latest_version_no | int | 当前最新版本号 |
| status | varchar | 作品状态 |
| owner_account_id | bigint nullable | 默认发布账号 |
| profile_id | bigint nullable | 作品生产配置档 |
| title | varchar nullable | 当前展示标题 |
| summary | text nullable | 当前摘要/文案预览 |
| last_error_code | varchar nullable | 最近失败码 |
| last_error_message | text nullable | 最近失败信息 |
| last_generated_at | datetime nullable | 最近生成时间 |
| last_checked_at | datetime nullable | 最近检查时间 |
| last_published_at | datetime nullable | 最近发布时间 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

### 职责

- 面向 UI 的主列表对象
- 聚合“当前版本”的业务展示字段
- 提供状态流转总入口
- 持有当前主版本指针

---

## 5.2 CreativeVersion（作品版本快照）

### 定位

表示某次生成/修改后的完整内容快照。

### 为什么需要快照

因为一条作品的“视频、封面、文案、话题”会不断变化，必须能回答：

- 当前版本是什么？
- 上一个版本是什么？
- 哪次改动导致可发布？
- 发布时用的是哪个版本？

### 建议字段

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | 主键 |
| creative_item_id | bigint | 所属作品 |
| version_no | int | 版本号，从 1 开始 |
| version_type | varchar | generated/manual_edit/regenerated |
| title | varchar nullable | 当时标题 |
| copywriting_text | text nullable | 当时主文案 |
| cover_asset_id | bigint nullable | 当时封面 |
| video_asset_id | bigint nullable | 当时最终视频 |
| audio_asset_id | bigint nullable | 当时音频 |
| topic_ids_json | json/text | 当时话题列表快照 |
| source_material_snapshot | json | 来源素材快照 |
| package_record_id | bigint nullable | 对应成品包 |
| created_by | varchar nullable | system / operator |
| created_reason | varchar nullable | 初次生成/重生成文案/重合成等 |
| created_at | datetime | 创建时间 |

### 职责

- 固化一条作品在某时刻的完整内容
- 支撑版本回放和比较
- 为检查和发布提供精确输入

---

## 5.3 PackageRecord（成品包记录）

### 定位

表示某个版本的标准成品输出结果。

### 建议字段

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | 主键 |
| creative_version_id | bigint | 所属版本 |
| package_status | varchar | building/ready/failed |
| manifest_json | json | 成品结构描述 |
| video_path | varchar nullable | 最终视频路径 |
| cover_path | varchar nullable | 封面路径 |
| title_path | varchar nullable | 商品名称/标题文件路径 |
| copywriting_path | varchar nullable | 文案文件路径 |
| preview_path | varchar nullable | 预览图/GIF |
| completeness_status | varchar | complete/incomplete |
| completeness_message | text nullable | 缺失说明 |
| built_at | datetime nullable | 构建完成时间 |
| created_at | datetime | 创建时间 |

### 职责

- 标准化作品交付物
- 做发布前完整性校验
- 为桌面工作台提供预览入口

---

## 5.4 CheckRecord（人工检查记录）

### 定位

表示操作者对某个作品版本做出的一次人工检查结论。

### 建议字段

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | 主键 |
| creative_item_id | bigint | 所属作品 |
| creative_version_id | bigint | 所检查版本 |
| result | varchar | approved/rework_required/rejected |
| note | text nullable | 检查意见 |
| changed_fields_json | json nullable | 本次检查后手动修改项 |
| operator_name | varchar nullable | 当前操作者（本地可空） |
| created_at | datetime | 时间 |

### 职责

- 记录“自己检查自己的产物”的过程
- 支撑可发布/需调整/废弃判断
- 为后续排查提供业务证据

---

## 5.5 Task（保持为执行承载对象）

Task 继续承担：

- 合成执行
- 发布执行
- 调度运行
- 执行失败记录

### 建议新增关联

| 字段 | 类型 | 说明 |
|------|------|------|
| creative_item_id | bigint nullable | 所属作品 |
| creative_version_id | bigint nullable | 本次执行针对的版本 |
| task_kind | varchar | composition/publish |

### 原则

- 一个作品可以对应多个 Task
- 一个版本可以有多个执行 Task（如多次重试）
- Task 不再承担“当前作品内容”的最终真相

## 6. 状态机设计

## 6.1 作品状态机（业务状态机）

### 状态枚举

```text
PENDING_INPUT
GENERATING
GENERATED
WAITING_REVIEW
APPROVED
REWORK_REQUIRED
REJECTED
READY_TO_PUBLISH
PUBLISHING
PUBLISHED
GENERATE_FAILED
RENDER_FAILED
PUBLISH_FAILED
ARCHIVED
```

### 推荐简化解释

| 状态 | 含义 |
|------|------|
| PENDING_INPUT | 素材/来源尚未齐备 |
| GENERATING | 正在生成版本或成品 |
| GENERATED | 已生成，但尚未进入检查队列 |
| WAITING_REVIEW | 等待人工检查 |
| APPROVED | 已确认当前版本可发布 |
| REWORK_REQUIRED | 需要调整后重新检查 |
| REJECTED | 决定放弃该作品 |
| READY_TO_PUBLISH | 已进入可发布池 |
| PUBLISHING | 正在发布 |
| PUBLISHED | 已发布成功 |
| GENERATE_FAILED | 生成/装配失败 |
| RENDER_FAILED | 合成失败 |
| PUBLISH_FAILED | 发布失败 |
| ARCHIVED | 归档，不再活跃 |

### 建议主路径

```text
PENDING_INPUT
→ GENERATING
→ GENERATED
→ WAITING_REVIEW
→ APPROVED
→ READY_TO_PUBLISH
→ PUBLISHING
→ PUBLISHED
```

### 返工路径

```text
WAITING_REVIEW → REWORK_REQUIRED → GENERATING
APPROVED → REWORK_REQUIRED → GENERATING
PUBLISH_FAILED → APPROVED / READY_TO_PUBLISH
```

### 终态

- `PUBLISHED`
- `REJECTED`
- `ARCHIVED`

---

## 6.2 任务状态机（执行状态机）

沿用当前实现：

```text
DRAFT
→ COMPOSING
→ READY
→ UPLOADING
→ UPLOADED
→ FAILED
→ CANCELLED
```

### 映射关系

| 作品状态 | 典型 Task 状态 |
|----------|----------------|
| GENERATING | draft / composing |
| WAITING_REVIEW | ready（已生成成品但未发布） |
| PUBLISHING | uploading |
| PUBLISHED | uploaded |
| RENDER_FAILED / PUBLISH_FAILED | failed |

### 原则

- `CreativeStatus` 面向业务和 UI
- `Task.status` 面向执行和后台
- 两者必须有关联，但不能互相替代

---

## 6.3 状态流转规则

## 规则 1：作品必须有当前版本才能进入检查

- `current_version_id != null`
- `PackageRecord.completeness_status == complete`

否则不能进入 `WAITING_REVIEW`

## 规则 2：未通过人工检查不得进入发布池

只有：

- `status == APPROVED`
- 当前版本完整
- 发布账号可用

才能转成 `READY_TO_PUBLISH`

## 规则 3：发布时必须绑定明确版本

发布动作必须针对：

- `creative_item_id`
- `creative_version_id`

禁止只对作品主实体做“模糊发布”。

## 规则 4：重生成会创建新版本，不覆盖旧版本

以下动作必须生成新 `CreativeVersion`：

- 重生成文案
- 重新合成视频
- 替换封面并保存为新版本
- 调整话题后提交为新版本

## 规则 5：Task 失败不必然代表作品失败终止

例如：

- 某次合成失败 → 作品可进入 `REWORK_REQUIRED`
- 某次发布失败 → 作品进入 `PUBLISH_FAILED`，但可再次进入发布池

## 7. 关系设计

## 7.1 核心关系

```text
Product 1 --- n Video/Copywriting/Cover
CreativeItem 1 --- n CreativeVersion
CreativeVersion 1 --- 0..1 PackageRecord
CreativeItem 1 --- n CheckRecord
CreativeItem 1 --- n Task
CreativeVersion 1 --- n Task
```

```sql
CREATE TABLE creative_versions (
```

## 8. 与现有 MVP 的兼容策略

## 8.1 第一阶段不推翻 Task

当前阶段建议：

- 保留 `Task` 作为现有执行主对象
- 新增作品层做业务聚合
- 不强行把所有页面立即迁移

## 8.2 推荐渐进式演进路线

### Phase A：只引入作品聚合层

- 新增 `creative_items`
- 新增 `creative_versions`
- Task 创建时同时创建/关联作品
- UI 新增作品工作台，但保留任务列表

### Phase B：把检查流接到作品层

- 新增 `check_records`
- 新增 `WAITING_REVIEW / APPROVED / REWORK_REQUIRED`
- 发布入口改为从作品池触发

### Phase C：把 Package 标准化

- 新增 `package_records`
- 强制发布前完整性校验
- Task 发布改为读取作品当前版本对应包

## 11. 不变量（Invariants）

以下规则建议作为后端强约束：

1. `CreativeItem.current_version_id` 必须属于该 `CreativeItem`
2. `CreativeVersion.version_no` 在同一作品下唯一递增
3. `APPROVED` 的作品必须存在完整的 `PackageRecord`
4. `READY_TO_PUBLISH` 的作品必须已有最近一次 `CheckRecord.result = approved`
5. `PUBLISHED` 的作品必须至少关联一条成功发布的 Task 或发布结果记录
6. `REJECTED` / `ARCHIVED` 不能再直接进入 `PUBLISHING`

## 12. 风险与取舍

## 风险 1：模型增加会让实现复杂化

### 应对

先做最小作品层：

- CreativeItem
- CreativeVersion
- CheckRecord

`PackageRecord` 可稍后补齐。

## 风险 2：Task 和 Creative 双状态可能不一致

### 应对

- 规定状态源头：业务看 Creative，执行看 Task
- 所有状态更新走统一 service 层
- 加一致性校验任务

## 风险 3：版本过多导致 UI 复杂

### 应对

默认只显示：

- 当前版本
- 最近一次已发布版本
- 最近一次失败版本

其余折叠到历史面板。

---

## 13. 推荐落地顺序

1. 新增 `creative_items` + `creative_versions` 表
2. Task 创建时建立作品映射
3. 新增作品列表/详情 API
4. 新增检查记录与作品状态流转
5. 发布入口改为面向作品版本
6. 最后补 `package_records` 标准化

---

## 14. 一句话定义

> 作品实体用于承接“内容本身”的业务真相，任务实体用于承接“后台执行过程”的运行真相；两者分层后，桌面产品才能同时做好作品管理、人工检查和执行追踪。
