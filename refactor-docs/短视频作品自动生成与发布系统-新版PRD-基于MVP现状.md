> 版本：Draft v0.6
> 更新时间：2026-04-23
> 文档定位：基于当前 MVP 现状梳理的新 PRD

## 1. 文档目的

本文档不是从零想象一个系统，而是：

1. 基于当前仓库已经实现的 MVP 能力，梳理真实可用边界；
2. 结合前期“短视频作品自动生成与发布系统”设计方向，收敛出新的业务主流程；
3. 明确下一阶段产品应补齐的关键能力、流程节点和系统边界；
4. 输出一份适合业务、产品、后端、前端共同对齐的新版 PRD。

## 2. 当前 MVP 结论摘要

结合当前代码、页面、接口与文档，现阶段 MVP 已经不是“空壳原型”，而是一个 **可运行的任务式发布系统雏形**，但它仍然偏向“素材管理 + 任务组装 + 调度发布”的第一阶段，距离“完整作品生产工作流系统”还有关键断层。

### 2.1 当前已实现的 MVP 主能力

1. **账号与授权能力**
   - 已有账号管理、登录态保存、会话校验、授权中心运行时门禁（当前实现能力来自 remote-auth）。
   - 调度器、合成链、发布链已经受授权状态控制。

2. **商品即素材包能力**
   - 已支持从得物链接创建商品。
   - 已支持解析商品页，自动拉取视频、封面、标题文案、话题。
   - 商品已成为素材组织入口之一。

3. **素材库能力**
   - 已有视频、文案、封面、音频、话题等独立素材管理。
   - 任务创建页支持素材篮模式。
   - 支持从商品快速导入素材到任务。

4. **任务组装能力**
   - 已实现资源集合模型：视频 / 文案 / 封面 / 音频 / 话题按集合关联到 Task。
   - 已支持“一份素材集合 × 多账号 = 多个任务”的分发模式。
   - 已引入 PublishProfile（配置档）和 ScheduleConfig（调度配置）。

5. **合成与发布基础链路**
   - 已有任务状态机：draft / composing / ready / uploading / uploaded / failed / cancelled。
   - 已有 Coze 合成任务提交、轮询、回写结果。
   - 已有发布调度器、发布时间窗口、账号日限额、乱序发布等运行参数。

6. **AI 工具页能力**
   - 已有独立 AIClip 页面。
   - 支持单视频信息探测、高光检测、智能裁剪、加音频、加封面、完整流水线工具调用。

## 2.2 当前已落地的 VIP / 产品控制后台能力基线

当前产品控制后台部分已经不只是抽象规划，而是已有明确实现边界。

### 已落地的远端终端用户授权能力

- `POST /login`
- `POST /refresh`
- `POST /logout`
- `GET /me`

### 已落地的远端后台管理能力

- `POST /admin/login`
- `GET /admin/session`
- `GET /admin/users`
- `GET /admin/devices`
- `GET /admin/sessions`
- `GET /admin/audit-logs`
- `GET /admin/metrics/summary`

### 已落地的远端自助能力

- `GET /self/me`
- `GET /self/devices`
- `GET /self/sessions`
- `GET /self/activity`
- `POST /self/sessions/{session_id}/revoke`

### 已落地的本地授权执行能力

- 本地 machine-session truth
- 本地 auth 状态机
- offline grace / hard deny / active 的运行时门禁
- 调度器、合成链、发布链受本地授权状态控制

### 当前阶段结论

当前产品控制后台的真实定位已经是：

- **产品控制后台基础设施已落地**
- **但产品控制后台仍未完全扩展到 VIP / License / 高级能力门禁 / 配置下发全范围**

## 3. 当前 MVP 的核心问题

当前 MVP 的主要问题不是“功能太少”，而是 **业务闭环还没有真正收口**。

### 3.1 问题一：系统主线仍是“任务发布”，不是“作品生产”

当前系统的核心对象还是 `Task`，而不是“作品包 / 成品单元 / 检查确认单元”。

结果是：

- 用户能创建任务，但很难以“作品”的视角进行预览、筛选、检查、改稿；
- 当前任务详情更像技术视图，偏执行语义，不是运营视图；
- 发布前缺少标准化“成品确认”环节。

### 3.2 问题二：AIClip 是工具页，未进入正式业务流程

当前 AIClip 更像单机工具：

- 输入本地路径；
- 输出本地文件；
- 不自动回写素材库；
- 不自动形成作品；
- 不自动进入任务人工检查/发布链路。

这意味着“合成能力”和“任务系统”仍然是分离的。

### 3.3 问题三：作品包标准未收口

前期设计里希望收敛成统一产物，例如：

- `1.mp4`
- `1.jpg`
- `商品名称.txt`
- `文案.txt`

但当前 MVP 中：

- 有 `final_video_path` 概念；
- 有封面、文案、话题、商品名等离散资源；
- 但没有统一的 `作品包 Package` 模型；
- 没有“成品文件齐套校验”机制；
- 没有面向人工检查/发布的标准成品视图。

### 3.4 问题四：人工检查链缺失

目前缺少清晰的：

- 待检查状态；
- 确认可发布 / 放弃；
- 局部重生成；
- 版本对比；
- 检查意见记录。

现状更像：

- 任务满足语义即可发布；
- 失败可重试；
- 但没有面向内容质量的人机协同流程。

### 3.5 问题五：配置档和页面已经暴露“local_ffmpeg”，但执行链未真正实现

当前配置档支持：

- `none`
- `coze`
- `local_ffmpeg`

但当前合成服务实际上只明确支持：

- `none`
- `coze`

因此：

- 产品层已经承诺了一部分能力；
- 实际执行层尚未闭环；
- 这会造成用户理解偏差和交付预期偏差。

### 3.6 问题六：文案生成仍未进入正式任务链

当前系统已有：

- 商品解析得到基础标题文案；
- 文案素材 CRUD；
- 但没有“AI 自动生成文案 -> 清洗 -> 人工检查 -> 发布使用”的正式流程。

缺少：

- 模板管理；
- 变量填充；
- 多版本文案；
- 敏感词过滤；
- 文案重试与人工检查。

### 3.7 问题七：话题管理仍偏素材化，未成为发布策略的一部分

当前话题更多是素材关系的一部分，但还缺少：

- 热门话题搜索/推荐；
- 发布策略级话题模板；
- 话题应用优先级规则；
- 任务/作品层的话题确认机制。

### 3.8 问题八：当前能力边界尚未在 PRD 中完全显式化

当前实现里其实已经有一些明确的执行边界，但文档层还没有完全写透，例如：

- direct publish 的硬约束；
- profile/global/topic-group 的真实自动注入边界；
- 本地 machine-session truth 与 offline grace；
- 产品控制后台已落地能力与后续能力之间的边界。

如果这些边界不写清楚，后续开发、产品沟通和商业化控制都会出现预期偏差。

## 4. 新版 PRD 的核心定义

### 4.1 产品形态定位

将本系统重新定义为：

> 一个最终以 **Electron PC 桌面程序** 交付、面向单个操作者使用、同时具备 **产品控制后台 / 授权中心** 的“短视频作品生产与发布工作流系统”。

它不是纯浏览器形态的 SaaS，也不是完全离线的单机工具，而是更接近：

> **Desktop-first connected product（桌面优先的联网产品）**

也可以理解为：

- **本地执行面（local execution plane）**：承担素材、合成、文件、浏览器自动化、发布执行；
- **产品控制面（control plane）**：承担用户、授权、设备、版本门禁、后台管理、产品运维；
- **Electron 前台**：承担桌面工作台体验。

### 4.2 这是不是 SaaS

如果从“前后端 + 用户授权 + 后台管理 + 可销售产品”角度看，它确实有 **SaaS-like** 特征；
但从核心执行形态看，它又不是标准 SaaS，因为：

1. 核心业务执行发生在用户本机，而不是远端计算节点；
2. 素材文件、FFmpeg、浏览器自动化、发布动作强依赖本地环境；
3. Electron 存在的原因，不只是做 UI 壳，而是承载本地前后端一体化运行。

因此更准确的说法是：

> **一个带远端控制后台的桌面软件产品**，而不是纯 Web SaaS。

### 4.3 系统拓扑定位

建议把系统明确拆成三层：

```text
[Electron Renderer / 桌面工作台]
        ↓
[Local Backend / 本地业务后端]
        ↓
[本地文件系统 + SQLite + FFmpeg + 浏览器自动化 + 调度器]
        ↓
[产品控制后台 / 授权中心（当前实现目录：remote/remote-backend）]
        ↓
[产品管理后台（当前实现目录：remote/remote-admin）]
```

其中：

- **本地后端** 是真正的业务执行内核；
- **产品控制后台** 不是可有可无的小附属，而是商品化交付所必须的产品控制中心；
- **授权中心能力**（当前实现中最早落地的是 remote-auth）只是最先被看到的一部分，不应把产品控制后台的价值误解为“只做授权”。

### 4.4 新的业务主线

新的业务主线应为：

```text
商品/素材导入
  → 作品生成
  → 人工检查与微调
  → 确认可发布
  → 自动发布
  → 结果回写
```

同时在产品级还存在一条支撑主线：

```text
用户登录/授权
  → 设备绑定与校验
  → 本地能力开放
  → 运行状态上报/支持排障
  → 远端后台管理与产品运维
```

也就是说，下一阶段系统要从“任务驱动”升级为“作品驱动 + 任务执行承载”，并且明确服务于 **本地桌面单人工作流 + 产品控制后台** 的双层结构。

### 4.5 核心设计原则

1. **桌面优先**：交互、状态、文件、日志、异常提示都要适配 Electron 本地桌面体验。
2. **单人优先**：本地业务流程不要引入过重的多角色审批/RBAC 模型，而应设计成一个操作者的连续工作流。
3. **本地执行优先**：重媒体处理、文件 IO、浏览器自动化、发布执行留在本地完成。
4. **产品控制后台优先**：用户、授权、设备、版本门禁、后台管理、运营控制放在产品控制后台统一治理。
5. **真相分层明确**：本地后端负责业务执行真相，产品控制后台负责产品控制与授权真相，避免 split-brain。
6. **作品优先**：操作者真正关心的是作品结果，而不是底层任务对象。
7. **任务承载执行**：Task 继续作为后台执行单元，但前台主视图应是作品。
8. **人工检查前置**：从“生成即发布”升级为“生成 -> 人工检查 -> 发布”。
9. **流程闭环**：素材、合成、文案、成品、发布必须进入统一状态机。
10. **局部重试**：失败或不满意时应支持单作品重生成，而不是整批重来。
11. **联网可控、断网可退化**：产品控制后台不可用时，非授权核心能力应尽量保留中间状态与恢复能力。
### 4.6 商业化控制原则

由于本产品涉及收费与授权控制，架构上必须接受一个现实：

> **如果所有关键能力都只落在本地客户端，最终一定会面临被破解后脱离后端独立运行的风险。**

因此本项目必须采用“**本地执行业务，远端掌握关键控制点**”的策略。

> **本地后端负责业务执行，产品控制后台逐步承接关键商业控制点。**  
> **这样既保留桌面端执行效率，也能避免产品在商业上完全失控。**

建议原则如下：

1. **业务主流程留在本地**：素材、合成、发布执行仍然在本机完成。
2. **关键控制点逐步上移到产品控制后台**：授权、设备、版本门禁、核心 entitlement、关键策略开关、关键能力许可应由产品控制后台决定。
3. **早期可以简单，后期逐步增强**：MVP 阶段先保证可用，随着产品商业化推进，再逐步提高产品控制后台覆盖度。
4. **目标不是绝对防破解，而是提高脱离后端独立运行的成本**：从商业上保障产品控制权和持续收费能力。

可逐步上移到产品控制后台的关键控制项示例：

- 用户授权状态与 License 套餐
- 设备绑定与设备数量限制
- 最低版本门禁
- 高级功能 entitlement（如高阶合成链、批量能力、远端模板同步）
- 关键模板/规则/工作流 ID 下发
- 部分关键运行参数与策略开关

## 5. 新版目标

## 5.1 产品目标

1. 打通“商品/素材 -> 作品 -> 发布”的完整闭环。
2. 建立统一作品包模型，形成标准成品交付物。
3. 支持人工检查和局部重生成，提升内容质量控制能力。
4. 保留当前 MVP 的调度发布能力，但将其纳入更完整的业务流程中。
5. 建立“本地执行 + 远端控制”的产品架构，为销售、授权、支持、版本治理打基础。
6. 为后续模板化生产、多渠道发布、远端配置下发、产品运维能力扩展打基础。

## 5.2 业务目标

- 让单个操作者以最少操作完成批量作品生产。
- 让同一个人在一个界面内完成素材整理、人工检查、发布确认。
- 让桌面端明确展示“哪些作品可发、哪些需要处理、哪些失败了”。
- 让本地系统具备可追踪、可回放、可重试的完整执行链。
- 让远端后台具备用户授权管理、设备管理、版本门禁、产品支持和后续商业化扩展能力。

## 6. 操作模式与系统参与方

### 6.1 本地操作模式：单人桌面模式

本地业务流程不按多角色协同后台设计，而按 **单人桌面操作模式** 设计。

### 6.2 全局参与方

| 对象 | 说明 |
|------|------|
| 操作者 | 同一个人完成商品导入、素材整理、作品生成、人工检查、发布确认 |
| Electron 桌面前台 | 提供主工作台、作品预览、配置、异常提示、本地入口 |
| 本地业务后端 | 提供素材、任务、作品、合成、发布、调度、日志等 API 与执行能力 |
| 本地执行组件 | SQLite、文件系统、FFmpeg、浏览器自动化、调度器、任务进程 |
| 产品控制后台 | 提供用户、授权、设备、版本门禁、后台管理、后续产品控制能力 |
| 产品管理后台 | 给产品运营/管理员/支持人员使用，用于授权与产品运维 |

### 6.3 设计含义

这意味着：

#### 本地产品侧应避免

- 复杂角色拆分；
- 多人协作流转；
- 审批型本地权限模型；
- 过重的工作台分岗设计。

#### 远端产品侧应保留

- 用户授权管理；
- 设备绑定与控制；
- 版本门禁；
- 后台支持与审计；
- 未来的产品配置/模板/套餐能力扩展。

#### 本地交互上应强调

- 一个主工作台完成主要操作；
- 人工检查是“自己检查自己的产物”；
- 所有关键异常都应在桌面端即时提示；
- 本地文件、日志、任务状态要可见、可定位、可重试。

## 7. 业务对象重构

## 7.1 核心对象

### A. 商品 Product

继续作为素材包入口，负责聚合：

- 得物链接
- 解析出的媒体资源
- 解析出的标题文案
- 解析出的话题

### B. 素材 Asset

继续保留：

- 视频
- 文案
- 封面
- 音频
- 话题

### C. 作品 Work / CreativeItem（新增一级概念）

新增“作品”概念，表示一条待人工检查、待发布的内容单元。

建议作品具备：

- 作品编号
- 来源商品 / 素材集合
- 当前视频版本
- 当前封面版本
- 当前文案版本
- 当前话题集合
- 当前检查状态
- 当前发布状态

### D. 作品包 Package（新增标准产物概念）

作品包用于承接标准化成品输出。

最小标准：

- 最终视频
- 封面图
- 标题/商品名称
- 文案
- 结构化 manifest

### E. 任务 Task

Task 不再作为前台唯一主角，而是：

- 作品生产任务
- 发布执行任务
- 调度运行任务

也就是“后台执行承载对象”。

## 8. 新版业务流程

## 8.1 一级流程

```text
阶段1：商品/素材准备
阶段2：作品生成
阶段3：成品人工检查
阶段4：发布准备
阶段5：自动发布
阶段6：结果回写与复盘
```

## 8.2 详细流程

### 阶段1：商品/素材准备

#### 输入方式

- 从得物链接创建商品并自动解析素材
- 手工上传视频/封面/音频/文案
- 从素材库选择已有素材
- 从商品一键导入素材到作品生成流程

#### 输出结果

- 商品已解析完成
- 素材库可用
- 素材元数据齐全

### 阶段2：作品生成

#### 输入

- 商品/素材集合
- 配置档（合成模式、默认话题、重试策略）
- 目标账号或账号池

#### 处理动作

- 创建作品生成批次
- 对每个作品生成单元分配素材
- 若为直接发布模式，则校验单视频语义
- 若为合成模式，则进入合成流程
- 生成文案（手工/复用/AI）

## 10.6 当前能力边界与规则基线

### 当前 direct publish 硬约束

结合当前实现，direct publish 不是开放式能力，而是保守模式。

当前应明确遵守：

- 仅支持 **1 个最终视频**
- 仅支持 **0 或 1 个文案**
- 仅支持 **0 或 1 个封面**
- **独立音频不能直接发布**
- 多视频、多文案、多封面场景不能静默降级，必须先进入合成链或人工调整

若任务已完成合成并生成 `final_video_path`，则 direct publish 使用该最终视频，而不是静默从多素材里取第一项。

### 当前话题自动注入边界

当前自动进入任务/作品组装流程的话题来源，仅包括：

- 显式传入的 `topic_ids`
- `PublishProfile` 的默认话题

当前 **不会** 自动参与组装的来源：

- global topics
- topic groups

这条规则需要在产品与实现上保持一致，避免“配置了全局话题却未生效”的理解偏差。

### 当前本地授权运行规则

当前本地业务执行不是“每次都直接问远端”，而是采用：

- 产品控制后台提供授权真相
- local 维护 machine-session truth
- local 根据 auth_state 执行门禁

至少要显式考虑三类状态：

- `authenticated_active`
- grace / restricted
- hard deny（如 revoked / device_mismatch / expired）

调度器、发布、合成相关流程都必须受该本地运行状态约束。

## 10.7 VIP / Entitlement 能力门禁矩阵（建议基线）

| 能力 | 当前执行位置 | 当前控制位置 | 后续建议 |
|------|--------------|--------------|----------|
| 登录 / 授权 / refresh | local + 产品控制后台 | 产品控制后台 | 保持产品控制后台为真相源 |
| 调度发布 | local | local | 增加 entitlement / License 门禁 |
| Coze 合成 | local | local | 增加产品控制后台 capability grant |
| local_ffmpeg 合成 | local | local | 可纳入套餐控制 |
| 批量作品上限 | local | local | 后续上移产品控制后台控制 |
| 高级模板 / 工作流 / 配置 | local | local | 后续由产品控制后台下发 |
| 产品诊断 / 支持能力 | 产品控制后台 | 产品控制后台 | 持续增强 |

这张矩阵的意义不是一步到位，而是明确：

- 早期先跑通；
- 中期开始 gate 高级能力；
- 后期把关键商业控制点逐步远端化。

## 12. 后台设计影响

## 12.0 先给结论：这是什么样的后端系统

从最佳实践看，本项目的后端体系更适合定义为：

> **一个桌面优先产品的双后端架构**

而不是单一后端，也不是纯 SaaS。

它包含两套后端：

1. **本地业务后端（Local Business Backend）**
   - 跑在用户电脑上；
   - 服务 Electron 前台；
   - 执行真实业务流程。

2. **产品控制后台（当前实现目录：`remote/remote-backend`）**
   - 跑在服务端；
   - 服务于产品授权、后台管理、设备控制、版本门禁、支持运维；
   - 为“可销售产品”提供统一控制面。

其中 remote-auth 只是产品控制后台当前最先落地的一条链路，但产品控制后台不应被定义为“只是授权小模块”。

更准确地说：

- **本地后端** = 业务执行内核；
- **产品控制后台** = 产品控制内核。

## 12.1 Electron 本地架构约束

由于最终交付形态是 Electron PC 桌面程序，后台设计需要遵守以下前提：

1. **本地优先**：重媒体处理、文件系统访问、浏览器自动化、发布执行主要留在本地。
2. **单用户优先**：本地默认只有一个操作者，不需要设计复杂的多人权限体系。
3. **文件可达优先**：视频、封面、日志、导出目录必须便于从桌面端直接访问。
4. **异常可见优先**：本地转码失败、文件缺失、登录失效、发布失败都要直接反馈给桌面端 UI。
5. **离线韧性优先**：除账号登录、模型调用、平台发布等必须联网环节外，其它流程尽量可本地继续执行或保留中间状态。

## 12.2 本地业务后端职责

本地后端应该继续作为主业务后端，核心职责包括：

### A. 本地领域数据管理

- 商品/素材/任务/作品/作品版本管理
- 本地 SQLite 真相维护
- 本地文件与对象路径管理

### B. 本地工作流编排

- 任务组装
- 作品生成
- 文案生成编排
- 合成编排
- 发布编排
- 重试与恢复

### C. 本地执行能力

- FFmpeg 调用
- Patchright/Playwright 浏览器自动化
- 素材下载与存储
- 调度器运行
- 日志和诊断信息收集

### D. 本地 API 层

- 给 Electron Renderer 提供稳定 API
- 作为 UI 与执行层之间的边界
- 负责本地状态聚合与错误语义统一

## 12.3 产品控制后台职责

产品控制后台不负责本地素材、任务、合成执行这些高 IO / 强本地依赖业务；它负责产品级能力。

当前已落地或已明确方向的职责包括：

### A. 用户与授权

- 用户登录
- access / refresh token 生命周期
- 本地机器授权上下文上游真相
- 授权状态（active / revoked / disabled / expired）

### B. 设备与版本控制

- 设备绑定 / 解绑 / mismatch
- minimum supported version gate
- offline grace 策略

### C. 远端后台管理

- 用户管理
- 授权管理
- 设备管理
- session 管理
- 审计日志

### D. 产品运营与支持扩展位

从最佳实践看，产品控制后台后续还可承接：

- 套餐/License 管理
- 版本发布策略
- 远端配置下发
- 模板/规则同步
- 远端诊断与支持数据
- 使用统计与产品运营指标

这部分不一定都要在当前版本实现，但 PRD 应把产品控制后台定位成 **产品控制面**，而不是“Electron 附带的授权小服务”。

## 12.3.1 当前产品控制后台能力基线（实现对齐）

当前产品控制后台实现已经覆盖三类 API 面：

### A. End-user auth 面

- login
- refresh
- logout
- me

### B. Admin control 面

- admin login / session
- users
- devices
- sessions
- audit logs
- metrics summary

### C. Self-service 面

- self me
- self devices
- self sessions
- self activity
- self session revoke

因此 PRD 中对产品控制后台的描述，不应再停留在“只有授权想法”，而应以“**已落地的产品控制基础设施**”为前提。

## 12.3.2 本地 machine-session truth 与 offline grace

本地后端除了业务执行，还承担一个非常关键的职责：

- 维护当前机器授权会话真相；
- 在产品控制后台不可达或 refresh 失败时执行 grace / deny 策略；
- 将 auth gate 下沉到本地调度器与业务运行时。

这意味着：

- 产品控制后台是上游授权真相；
- local 是当前机器运行真相；
- Electron 前台只消费 local 状态，而不直接拼装产品控制后台语义。

## 12.4 本地 / 远端的边界建议

最佳实践是按“谁更适合作为真相源”来划分职责。

| 领域 | 建议真相源 | 原因 |
|------|------------|------|
| 商品/素材/作品/任务 | 本地后端 | 强依赖本地文件与执行上下文 |
| 合成结果 / 发布结果执行态 | 本地后端 | 执行发生在本机 |
| 当前机器授权会话 | 本地后端（machine-session truth） | 需要驱动本地门禁与容错 |
| 用户身份与授权策略 | 产品控制后台 | 需要统一治理与可销售控制 |
| 设备绑定 / 版本门禁 | 产品控制后台 | 需要统一控制 |
| 管理员操作 / 审计 | 产品控制后台 | 需要中心化留痕 |

设计上必须避免：

- 同一类状态同时在本地和远端各自定义一套语义；
- UI 直接推断授权真相；
- 本地与远端错误码漂移；
- 本地把产品控制后台结果再发明成另一套状态机。

## 12.5 现阶段本地后端应新增的核心概念

建议新增：

- `creative_items`
- `creative_versions`
- `package_records`
- `review_records`（也可命名为 `check_records`，语义更贴近单人检查）

### creative_items

表示作品主实体。

### creative_versions

表示作品当前版本对应的视频、封面、文案、话题组合。

### package_records

表示最终成品包及其结构化 manifest。

### review_records / check_records

表示人工检查动作、操作者确认结果、调整意见、操作时间。

## 12.6 现有 Task 的职责调整

Task 继续存在，但其职责要更聚焦：

- 承接一次合成执行；
- 承接一次发布执行；
- 为作品提供底层执行记录。

不要再让 Task 同时承担：

- 业务主对象
- 人工检查对象
- 成品对象
- 发布结果对象

这会继续造成语义拥挤。

## 12.7 AIClip 服务的整合方向

当前 `AIClipService` 偏工具能力，下一阶段建议分两层：

### Tool 层

继续保留：

- 获取视频信息
- 检测高光
- 裁剪
- 加音频
- 加封面

### Workflow 层

新增：

- 接收作品生成请求
- 读取素材库资源

- 回写 creative version / package record

## 12.8 配置档模型增强方向

`PublishProfile` 建议从“合成配置档”升级为“作品生产配置档”：

新增配置维度：

- 文案策略
- 视频输出规格
- 默认检查策略
- 发布渠道限制
- 素材选择规则

## 12.9 产品控制后台的最佳实践建议

### 1）把产品控制后台当成 control plane，不当成业务执行 plane

产品控制后台负责：

- 用户
- 授权
- 设备
- 版本
- 后台管理
- 审计

本地执行面负责：

- 素材
- 合成
- 文件
- 发布执行

### 2）保持契约稳定

产品控制后台 `/login /refresh /logout /me` 以及 admin API 的错误码和字段必须稳定，不能频繁漂移。

### 3）本地后端要成为 Electron 唯一业务入口

不要让 Electron Renderer 直接碰底层执行逻辑；统一走本地 API。

### 4）支持可观测性与支持排障

除了 auth，产品控制后台最终还应具备：

- 用户维度问题排查；
- 设备维度问题排查；
- 版本兼容排查；
- 关键事件审计。

### 5）为商业化预留扩展位

如果这是可销售产品，产品控制后台未来至少要能承接：

- License 套餐；
- 试用/正式授权；
- 版本灰度；
- 功能 entitlements；
- 产品运营配置。

## 12.10 商业化与防脱离后端运行策略

从最佳实践看，收费型桌面软件不应把全部价值都放在本地，否则很容易出现：

- 破解后跳过登录/授权；
- 伪造本地状态绕过门禁；
- 脱离远端长期独立运行；
- 复制安装包即可无限扩散。

因此架构上应采用 **分层控制**：

### 第一层：基础授权控制（当前已具备雏形）

- 登录
- refresh
- `/me`
- revoke
- device mismatch
- minimum version gate

### 第二层：关键能力控制（下一阶段建议增强）

- 按 License/套餐开放能力
- 按 entitlement 控制高级功能
- 关键模板/工作流/配置由产品控制后台下发
- 本地执行前校验 capability grant

### 第三层：产品运营控制（后续演进）

- 试用/正式版区分
- 套餐升级/降级
- 功能灰度
- 使用限制与策略控制
- 支持与审计

需要强调：

- **不可能完全杜绝破解**；
- 但可以通过把关键控制点放在产品控制后台，显著提高破解后“完全脱离后端运行”的难度；
- 这对于商业利益保护是必要的，也是桌面收费软件的常见实践。
