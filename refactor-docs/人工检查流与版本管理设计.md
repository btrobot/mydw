## 1. 文档目标

本文档聚焦两个核心问题：

1. 在桌面单人模式下，“人工检查”到底检查什么、如何流转、如何落日志。
2. 一条作品在多次生成、改稿、重合成、重发过程中，如何做版本管理而不破坏当前 MVP 的 Task 执行链。

目标是把“人工检查”从模糊概念，落成可实现的产品流程和后端结构。

---

## 2. 设计结论先行

### 2.1 核心结论

建议将“人工检查流”定义为：

> **操作者对某个作品版本做出“可发布 / 需调整 / 放弃”的显式业务判断，并把该判断沉淀为可追溯记录。**

建议将“版本管理”定义为：

> **每一次影响作品最终内容的修改，都产生新的 CreativeVersion；人工检查永远针对具体版本，而不是针对抽象作品。**

### 2.2 一句话解释

- **检查流** 负责决定“这版能不能发”。
- **版本管理** 负责决定“到底是哪一版被检查、被确认、被发布”。

---

## 3. 为什么现在必须补人工检查流

当前 MVP 已经具备：

- 任务创建
- 素材篮组装
- 合成提交
- 发布执行
- 失败后 quick retry / edit retry

但当前缺少一层明确的“内容确认”流程，因此存在以下问题：

### 3.1 当前 TaskDetail 更像执行页，不是检查页

当前页面展示重点是：

- Task 状态
- 执行语义
- 素材集合
- 合成状态
- 重试 / 取消 / 删除

这适合开发与调试，但不适合操作者做“内容是否满意”的判断。

### 3.2 当前失败后只有执行重试，没有内容返工语义

现在有：

- `quick_retry`：失败后回到失败前状态
- `edit_retry`：失败后回到 `draft`

但没有区分：

- 因网络故障失败，应该重试执行
- 因文案不满意、封面不合适、视频不理想，应该进入内容返工

### 3.3 当前没有“哪一版被确认”的业务证据

如果一条作品经历：

1. 初次生成
2. 重生成文案
3. 替换封面
4. 再次合成

当前缺少明确记录：

- 操作者检查的是哪一版
- 哪一版被确认可发布
- 发布时实际用了哪一版

这会影响：

- UI 可解释性
- 失败追踪
- 商业化环境下的支持排障

---

## 4. 人工检查流定义

## 4.1 检查对象

人工检查的对象不是 Task，而是：

- `CreativeItem` 的 **当前版本**
- 或者某个指定的 `CreativeVersion`

### 必须展示的检查输入

- 最终视频预览
- 封面图
- 主文案
- 标题/商品名
- 话题列表
- 来源素材摘要
- 当前版本号
- 最近失败原因（如有）

---

## 4.2 检查动作

建议保留 3 个核心动作：

### A. 确认可发布（Approve）

含义：

- 当前版本满足发布要求
- 可以进入发布池

### B. 需调整（Rework Required）

含义：

- 当前版本不满意
- 需要继续修改并生成新版本
- 不允许进入发布池

### C. 放弃（Reject）

含义：

- 当前作品不再继续推进
- 后续不再参与发布

---

## 4.3 检查输出

每次检查至少要落一条 `CheckRecord`，包含：

- 作品 ID
- 版本 ID
- 检查结果
- 检查意见
- 是否产生后续返工动作
- 时间

### 结果与作品状态对应

| 检查结果 | 作品状态 |
|----------|----------|
| approve | `APPROVED` |
| rework_required | `REWORK_REQUIRED` |
| rejected | `REJECTED` |

---

## 5. 版本管理定义

## 5.1 什么叫版本

版本不是 Task 的状态变化，而是“最终内容快照”的变化。

### 应触发新版本的操作

以下动作建议都创建新 `CreativeVersion`：

1. 初次生成作品
2. 重生成文案
3. 重新合成视频
4. 替换封面并保存
5. 修改标题后保存
6. 修改话题并提交为新版本
7. 手工导入替换最终视频

### 不应触发新版本的操作

以下动作只影响执行，不影响内容：

1. quick retry
2. 重新发布同一版本
3. 重新拉起调度
4. 查看日志
5. 取消某次执行任务

---

## 5.2 版本粒度原则

### 原则 1：版本必须完整快照

一个版本必须能独立回答：

- 视频是谁
- 封面是谁
- 文案是什么
- 话题有哪些
- 来源素材是什么

### 原则 2：版本不可被原地覆盖

即使是小改动，也建议：

- 新建版本
- 更新 `CreativeItem.current_version_id`

而不是修改旧版本内容。

### 原则 3：检查与发布必须绑定版本

- 检查针对版本
- 发布针对版本
- 发布成功记录也必须回写版本

---

## 5.3 版本类型建议

| version_type | 含义 |
|--------------|------|
| generated | 系统初次生成 |
| regenerated_copy | 重生成文案 |
| rerendered_video | 重合成视频 |
| manual_edit | 手动编辑文案/标题/话题 |
| manual_replace_cover | 手动替换封面 |
| manual_replace_video | 手动替换最终视频 |
| publish_snapshot | 发布前冻结快照（可选） |

---

## 6. 人工检查流状态机

## 6.1 主流程

```text
GENERATED
→ WAITING_REVIEW
→ APPROVED
→ READY_TO_PUBLISH
→ PUBLISHING
→ PUBLISHED
```

## 6.2 返工流程

```text
WAITING_REVIEW
→ REWORK_REQUIRED
→ GENERATING
→ GENERATED
→ WAITING_REVIEW
```

## 6.3 放弃流程

```text
WAITING_REVIEW → REJECTED
REWORK_REQUIRED → REJECTED
APPROVED → REJECTED（仅手动撤销时）
```

---

## 6.4 状态流转规则

### 规则 1：必须有版本才能进入 WAITING_REVIEW

至少满足：

- `current_version_id != null`
- 当前版本有最小可预览内容

### 规则 2：APPROVED 只能针对当前版本生效

如果操作者批准的是版本 3：

- `CreativeItem.current_version_id` 必须是 3
- 当前可发布结论也只对版本 3 生效

如果后来生成了版本 4，则版本 3 的批准结论默认失效，不自动继承。

### 规则 3：REWORK_REQUIRED 必须指向返工动作

`REWORK_REQUIRED` 不是终态，而是返工入口。

建议同时记录返工类型：

- copy
- cover
- video
- topic
- mixed

### 规则 4：REJECTED 是业务终态，不等于执行失败

- `REJECTED` 表示放弃内容推进
- 不应自动触发 retry
- 与 `Task.failed` 不是一回事

---

## 7. 检查清单设计

为避免桌面端检查流过于主观，建议定义最小检查清单。

## 7.1 必查项

### 内容侧

- 视频是否能正常播放
- 封面是否可接受
- 文案是否通顺/合规
- 话题是否正确
- 标题/商品名是否正确

### 技术侧

- 是否存在最终视频文件
- 是否存在封面文件
- 视频时长是否合理
- 当前版本是否完整

### 发布侧

- 是否有可用发布账号
- 当前配置档是否满足语义
- 是否满足 direct publish / composition 规则

---

## 7.2 检查结果模板建议

### 通过

- 可发布
- 文案正常
- 封面正常
- 视频正常

### 需调整

- 文案需重写
- 封面需更换
- 视频节奏不对，需重合成
- 话题不合适

### 放弃

- 该作品不再推进
- 当前素材质量不达标
- 当前内容不适合发布

---

## 8. 与当前 Task 执行链的协同

## 8.1 当前 Task 保持不变的部分

保留：

- `draft / composing / ready / uploading / uploaded / failed / cancelled`
- `quick_retry`
- `edit_retry`
- `submit_composition`
- `publish`

## 8.2 新增作品层后的解释方式

### 当前语义映射

| Task 行为 | 作品语义 |
|----------|----------|
| 创建 task | 创建作品或作品执行任务 |
| submit composition | 生成当前版本的执行过程 |
| task ready | 成品已生成，可进入检查阶段 |
| task uploaded | 某版本发布成功 |
| task failed | 某次执行失败，不等于作品终止 |

## 8.3 推荐协同模式

### 模式 A：一个版本对应一个主执行 Task

适合 MVP 过渡期：

- 当前版本由一个主 Task 驱动生成
- 重生成产生新版本 + 新 Task
- UI 先通过 Task 兼容现有链路

### 模式 B：一个版本可对应多个执行 Task

适合后续完整设计：

- 版本生成 Task
- 发布 Task
- 重试 Task
- 补偿 Task

这种模式更灵活，但复杂度更高。

建议先落地 **模式 A**。

## 9. 后端数据结构建议

## 9.1 check_records

```sql
CREATE TABLE check_records (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  creative_item_id INTEGER NOT NULL,
  creative_version_id INTEGER NOT NULL,
  result VARCHAR(32) NOT NULL,
  note TEXT NULL,
  rework_type VARCHAR(32) NULL,
  changed_fields_json TEXT NULL,
  operator_name VARCHAR(128) NULL,
  created_at DATETIME NOT NULL
);
```

### 说明

- `result`: approved / rework_required / rejected
- `rework_type`: copy / cover / video / topic / mixed
- `changed_fields_json`: 记录本次检查后手工改了什么

## 9.2 CreativeVersion 的补充字段建议

若需要更强版本追踪，可补充：

| 字段 | 用途 |
|------|------|
| parent_version_id | 溯源版本链 |
| superseded_by_version_id | 指向新版本 |
| is_current | 是否当前版本（可选，推荐仍以 current_version_id 为准） |
| published_at | 若该版本被发布成功，可记录时间 |
| publish_count | 该版本被发布的次数 |

## 10. API 设计建议

## 10.1 检查流 API

- `GET /api/creatives/{id}/checks`
- `POST /api/creatives/{id}/approve`
- `POST /api/creatives/{id}/rework`
- `POST /api/creatives/{id}/reject`

### approve 请求建议

```json
{
  "creative_version_id": 3,
  "note": "文案和封面都可以，允许发布"
}
```

### rework 请求建议

```json
{
  "creative_version_id": 3,
  "rework_type": "copy",
  "note": "文案偏硬，需要重写"
}
```

### reject 请求建议

```json
{
  "creative_version_id": 3,
  "note": "该作品不再推进"
}
```

## 10.2 版本 API

- `GET /api/creatives/{id}/versions`
- `GET /api/creative-versions/{id}`
- `POST /api/creative-versions/{id}/activate`
- `POST /api/creatives/{id}/regenerate-copy`
- `POST /api/creatives/{id}/rerender`
- `POST /api/creatives/{id}/replace-cover`

## 11. UI 交互建议

## 11.1 作品检查页布局

### 左侧：预览区

- 视频播放器
- 封面图
- 文件完整性状态

### 右侧：检查区

- 标题
- 文案
- 话题
- 当前版本号
- 上次检查结果
- 操作按钮：
  - 确认可发布
  - 需调整
  - 放弃

### 底部：版本历史

- 版本号
- 创建时间
- 创建原因
- 检查结论
- 是否当前版本
- 是否已发布

---

## 11.2 返工交互建议

当操作者点击“需调整”时，不建议只弹一个确认框，建议让其选择返工方向：

- 仅重生成文案
- 仅重新合成视频
- 替换封面
- 修改话题
- 手工编辑当前版本并另存为新版本

这样检查结论就能直接驱动后续动作。

---

## 12. 不变量（Invariants）

建议后端强约束以下规则：

1. 每条 `CheckRecord` 必须绑定 `creative_item_id + creative_version_id`
2. `approve` 只能作用于当前版本
3. 若当前版本变化，则旧版本的“可发布结论”不自动继承
4. `READY_TO_PUBLISH` 必须存在最近一次 `CheckRecord.result = approved`
5. `REWORK_REQUIRED` 后，必须通过新版本重新进入检查流
6. `PUBLISHED` 必须可追溯到某个被批准过的版本

---

## 13. 渐进式落地顺序

### Step 1

新增：

- `check_records`
- 作品状态字段
- 当前版本与检查记录绑定

### Step 2

在作品工作台加入：

- 通过
- 需调整
- 放弃
- 历史版本列表

### Step 3

接入返工动作：

- 重生成文案
- 重合成视频
- 替换封面

### Step 4

发布入口改为：

- 仅对 `APPROVED / READY_TO_PUBLISH` 作品开放
- 发布时绑定具体版本

---

## 14. 一句话定义

> 人工检查流解决“这版到底能不能发”的业务判断，版本管理解决“到底是哪一版被检查、被确认、被发布”的追踪问题；两者一起，才能让桌面工作台从执行系统升级为内容工作流系统。
