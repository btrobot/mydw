## 1. 文档目标

本文档用于说明：

1. 当前 `TaskScheduler + PublishService` 的运行方式；
2. 为什么下一阶段需要新增“发布池（Publish Pool）”概念；
3. 发布池如何与作品层、人工检查流、版本管理协同；
4. 调度器如何从“选 Task”升级为“选可发布作品版本”；
5. 如何在不破坏现有 MVP 的前提下渐进改造。

---

## 2. 现状结论先行

### 2.1 当前 MVP 的真实发布模型

当前系统本质上是：

> **调度器直接从 `Task.status = ready` 的任务里挑一个，按优先级和时间窗口执行发布。**

### 2.2 当前实现优点

- 实现简单
- 易于快速跑通发布主链路
- 已支持：
  - 调度配置
  - 时间窗口
  - 单账号日限额
  - shuffle
  - auth gate

### 2.3 当前实现问题

但当前模型无法很好支持：

- 作品层人工检查
- 版本化发布
- 只发布“被确认可发布的版本”
- 对失败作品进行内容返工后再回到发布队列

因此需要引入：

> **发布池（Publish Pool）= 当前所有可发布作品版本的统一候选集合**

---

## 3. 当前调度器实现摘要

## 3.1 当前挑选逻辑

当前 `PublishService.get_next_task()` 的核心逻辑是：

1. 读取 `ScheduleConfig`
2. 校验当前时间是否在允许发布时间范围内
3. 从 `Task.status == ready` 中按 `priority desc, created_at asc` 取 1 条
4. 再检查账号当日发布上限
5. 选中后由 `publish_task()` 执行上传

## 3.2 当前调度真相

当前 runtime truth 已比较明确：

- 调度配置真相：`ScheduleConfig`
- 发布运行状态真相：`TaskScheduler runtime state`
- `/api/publish/status` 是派生视图，不是独立真相

## 3.3 当前限制

当前调度器并不知道：

- 这条任务属于哪条作品
- 对应的是哪个作品版本
- 是否经过人工检查
- 是否只是“执行 ready”，但业务上并不应该发布

---

## 4. 为什么必须新增发布池

## 4.1 “Task ready” 不等于“业务可发布”

在旧模型里：

- `task.status = ready` 表示执行层准备好了

但在新模型里，业务上至少还要满足：

- 对应作品版本完整
- 已经过人工检查
- 检查结论为可发布
- 当前版本仍然是作品主版本

因此：

> **执行 ready 只是技术前置条件，不是业务放行条件。**

## 4.2 发布池的作用

发布池用于把“执行条件”和“业务条件”组合起来。

它回答的是：

- 哪些作品现在可以发布？
- 每条作品当前应该发布哪一版？
- 哪些作品虽然有 ready task，但还不应该发？
- 哪些作品因为检查结论失效，应退出发布池？

## 4.3 发布池和调度器的关系

- **发布池**：解决“候选集是什么”
- **调度器**：解决“从候选集里先发谁”

---

## 5. 发布池定义

## 5.1 一句话定义

> 发布池是由“已被确认可发布的作品当前版本”构成的可调度候选集合。

## 5.2 推荐最小字段

建议新增 `publish_pool_items` 表，或用等价投影实现。

### 推荐字段

| 字段 | 类型 | 说明 |
|------|------|------|
| id | bigint | 主键 |
| creative_item_id | bigint | 作品 |
| creative_version_id | bigint | 当前可发布版本 |
| account_id | bigint nullable | 计划发布账号 |
| profile_id | bigint nullable | 作品配置档 |
| pool_status | varchar | pending/locked/published/failed/removed |
| priority | int | 调度优先级 |
| scheduled_after | datetime nullable | 最早可发布时间 |
| last_check_record_id | bigint | 最近一次检查结论 |
| publishability_snapshot | json | 入池时的可发布校验结果 |
| source_task_id | bigint nullable | 当前版本对应的执行 Task |
| locked_at | datetime nullable | 被调度器锁定时间 |
| locked_by | varchar nullable | 锁定来源 |
| last_error_message | text nullable | 最近失败原因 |
| created_at | datetime | 创建时间 |
| updated_at | datetime | 更新时间 |

---

## 5.3 入池条件

一条作品版本要进入发布池，建议至少满足：

1. `CreativeItem.status == APPROVED` 或 `READY_TO_PUBLISH`
2. `CreativeItem.current_version_id == creative_version_id`
3. 存在最近一次 `CheckRecord.result = approved`
4. 该版本的 `PackageRecord.completeness_status == complete`
5. 存在有效发布账号
6. 满足当前 direct publish / composition 语义约束
7. 当前未被标记为 `REJECTED / ARCHIVED`

## 5.4 出池条件

以下情况应移出发布池或标记为无效：

1. 作品进入 `REWORK_REQUIRED`
2. 作品进入 `REJECTED`
3. 作品生成了新版本，旧版本不再是当前版本
4. 发布账号失效且无法补位
5. 当前授权状态不允许发布
6. 当前版本完整性检查失败

---

## 6. 发布池状态机

## 6.1 状态枚举建议

```text
PENDING
LOCKED
PUBLISHED
FAILED
REMOVED
```

### 含义

| 状态 | 含义 |
|------|------|
| PENDING | 在池中，等待调度 |
| LOCKED | 已被调度器选中，防止并发重复发布 |
| PUBLISHED | 对应版本已发布成功 |
| FAILED | 最近一次发布失败，待判断是否回池 |
| REMOVED | 不再属于池中候选 |

## 6.2 状态流转

```text
PENDING → LOCKED → PUBLISHED
PENDING → LOCKED → FAILED
FAILED → PENDING      （允许重回池）
PENDING → REMOVED     （版本失效/被返工/被放弃）
LOCKED → REMOVED      （取消/终止）
```

---

## 7. 调度器重构目标

## 7.1 当前调度器

当前调度器选择对象：

- `Task`

## 7.2 目标调度器

下一阶段调度器选择对象建议改为：

- `PublishPoolItem`

### 新调度流程建议

```text
1. 检查 auth runtime state
2. 读取 ScheduleConfig
3. 从 Publish Pool 选出 PENDING 候选
4. 根据时间窗口/优先级/账号限额/账号可用性筛选
5. 锁定一条候选
6. 找到其 creative_version_id 对应的发布输入
7. 触发发布执行 Task
8. 成功/失败后回写池状态和作品状态
```

---

## 7.3 为什么不能直接让调度器继续扫 Task

如果继续只扫 Task，会有这些问题：

1. 一个作品多个 Task，无法知道谁是当前应发布版本
2. 一条旧版本 Task 仍然是 ready，但业务上已失效
3. 人工检查通过的是版本 3，但调度器可能拿到版本 2 对应 Task
4. 无法优雅表示“批准了，但暂时不发 / 延后发 / 退回池”

因此调度器前面必须有发布池这层中间语义。

---

## 8. 发布输入模型建议

## 8.1 推荐新增 PublishExecutionSnapshot

调度器不应直接从零拼 Task，而应在发布前构造一个明确快照：

| 字段 | 说明 |
|------|------|
| creative_item_id | 作品 |
| creative_version_id | 版本 |
| video_path | 发布视频 |
| cover_path | 封面 |
| title | 标题 |
| content | 文案 |
| topic_text | 话题 |
| account_id | 发布账号 |
| profile_id | 配置档 |
| source_task_id | 关联 Task |

### 作用

- 避免发布时动态拼装导致语义漂移
- 提高日志可追溯性
- 为未来支持“重放发布”留入口

---

## 9. 与当前 Task 执行链的协同模式

## 9.1 过渡方案（推荐）

短期内不废弃 Task，而是：

- 发布池负责挑选作品版本
- 选中后创建或复用一个 publish Task
- 调度器仍调用现有 `PublishService.publish_task()`
- 只是发布输入从“直接扫 ready Task”变成“先从池中选，再映射到 Task”

## 9.2 推荐中间层

建议新增：

- `PublishPoolService`
- `PublishPlannerService`（可选）

### PublishPoolService 职责

- 入池
- 出池
- 重入池
- 锁定/解锁
- 可发布性校验

### PublishPlannerService 职责

- 从 pool item 生成发布执行快照
- 创建/绑定 publish Task
- 校验版本一致性

---

## 10. 与人工检查流、版本流的衔接

## 10.1 检查通过后入池

当操作者点击“可发布”时：

- 写 `CheckRecord(approved)`
- 作品状态 -> `APPROVED`
- 满足条件则自动入池 `PENDING`

## 10.2 返工后出池

当操作者点击“需调整”时：

- 写 `CheckRecord(rework_required)`
- 当前版本对应 pool item -> `REMOVED`
- 作品状态 -> `REWORK_REQUIRED`

## 10.3 新版本生成后旧池项失效

当生成版本 4 时：

- 版本 3 对应 pool item 应失效或移除
- 只有版本 4 在被检查通过后才能重新入池

## 10.4 发布失败后的处理

建议区分：

### 执行失败但版本仍有效

例如：
- 网络问题
- 页面打开失败
- 临时登录失效

处理：
- pool item -> `FAILED`
- 可人工或自动恢复到 `PENDING`

### 业务上已不应继续发布

例如：
- 版本已被新版本替代
- 作品已被 rework/rejected

处理：
- pool item -> `REMOVED`

---

## 11. 调度排序规则建议

## 11.1 排序因子

在发布池基础上，调度排序建议综合：

1. `priority`
2. `scheduled_after`
3. `created_at`
4. 账号日限额剩余额度
5. 账号当前可用性
6. 是否最近失败过

### 推荐默认排序

```text
priority DESC
scheduled_after ASC
created_at ASC
```

## 11.2 账号维度策略

当前已有：

- `max_per_account_per_day`
- shuffle

后续建议扩展：

- 账号健康度
- 最近失败率
- 账号冷却时间
- 账号负载均衡

## 14. 迁移路线建议

## Phase 1：池前置但调度器仍以 Task 为主

- 增加作品入池/出池逻辑
- UI 先出现“可发布池”视图
- 调度器仍可兼容 ready task

## Phase 2：调度器改为先选池项

- `TaskScheduler` 先查 publish_pool_items
- 锁定池项
- 再映射为 publish Task

## Phase 3：发布成功/失败彻底围绕作品版本回写

- pool item、creative、task 三方统一回写
- `/api/publish/status` 展示当前作品而不只是当前 task

## 15. 风险与取舍

## 风险 1：引入发布池会增加一层复杂度

### 应对

先把发布池做成最小投影，不一定一步到位做复杂策略系统。

## 风险 2：池状态、作品状态、Task 状态可能不一致

### 应对

- 明确真相边界：
  - 作品看 `CreativeStatus`
  - 执行看 `Task.status`
  - 发布候选看 `pool_status`
- 所有变更走统一 service 层

## 风险 3：旧 ready task 兼容期容易混淆

### 应对

在迁移期显式标记：

- 是否来自发布池
- 是否直连旧 Task 模式

---

## 16. 一句话定义

> 发布池负责定义“哪些作品版本现在允许被调度器挑选”，调度器负责定义“这些候选里先发谁”；两者分开后，作品层、检查流和执行层才能真正打通。
