# 作品工作台 / 任务管理 / Dashboard 职责边界

## 目的

这份文档用于澄清当前前端三类主页面的职责边界，避免把：

- 业务主对象
- 执行对象
- 系统运行对象

混在同一个入口里，导致用户心智混乱。

---

## 一句话结论

- **作品工作台**：主业务入口，面向“作品”处理。
- **任务管理**：执行 / 合成 / 上传 / 失败排查入口，面向“任务”处理。
- **Dashboard**：运行态与系统概览入口，面向“系统状态”查看。

三者不是重复关系，而是**分层协作关系**。

---

## 1. 作品工作台

### 核心定位

作品工作台是当前系统的**默认业务主入口**。

它承接的是“作品域”主线，而不是任务执行诊断。

### 主要回答的问题

用户进入作品工作台，首先应该回答：

1. 今天有哪些作品需要处理？
2. 哪些作品待审核？
3. 哪些作品待补充？
4. 哪些作品已经进入发布池？
5. 当前作品版本是否与发布池版本对齐？
6. 我下一步应该进入哪个作品做详情、审核、返工或 AIClip？

### 它处理的对象

- Creative
- Creative Version
- Review Summary / Review Record
- Publish Pool 状态投影

### 它应该提供的动作

- 搜索作品
- 筛选作品状态
- 查看作品详情
- 进入审核
- 进入 AIClip
- 判断是否具备进入后续发布链路的条件

### 不应该承担的职责

- 不负责展示任务执行细节为主视图
- 不负责承担系统运行监控入口
- 不负责成为错误堆栈/失败重试的首要入口

### 用户心智

用户看到作品工作台时，应该理解为：

> “这是我每天进入系统后处理业务内容的地方。”

---

## 2. 任务管理

### 核心定位

任务管理是**执行链路与诊断链路入口**。

它面向的是 Task，而不是 Creative。

### 主要回答的问题

用户进入任务管理，首先应该回答：

1. 哪些任务待合成？
2. 哪些任务合成中？
3. 哪些任务待上传？
4. 哪些任务失败了？
5. 当前失败卡在哪个阶段？
6. 我应该提交合成、立即发布、快速重试，还是取消/删除？

### 它处理的对象

- Task
- CompositionJob
- 发布状态流转
- 失败阶段 / 重试次数 / 错误信息

### 它应该提供的动作

- 提交合成
- 取消合成
- 立即发布
- 快速重试
- 重置为待合成
- 取消任务
- 删除任务
- 查看任务详情

### 不应该承担的职责

- 不应该成为作品审核的主入口
- 不应该承接作品版本管理
- 不应该成为系统全局健康页

### 用户心智

用户看到任务管理时，应该理解为：

> “这是我处理执行链路、失败链路和任务动作的地方。”

---

## 3. Dashboard

### 核心定位

Dashboard 是**系统运行概览入口**，不是业务处理主入口。

它用于展示系统当前是否正常运行，而不是承接具体业务处理。

### 主要回答的问题

用户进入 Dashboard，首先应该回答：

1. 发布器当前是否在运行？
2. 调度器模式是什么？
3. 发布状态是否健康？
4. 系统统计是否异常？
5. 最近日志里有没有明显问题？

### 它处理的对象

- Publish runtime status
- Scheduler mode
- System stats
- Logs
- 全局运行摘要

### 它应该提供的动作

- 查看运行状态
- 发现异常后跳转到作品工作台或任务管理
- 作为监控与辅助判断入口

### 不应该承担的职责

- 不应该承接作品逐条处理
- 不应该承接任务逐条操作
- 不应该替代作品工作台成为业务首屏

### 用户心智

用户看到 Dashboard 时，应该理解为：

> “这是系统运行看板，不是我日常处理作品的主工作台。”

---

## 4. 三者关系

## 4.1 对象层级

- **作品工作台**：业务对象层
- **任务管理**：执行对象层
- **Dashboard**：系统运行层

## 4.2 正常工作流

推荐主链路：

1. 从**作品工作台**进入
2. 找到今天要处理的作品
3. 进入作品详情 / 审核 / AIClip
4. 当进入执行阶段后，再到**任务管理**处理任务动作
5. 需要看系统健康和调度状态时，再去**Dashboard**

## 4.3 页面跳转关系

- **作品工作台 → 任务管理**
  - 查看某个作品关联任务的执行情况
- **任务管理 → 作品工作台 / 作品详情**
  - 回到业务主对象上下文
- **Dashboard → 作品工作台**
  - 回到主业务入口
- **Dashboard → 任务管理**
  - 进入任务级排障

---

## 5. 为什么不能混成一个页面

如果把三者混成一个页面，会出现几个问题：

1. **业务对象和执行对象混淆**
   - 用户不知道自己当前在处理作品，还是在处理任务

2. **首屏信息噪音过大**
   - 作品、发布池、日志、调度器、失败重试全堆在一起，优先级丢失

3. **主动作不明确**
   - 用户不知道第一步该审作品，还是该重试任务，还是该看运行状态

4. **责任边界不清**
   - 页面既像工作台，又像监控页，又像诊断页，最后三种都做不好

---

## 6. 当前推荐的信息架构

### 一级主入口

1. **作品工作台**（默认业务入口）
2. **任务管理**（执行与诊断入口）
3. **Dashboard**（运行监控入口）

### 角色分工

- **作品工作台**
  - 主业务面
  - 面向内容处理

- **任务管理**
  - 动作面
  - 面向执行状态与任务操作

- **Dashboard**
  - 监控面
  - 面向系统与运行时状态

---

## 7. 最终判断

按当前项目的重构方向：

- 主业务真相正在从 Task 迁移到 Creative / Version / Review / Publish Pool
- 因此**作品工作台作为默认业务入口是合理的**
- 任务管理应该收口为执行与诊断位
- Dashboard 应继续收口为监控位

也就是说，推荐保持以下原则不变：

> **业务先看作品工作台，执行排障再进任务管理，系统健康再看 Dashboard。**
