# 作品驱动创作流程重构建议

## 1. 背景

基于当前实现，系统已经具备以下能力：

- 素材管理
- 任务创建
- 本地合成 / 执行链路
- 作品版本回写
- 审核
- 发布池
- 发布任务执行

但当前主线更接近：

> 素材 → 任务 → 合成 → 回写作品 → 审核 → 发布

这条链路在工程实现上是成立的，但在用户心智上会产生明显割裂：

- 用户以为自己在“做作品”
- 系统却要求用户先“创建任务”
- 作品直到合成后才真正出现

因此，当前问题不是功能缺失，而是：

> **缺少一个从创作开始就存在，并贯穿全过程的“作品主对象”。**

---

## 2. 当前问题诊断

## 2.1 当前模型

当前系统里：

- **Task** 是执行主对象
- **Creative / CreativeVersion** 更像执行结果沉淀后的业务对象

也就是说，作品域目前更偏向：

- 成片结果
- 审核结果
- 发布准备结果

而不是：

- 创作过程本身

## 2.2 用户为什么会觉得割裂

因为用户自然会按下面的方式理解系统：

1. 我要创建一个作品
2. 我要给这个作品加素材
3. 我要选择这个作品如何合成
4. 我要查看这个作品的合成结果
5. 我要继续修改这个作品
6. 我要审核并发布这个作品

但当前系统让用户经历的是：

1. 先准备素材
2. 再创建任务
3. 再执行任务
4. 合成成功后作品才出现

这会造成几个典型问题：

- “我到底是在创建作品，还是在创建任务？”
- “我合成的是任务还是作品？”
- “为什么作品工作台不能承接创作起点？”
- “为什么作品只在合成后才出现？”

---

## 3. 重构目标

本次建议的目标不是推翻执行体系，而是把前台产品主线收口成：

> **作品驱动全过程，任务作为作品下的执行动作存在。**

也就是把系统统一成三层：

### 3.1 顶层：作品

作品是用户真正理解和操作的主对象。

它贯穿：

- 创作起点
- 素材组织
- 合成配置
- 版本产出
- 审核
- 发布准备
- 发布结果

### 3.2 中层：作品版本

版本是作品在某个时刻的产出快照。

它承接：

- 合成结果
- AIClip 结果
- 文案 / 封面 / 话题打包结果
- 审核状态
- 是否已进入发布池

### 3.3 底层：任务

任务不再作为用户第一心智对象，而是作品生命周期中的执行动作：

- 合成任务
- 发布任务
- AIClip 任务
- 重试任务

也就是说：

> **作品管业务主线，版本管结果沉淀，任务管执行动作。**

---

## 4. 目标心智模型

## 4.1 用户视角

用户应该始终觉得自己在处理“作品”：

1. 新建作品
2. 给作品添加素材
3. 选择合成配置
4. 提交合成
5. 查看作品的新版本
6. 审核作品
7. 进入发布准备
8. 提交发布
9. 查看发布结果

## 4.2 系统视角

系统内部仍然可以保持现有执行框架：

1. 创建 CreativeItem
2. 保存素材选择与配置
3. 创建 composition task
4. 执行合成
5. 成功后生成 CreativeVersion
6. 审核
7. 进入 publish pool
8. 创建 publish task
9. 执行发布

这样可同时满足：

- 用户主线连续
- 后端执行模型稳定

---

## 5. 建议的领域模型

## 5.1 作品（CreativeItem）

建议把 CreativeItem 前置为真正的“作品单 / 创作单”。

它至少应承接：

- 标题
- 创作类型
- 当前阶段
- 当前配置档
- 当前选中的素材摘要
- 当前版本
- 最近执行任务摘要
- 发布状态摘要

### 建议新增的语义

CreativeItem 不应只表示“成片后对象”，而应表示：

> **从开始创作到最终发布的全过程业务对象**

---

## 5.2 作品版本（CreativeVersion）

CreativeVersion 继续保留，但语义更明确：

- 每次合成成功产生一个版本
- 每次 AIClip 成功产生一个版本
- 每次人工重新包装，也可以形成一个版本

版本用于回答：

- 这个作品当前能审核的是哪个版本
- 当前发布池对应的是哪个版本
- 历史上有哪些迭代版本

---

## 5.3 任务（Task）

Task 明确降级为“执行记录对象”。

它负责：

- 调用合成链路
- 调用发布链路
- 记录执行状态
- 记录失败原因
- 提供重试能力

Task 不应再成为用户理解创作主线的第一入口。

---

## 6. 建议的状态机

## 6.1 作品状态机

建议把作品状态机前置并完整化：

```text
PENDING_INPUT        待补充素材
READY_TO_COMPOSE     可提交合成
COMPOSING            合成中
WAITING_REVIEW       待审核
REWORK_REQUIRED      待返工
APPROVED             审核通过
IN_PUBLISH_POOL      已入发布池
PUBLISHING           发布中
PUBLISHED            已发布
FAILED               存在待处理失败
```

说明：

- `PENDING_INPUT` / `READY_TO_COMPOSE` 解决“作品一开始不存在”的问题
- `COMPOSING` / `FAILED` 让作品可以感知任务执行态
- `WAITING_REVIEW` 之后继续沿用现有审核/发布语义

## 6.2 任务状态机

任务仍保持执行态语义：

```text
draft -> composing -> ready -> publishing -> success / failed / cancelled
```

任务状态不再承担完整业务含义，只承担执行语义。

---

## 7. 页面职责重构建议

## 7.1 作品工作台

重构后的作品工作台应成为：

> **创作全过程主入口**

应支持：

- 新建作品
- 查看作品阶段
- 查看素材是否齐备
- 查看是否可提交合成
- 查看最近一次合成结果
- 进入作品详情
- 进入审核
- 进入发布准备

它不应该只展示“合成后的审核对象”，而应贯穿从草稿到发布的全过程。

## 7.2 作品详情

作品详情应成为全过程编排页：

- 基本信息
- 素材区
- 合成配置区
- 当前版本区
- 执行记录区
- 审核记录区
- 发布记录区

这里才应该承接：

> “我要怎么把这个作品继续往前推进？”

## 7.3 任务管理

任务管理应明确降级为：

> **执行中心 / 排障中心**

它主要处理：

- 合成任务执行
- 发布任务执行
- 失败重试
- 错误排查

不再承担创作起点职责。

## 7.4 Dashboard

Dashboard 继续保持：

- 运行态总览
- 调度器 / 发布器健康情况
- 系统摘要

不承接创作主线。

---

## 8. 推荐的用户主流程

建议把前台主流程统一为：

```text
新建作品
  -> 补充素材
  -> 选择合成配置
  -> 提交合成
  -> 生成作品版本
  -> 审核通过
  -> 进入发布池
  -> 提交发布
  -> 发布完成
```

用户看到的始终是“作品”在前进。

系统底层则变成：

```text
CreativeItem
  -> composition task
  -> CreativeVersion
  -> review
  -> publish pool
  -> publish task
  -> publish result
```

---

## 9. 重构策略

## 9.1 不建议推翻后端执行层

当前后端的任务执行、版本回写、审核、发布池、发布任务克隆机制已经基本成形。

因此不建议：

- 推翻 Task 体系
- 推翻 CreativeVersion 体系
- 推翻发布池 / 发布任务链路

## 9.2 建议做“前置作品、后置任务”的收口

更合适的策略是：

### 第一层：语义重构

把 CreativeItem 从“合成结果对象”前置为“创作主对象”。

### 第二层：UI 重构

把“创建任务”改写成“从作品发起合成任务”。

### 第三层：关系重构

确保新建作品时就创建 CreativeItem，并允许：

- 先挂素材
- 先保存草稿
- 后创建任务

### 第四层：投影重构

让作品状态能够感知最近任务执行状态，避免用户必须跳到任务页才能理解作品当前阶段。

---

## 10. 最小可行落地方案

如果希望低风险演进，建议优先落这几个动作：

### MVP-1：新建作品即创建 CreativeItem

不再等合成成功后才有作品。

### MVP-2：素材与配置挂到作品上

让作品可以先处于：

- 待补充
- 可合成

### MVP-3：从作品详情发起合成

点击“提交合成”时，后台再创建 composition task。

### MVP-4：合成成功生成新版本

保留现有 CreativeVersion 机制。

### MVP-5：任务页只做执行与排障

任务管理页不再承担“用户先从这里开始创作”的暗示。

---

## 11. 预期收益

完成上述重构后，系统会获得几个直接收益：

### 11.1 用户心智统一

用户始终围绕“作品”工作，不需要在“素材 / 任务 / 作品”之间自己拼流程。

### 11.2 页面职责更清晰

- 作品工作台：业务主线
- 作品详情：全过程编排
- 任务管理：执行与排障
- Dashboard：系统监控

### 11.3 执行层仍可复用

不用推翻现有：

- TaskAssembler
- CompositionService
- CreativeGenerationService
- CreativeReviewService
- PublishPlannerService

### 11.4 后续扩展更自然

比如以后增加：

- 一键重新合成
- 版本对比
- 人工微调后再次提交审核
- 多轮 AIClip

都会更自然地挂在“作品主线”下面。

---

## 12. 风险与注意事项

## 12.1 不要把“作品”和“版本”混成一个概念

作品是全过程对象，版本是阶段性产出。

## 12.2 不要让“任务管理”重新长回业务入口

任务页应保持执行中心定位。

## 12.3 不要把作品状态完全等同于任务状态

作品状态是业务阶段，任务状态是执行阶段，两者应投影关联，但不能混为一谈。

## 12.4 要允许“作品存在但尚未产出版本”

这是本次重构最关键的变化之一。

---

## 13. 最终建议

当前系统的问题，不是没有作品域，而是：

> **作品域出现得太晚，没能成为创作全过程的主线对象。**

因此推荐的方向不是“废掉任务体系”，而是：

> **把作品前置成创作主对象，把任务收口成作品下的执行动作。**

用一句话总结这次重构建议：

> **用户看作品，系统跑任务，结果沉淀为版本。**

这会比“先任务、后作品”的现状更符合创作类产品的用户心智，也更适合作为后续 UI/UX 收口和 IA 收口的基础。
