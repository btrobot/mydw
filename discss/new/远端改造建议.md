
  现状诊断                                                                                                                         
  远端的问题不是一个问题，是两层问题：                                                                                             
  前端：完全没有框架
    ├── 2,568 行 TypeScript 做字符串拼接 HTML
    ├── 没有组件系统，一个 renderApp() 函数渲染所有页面
    ├── 样式全部内联在字符串里
    └── 直接操作 DOM，window.confirm() 做确认弹窗

  后端：MVP 质量
    ├── 架构分层正确（routes → services → repositories）
    ├── 但几乎没有测试
    ├── 内存限流（多进程/多服务器就失效）
    └── 自定义迁移（无 Alembic）

  后端架构本身没问题，需要的是补测试、换限流、加迁移工具。真正需要重做的是前端。

  ---
  前端改造：最佳技术栈选择

  这是一个后台管理系统（Admin Panel），功能已定：用户/设备/会话/审计日志管理，6 个页面，CRUD 操作为主。

  推荐栈

  React 18 + TypeScript
  Ant Design Pro（或 Refine）
  React Query（@tanstack/react-query）
  React Router 6
  Vite

  为什么和本地端用同一套？

  - 组件库统一，Admin 和本地 Electron 前端可以共享 UI 组件
  - 团队只需维护一套技术选型
  - 已有的 OpenAPI → SDK 生成流程直接复用

  但有一个更好的选项值得认真考虑

  Refine（refine.dev）

  Refine 是专为 Admin Panel / Dashboard 场景设计的 React 框架，底层用 Ant Design，上层封装了 CRUD 页面的所有通用逻辑：

  // 一个完整的用户列表页，包含搜索/筛选/分页/详情：
  <List>
    <Table>
      <Table.Column dataIndex="username" title="用户名" />
      <Table.Column dataIndex="license_status" title="授权状态" />
      <Table.Column title="操作" render={(_, record) => (
        <EditButton recordItemId={record.id} />
      )} />
    </Table>
  </List>

  对比你们现在 1,288 行字符串拼接实现同样的功能。

  Refine 的核心价值：

  ┌────────────────────┬──────────┬───────────────────────┐
  │        功能        │   手写   │        Refine         │
  ├────────────────────┼──────────┼───────────────────────┤
  │ 列表 + 分页 + 筛选 │ ~200 行  │ <List>                │
  ├────────────────────┼──────────┼───────────────────────┤
  │ 详情页             │ ~150 行  │ <Show>                │
  ├────────────────────┼──────────┼───────────────────────┤
  │ 编辑表单 + 验证    │ ~200 行  │ <Edit> + useForm      │
  ├────────────────────┼──────────┼───────────────────────┤
  │ 权限控制（RBAC）   │ 自己实现 │ accessControlProvider │
  ├────────────────────┼──────────┼───────────────────────┤
  │ 审计/操作确认      │ 自己实现 │ useCustomMutation     │
  └────────────────────┴──────────┴───────────────────────┘

  你们的后台正好是 Refine 的甜区：有清晰的资源定义（users/devices/sessions/audit-logs），有
  RBAC（super_admin/auth_admin/support_readonly），有 CRUD 操作。

  ---
  后端改造

  后端架构方向对，改造点较少：

  必须改

  1. 测试覆盖
     pytest + pytest-asyncio，补 admin_service 的核心逻辑测试
     至少：用户管理、设备操作、权限校验

  2. 迁移工具
     换成 Alembic（远端用 PostgreSQL 生产环境，Alembic 是标配）
     现有自定义迁移可以做一次性基线 migration

  3. 限流
     换成 slowapi（基于 Redis 或 token bucket）
     或直接在 Nginx/反向代理层限流

  可以改（提升生产就绪度）

  4. 分页一致性
     所有列表接口统一返回 { items, total, page, page_size }

  5. 密码策略
     现在用 pbkdf2_sha256，可升级到 bcrypt 或 argon2

  6. 管理员操作双确认
     高危操作（revoke user）走 TOTP 或二次密码确认
     （window.confirm 在前端改掉就够了，后端不需要变）

  不需要改

  × FastAPI 框架本身——没问题
  × SQLAlchemy 2.0——正确选择
  × 分层架构（routes/services/repositories）——保留
  × Pydantic schemas——保留
  × Bearer token 认证——保留
