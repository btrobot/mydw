# 文件级迁移能不能解决 UI 不漂亮（2026-04-26）

## 结论

**不能直接解决，但能为解决 UI 不漂亮创造前提。**

文件级迁移主要解决的是：

- 结构乱
- 职责乱
- 复用乱
- 页面模式不统一

它**不直接解决**：

- 视觉层级弱
- token 不统一
- 间距/密度不稳
- 壳层不够高级
- 页面模板不统一
- 空态/错误态/危险操作区不够产品化

---

## 它能解决什么

### 1. 让 UI 改造有“落点”

如果目录和职责混乱，你很难系统性改 UI。

例如：

- AppShell 放哪？
- shared 组件放哪？
- 页面模式组件放哪？
- danger zone 放哪？

文件迁移能先把这些位置理顺。

### 2. 让“统一改造”变可能

UI 不漂亮，通常不是某一页问题，而是：

- 每页各写各的
- hooks 各挂各的
- 组件层级乱

重组后才能做：

- 统一 shell
- 统一 token
- 统一状态组件
- 统一 master-detail 模板

### 3. 降低后续美化成本

如果不先重组，后面你每改一个 UI，都像做一次局部手术。

如果先重组，后面就能按域、按模板、按 shared component 改。

---

## 它不能解决什么

### 1. 不能自动让页面“更高级”

目录变漂亮，不等于界面变漂亮。

### 2. 不能自动产生设计系统

你还是得补：

- token
- shell 设计
- 页面模板
- typography hierarchy
- 状态组件

### 3. 不能自动修复“产品感不足”

比如：

- 空态像 demo
- readonly 只是 disabled
- danger zone 不明显
- 页面信息密度不对

这些都还是 UI/交互设计问题。

---

## 对你当前项目的准确判断

**文件级迁移不是 UI 改造的终点，而是 UI 改造的前置工程。**

如果只做迁移，不做：

- theme/token 收口
- AppShell 提升
- 页面模板统一
- shared 状态组件补齐

那 UI 还是不会本质变漂亮。

但如果不迁移，后面这些事又会做得很痛苦。

---

## 最合理的做法

不是二选一，而是两段式：

### 第一段：先做结构重组

- app/
- features/
- lib/
- shared components

### 第二段：在新结构上做 UI 系统化改造

- 统一 theme token
- 升级 AppShell
- 抽 PageHeader / LoadingState / ErrorState / EmptyState
- 抽 master-detail 模板
- 再逐页 polish

---

## 一句话总结

> **文件级迁移不能直接让 UI 变漂亮，但它能把“漂亮而一致的 UI 改造”从困难模式，变成可执行模式。**
