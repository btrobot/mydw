# 执行档位建议：从 `ralplan -> ralph` 默认链路，切换为按风险分级

> 日期：2026-04-23
> 背景：针对“过去一直走 `ralplan -> ralph`，很严谨但代价太大、速度太慢”的讨论整理

## 结论

我同意这个判断：**`ralplan -> ralph` 不应该是默认流程，而应该是升级流程。**

这套链路的优点是：

- 边界清
- 风险低
- 留痕强
- 大改动不容易失控

但缺点也很明显：

- 前置成本高
- 文档 / 审查 / 验证链太长
- 对“小修、小调、小闭环”明显过载

因此，更合适的方式不是“一律重流程”，而是：**按风险分级执行。**

---

## 建议的新默认策略

### L0：直接做

适用场景：

- 文案修改
- UI 微调
- 小 bug 修复
- 单文件或少量文件修复
- 已有明确报错 / 明确目标 / 明确触点

执行方式：

- 直接修改
- 跑必要验证
- 汇报结果

**不进 `ralplan`，不启 `ralph`。**

### L1：短计划 + 直接执行

适用场景：

- 中小改动
- 涉及 2~6 个文件
- 需要先定一下顺序，但不值得出完整 PRD / test-spec

执行方式：

- 先给 3~6 条短计划
- 直接实现
- 跑 targeted verification

**这应成为默认档位。**

### L2：轻量 `ralplan`

适用场景：

- 有一定边界或方案分歧
- 会碰到接口、状态流、页面结构
- 但还没到“高风险系统工程”

执行方式：

- 做短版共识计划
- 不展开完整重文档，不跑长 deliberation
- 计划清楚后直接执行

**核心是借用 `ralplan` 的“先想清楚”，而不是吃满全部成本。**

### L3：完整 `ralplan -> ralph`

仅在这些场景启用：

- 认证 / 权限 / 安全
- migration / schema 变更
- 公共 API / 合同变更
- 跨前后端 / 跨系统联动
- 需要强验收闭环
- 明确要求“必须做完、不能漏、要严谨”

也就是说：

> 以后不是“先 `ralplan` 再 `ralph`”，  
> 而是“先判断风险，必要时才升级到 `ralplan` / `ralph`”。

---

## 为什么这个调整适合当前项目

当前项目已经不在早期混沌探索期，而是处于：

- 主架构已成型
- auth 主线已经收口很多
- creative domain 最近也完成了一轮大收口
- 后续任务里，越来越多会是局部修复、体验优化、边界清理

在这个阶段，如果所有事情都走满 `ralplan -> ralph`，会明显拖慢吞吐。

更合理的策略是：

- **轻执行**
- **重验证**
- 但**不默认重流程**

---

## 推荐的实际协作规则

默认优先级建议如下：

1. **能直接做就直接做**
2. 需要一点结构化就给短计划
3. 只有高风险才上 `ralplan`
4. 只有需要强闭环时才上 `ralph`

---

## 一个实用判断表

### 直接做

示例：

- “把这个页面按钮文案改短一点”
- “修这个空态”
- “这个筛选逻辑不对”
- “这个测试挂了，修掉”

### 短计划后做

示例：

- “把工作台筛选再整理一下”
- “调整详情页信息层级”
- “把这个模块顺手去一点技术债”

### 完整 `ralplan`

示例：

- “重做登录状态流”
- “改 creative 核心数据模型”
- “改 schema / migration / contract”
- “做跨前后端的发布流程重构”

### `ralph`

示例：

- “这一轮必须完整收口”
- “直到验证全绿再停”
- “这是阶段性关闭项”

---

## 最终建议

后续默认按下面的节奏协作：

- **小事：直接推进**
- **中事：短计划后推进**
- **大事：才用 `ralplan`**
- **高风险闭环：才用 `ralph`**

一句话概括：

> **新默认应当是：轻执行、重验证，而不是重流程。**
