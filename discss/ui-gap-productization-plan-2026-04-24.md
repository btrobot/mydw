# 当前页面与期待差距：产品化收口方案

> 日期：2026-04-24  
> 背景：基于当前仓库现状、既有 UI/UX 评审文档，以及“页面功能实现距离期待较大，如何解决”的讨论整理  
> 性质：讨论稿 / 执行前共识稿

---

## 1. 一句话结论

当前页面与期待之间的差距，本质上不是“再补几个功能点”就能解决，而是：

> **当前界面更像“已跑通的工程过渡态 / 诊断台”，还不是“收口后的产品界面”。**

因此，解决办法不应是继续零散补功能，而应采用：

> **产品化收口法：先定义目标形态，再按主线逐步收口。**

---

## 2. 核心判断：差距到底在哪里

当前系统已经完成了从 Task-first 向 Creative-first 的主路径迁移，主业务方向是对的；但“能跑通”不等于“达到了期待中的产品完成度”。

当前差距主要不在“有没有页面”，而在下面几类问题没有彻底收口：

1. **信息架构没有完全靠结构自解释**
2. **业务层与诊断层仍有混放**
3. **Workbench 还不够像真正日常可用的工作台**
4. **文案、四态、交互层级、响应式等产品完成度不足**

也就是说：

> **当前问题不是单点缺功能，而是页面职责、信息层级、交互优先级和产品完成度还没有被统一收成一套稳定界面。**

---

## 3. 为什么会感觉“距离期待很大”

当用户说“页面实现离期待差很多”时，往往不是因为某一个按钮没做，而是因为以下几类差距叠加在一起：

### 3.1 信息架构差距

- 页面职责不够清晰
- 还需要靠说明型 `Alert` 告诉用户“这个页是什么”“那个页不是主入口”
- 用户理解系统仍依赖解释，而不是依赖结构本身

### 3.2 页面分层差距

- 默认业务页里混入较多研发/诊断/排障信息
- 用户视角的“我该做什么”不够突出
- 工程视角的“系统现在怎么跑”占据了过多可见空间

### 3.3 工作面可管理性差距

- 能浏览，但不一定能高效管理
- 找作品、定位待处理项、切换高频视角的效率还不够高
- 当数据量上来后，界面容易从“可用”退化成“凑合能看”

### 3.4 产品完成度差距

- 文案不统一
- loading / empty / error / success 四态不统一
- 主次按钮层级不够稳定
- 布局在窄窗口/非最大化窗口下的表现还不够稳

---

## 4. 正确的解决方式：不要想到哪补到哪

不建议继续按“看见哪不爽就补哪”的方式推进。

原因很简单：

- 这种方式会持续放大局部复杂度
- 会让页面越来越像“叠加后的历史产物”
- 容易一边做 UI，一边不受控地混入模型、接口、诊断逻辑改动
- 最终既没形成产品界面，也没形成可维护结构

因此应该采用：

> **先定义期待中的产品形态，再把差距拆成几条主线，按主线逐步收口。**

---

## 5. 建议采用的方法：产品化收口法

## 5.1 第一步：先把“期待”变成明确目标

先不要只说“差很多”，而要把每个核心页面都明确成下面四个问题：

1. **这个页面给谁用**
2. **用户来到这里最主要要完成什么**
3. **首屏必须看到什么**
4. **哪些信息不应该默认出现**

例如：

### CreativeWorkbench

应该是：

- 日常分诊台
- 快速定位待处理作品
- 快速筛选 / 搜索 / 排序 / 进入详情

不应该是：

- 工程诊断信息堆叠页
- 靠解释文案才能理解用途的页面

### CreativeDetail

应该是：

- 定义作品
- 审核当前版本
- 决定是否可发布

不应该是：

- 默认承载过多 scheduler / shadow diff / raw diagnostics

---

## 5.2 第二步：把差距拆成 4 类，不要混着做

建议统一按以下四类来审视当前界面：

### A. 信息架构问题

- 页面职责是否清楚
- 导航是否自然表达主语义
- 是否还依赖说明型 `Alert`

### B. 业务层 / 诊断层混放问题

- 默认页是否暴露了太多研发视角信息
- 诊断信息是否应该下沉到高级入口

### C. 可管理性问题

- 搜索、筛选、排序、分页是否足够支撑日常操作
- 是否有高频 preset
- 是否支持恢复上下文 / 分享视图

### D. 产品完成度问题

- 文案是否统一
- 四态是否统一
- 主 CTA 是否清晰
- 布局和响应式是否稳定

---

## 5.3 第三步：按主线收口，不按页面乱修

建议采用如下顺序，而不是一次性大改整套前端：

### 主线 1：Workbench 可管理性收口

优先解决：

- 搜索
- 状态筛选
- 排序
- 分页
- 高频 preset
- URL 状态可恢复 / 可分享

原因：

- 这是最直接影响日常使用效率的界面
- 它的收益最大
- 它最适合作为“产品化收口”的第一站

### 主线 2：业务层 / 诊断层彻底分层

把下面这些内容下沉到高级诊断层：

- scheduler
- shadow diff
- kill switch
- 原始 diff / raw JSON
- 运行态诊断细节

默认业务层只保留用户真正用于干活的信息和操作。

### 主线 3：文案与四态统一

统一：

- 页面标题
- 按钮文案
- 空态 / 错态 / loading / success 表达
- 中英文策略

### 主线 4：详情页 / AIClip 的进一步产品化

等前面三条稳定后，再继续做更深层的工作流体验优化。

---

## 6. 落地时建议使用的工件

为了避免“只讨论，不落地”，建议后续配套以下工件：

## 6.1 页面期待清单

针对每个核心页面写：

- 页面目标
- 首屏必须内容
- 主按钮
- 次级按钮
- 不应默认出现的信息
- success / empty / error / loading 的目标形态

## 6.2 Gap Audit（差距清单）

建议整理成如下结构：

| 页面 | 期待形态 | 当前差距 | 优先级 |
| --- | --- | --- | --- |
| Workbench | 快速定位待处理作品 | 管理能力不足 / 诊断信息偏多 | P0 |
| Detail | 清晰定义作品与审核 | 职责过载 / 默认层级偏重 | P0 |
| AIClip | 更像产品流程而非工具面板 | 低层参数暴露较多 | P1 |
| Dashboard/Auth | 角色清楚、表达统一 | 文案和层级不够一致 | P1 |

## 6.3 主线级 PRD / test spec

每一轮只打一条主线，并明确：

- 做什么
- 不做什么
- 验收标准
- 需要的 E2E / 页面验证

---

## 7. 执行原则

后续如果正式推进页面收口，建议坚持以下原则：

1. **先定页面职责，再动代码**
2. **先定信息层级，再做样式和组件替换**
3. **每次只打一条主线，不多线混做**
4. **不把 UI 收口和底层语义重构捆死**
5. **每轮都要有明确验证标准**

---

## 8. 不建议走的路

下面这些做法风险很高，不建议采用：

### 8.1 继续零散补功能

- 看到哪不爽改哪
- 今天补一个筛选，明天补一个 Alert，后天补一个按钮
- 没有统一目标形态

结果通常是：

- 页面越来越复杂
- 用户体验不统一
- 后续更难收口

### 8.2 一边做 UI，一边顺手大改底层模型

这会导致：

- 范围失控
- 回归范围暴涨
- UI 问题和数据模型问题纠缠在一起

### 8.3 一次性大重写整套前端

这听起来很痛快，但对当前仓库并不合适。

原因：

- 当前已有不少正确资产可以复用
- 主业务方向是对的
- 更适合 small diff / reversible 的渐进式收口

---

## 9. 最终建议

如果要真正解决“页面距离期待较大”的问题，应把它定义为：

> **一次产品化收口工程，而不是一次零散补功能。**

推荐顺序：

1. 先写页面期待清单
2. 再写 Gap Audit
3. 再按主线推进：
   - Workbench 可管理性
   - 业务层 / 诊断层分层
   - 文案与四态统一
   - 详情页 / AIClip 产品化
4. 每条主线独立验证、独立收口

---

## 10. 一句话收尾

这件事真正要解决的，不是“页面还差几个功能”，而是：

> **把当前“能跑的工程过渡态”，收成“符合预期的产品界面”。**
