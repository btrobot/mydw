# 页面期待清单（用于 UI 差距收口）

> 日期：2026-04-24  
> 前置讨论：`discss/ui-gap-productization-plan-2026-04-24.md`  
> 性质：执行前页面目标定义稿

---

## 1. 文档目的

本文件不讨论“代码怎么改”，先只回答：

1. 每个核心页面**到底应该承担什么职责**
2. 用户进入页面时，**首屏应该看到什么**
3. 页面里**什么信息不应该默认出现**
4. 后续做 Gap Audit 和 PRD 时，**以什么目标形态作为参照**

一句话：

> **先定义“理想中的页面应该长什么样”，再讨论怎么收口到那里。**

---

## 2. 页面分层总原则

当前产品建议按下面五层理解：

1. **创作工作台层**
   - 代表页面：`/creative/workbench`
   - 任务：找作品、分诊、进入详情
2. **作品定义层**
   - 代表页面：`/creative/:id`
   - 任务：定义作品、审核版本、决定是否可发布
3. **执行/运行层**
   - 代表页面：`/task/list`、`/task/:id`
   - 任务：看执行结果、排查执行问题、追踪任务
4. **素材与资源层**
   - 代表页面：`/material/*`
   - 任务：管理输入资源
5. **系统与配置层**
   - 代表页面：`/settings`、`/schedule-config`、`/profile-management`、`/settings/auth-admin`
   - 任务：系统配置、授权治理、策略设置

所有页面收口都应遵守：

> **默认业务层优先展示“用户要完成的动作”，高级诊断信息通过次级入口展开。**

---

## 3. 核心页面期待定义

## 3.1 CreativeWorkbench（`/creative/workbench`）

### 页面角色

这是**日常工作总入口**，不是诊断台，不是大杂烩控制台。

### 用户来到这里最主要想做什么

1. 快速找到待处理作品
2. 判断哪些作品需要我优先处理
3. 进入详情页继续操作
4. 只在必要时进入运行诊断

### 首屏必须看到什么

- 页面标题与一句简短定位说明
- 汇总卡片或轻量 summary
- 高频 preset views
- 搜索框
- 状态筛选 / 排序 / 分页
- 清晰的列表主体（table-first）
- 进入详情的主入口

### 主操作

- 搜索作品
- 按状态筛选
- 使用 preset 快速切视角
- 排序
- 打开作品详情

### 次级操作

- 查看运行诊断
- 打开次级技术信息

### 不应默认出现的内容

- 大段解释型 `Alert`
- scheduler / shadow / kill switch 等工程术语占据首屏主要区域
- 原始 diff / raw diagnostics 直接平铺在主视图

### 成功状态应该像什么

- 用户能在很短时间内定位今天该处理的作品
- 从列表到详情的主链路清晰
- 过滤、排序、分页、返回后上下文不丢

### 空态应该像什么

- 明确告诉用户“当前没有匹配结果”还是“还没有任何作品”
- 提供下一步动作：
  - 清空筛选
  - 新建作品
  - 返回默认视图

### 错态应该像什么

- 告诉用户是列表加载失败，而不是泛化成系统异常
- 提供重试入口
- 不把调试细节默认甩给用户

## 3.2 CreativeDetail（`/creative/:id`）

### 页面角色

这是**作品定义页**，是围绕作品本身做决策的地方。

### 用户来到这里最主要想做什么

1. 看当前作品定义是否完整
2. 编辑作品关键信息
3. 看当前版本与审核状态
4. 判断是否可以发布
5. 在必要时查看更深层诊断

### 首屏必须看到什么

- 页面标题与返回路径
- 作品核心定义信息
- 当前版本状态
- 当前审核结论
- 是否可发布
- 主操作按钮（编辑 / 审核 / 发布相关动作）

### 主操作

- 编辑作品定义
- 审核当前版本
- 进入 AIClip 或相关创作动作
- 触发与发布准备相关的主流程

### 次级操作

- 查看发布池投影
- 查看运行诊断
- 查看 shadow diff / raw diagnostics
- 跳转任务诊断

### 不应默认出现的内容

- 过量的 scheduler cutover 信息
- 原始 diff JSON
- 把所有调试细节与业务信息混放在同一层级

### 成功状态应该像什么

- 用户一眼知道：
  - 这是什么作品
  - 当前版本是否通过审核
  - 离发布还差什么
  - 下一步应该点哪个按钮

### 空态应该像什么

- 某个模块缺数据时，应说明“缺的是什么”
- 要给出补齐路径，而不是只说暂无数据

### 错态应该像什么

- 区分“详情加载失败”“版本信息加载失败”“某个子模块失败”
- 错误粒度应尽可能局部化

---

## 3.3 AIClip（`/ai-clip`）

### 页面角色

这是**创作/合成工作流页面**，不是参数实验面板。

### 用户来到这里最主要想做什么

1. 明确输入是什么
2. 明确要生成什么
3. 明确当前流程走到哪一步
4. 完成一次可理解、可预期的生成/剪辑动作

### 首屏必须看到什么

- 清楚的页面标题
- 输入与输出的核心概览
- 当前工作流步骤或阶段
- 主 CTA（开始生成 / 继续处理 / 重新生成）

### 主操作

- 选择输入
- 确认生成目标
- 启动处理
- 查看结果

### 次级操作

- 展开高级参数
- 查看底层配置
- 查看详细日志

### 不应默认出现的内容

- 过多底层路径输入
- 低层参数充满首屏
- 需要懂工程细节才能用的交互结构

### 成功状态应该像什么

- 用户知道当前输入、处理步骤、结果去向
- 能区分“处理中”“完成”“失败待处理”

### 空态 / 错态重点

- 空态不是“什么都没有”，而要说明先准备什么输入
- 错态不能只给技术报错，要告诉用户下一步能做什么

---

## 3.4 Dashboard（`/dashboard`）

### 页面角色

这是**概览页**，不是系统职责解释页。

### 用户来到这里最主要想做什么

1. 快速看整体状态
2. 从概览进入关键工作面

### 首屏必须看到什么

- 高层概览指标
- 明确的主入口导航
- 不同业务面之间的简洁分流

### 不应默认出现的内容

- 大量“这个页面不是主入口”的解释
- 与用户当下任务无关的说明堆叠

---

## 3.5 TaskList / TaskDetail（`/task/list`、`/task/:id`）

### 页面角色

这是**执行层 / 运行追踪层**，不是业务主入口。

### 用户来到这里最主要想做什么

1. 看任务执行情况
2. 排查失败
3. 追踪历史执行

### 首屏必须看到什么

- 执行状态
- 最近结果
- 失败任务入口
- 与作品/版本/发布包的关联线索

### 不应默认承担的职责

- 不应该重新成为“真正业务中心”
- 不应该替代 CreativeDetail 承担作品定义职责

---

## 3.6 Auth / Settings / ScheduleConfig / ProfileManagement

### 页面角色

这些页都属于**系统配置与治理层**。

### 期待中的共同特征

- 职责要单一
- 文案要稳定
- 成功 / 失败 / 权限状态表达要非常清楚
- 不要混入过多业务主流程解释

### 特别要求

- Auth 页：权限、失效、恢复路径必须清楚
- Settings 页：配置项分组要清楚
- ScheduleConfig 页：要像策略页，不像底层配置散点
- ProfileManagement 页：要强调“配置模板/发布画像”的角色，而不是通用数据表

---

## 4. 页面目标形态的统一要求

所有核心页面后续收口时，都建议统一检查以下问题：

1. 页面标题是否直接表达职责
2. 主操作是否足够显眼
3. 首屏是否只放最重要的信息
4. 诊断信息是否下沉到次级层
5. loading / empty / error / success 是否有一致策略
6. 页面是否适合 Electron 非最大化窗口

---

## 5. 这份清单怎么用

后续建议按以下方式使用本文件：

1. 作为 Gap Audit 的目标参照
2. 作为后续 UI/UX 主线 PRD 的“目标状态”
3. 在真正改页面前，先核对：
   - 这次改动是在让页面更接近目标形态，还是只是继续叠局部功能
