# Slice 10：Alembic 标准化迁移（Day 5.1 ~ Day 5.3）

## Day 5.1：建立 Alembic baseline，但不替换运行时入口

### 目标

先把 Alembic 框架搭起来，并让它能描述**当前已存在 schema 的基线状态**，但先不立刻删掉旧 runner。

### 任务

1. 新增 Alembic 基础目录与配置
   - 预期落点：
     - `remote/remote-backend/alembic.ini`
     - `remote/remote-backend/alembic/env.py`
     - `remote/remote-backend/alembic/versions/`
2. 让 Alembic 能读取当前 SQLAlchemy metadata
   - 目标引用：`remote/remote-backend/app/models.py`
3. 生成或手写一个 **baseline revision**
   - 表示“当前线上/本地库已经处于这个 schema”
4. 新增一条开发/测试命令说明
   - 如何执行 `upgrade`
   - 如何执行 `downgrade`
   - 如何执行 `current`
5. 保留旧 `app/migrations/runner.py`，但明确标注为待退役

### 建议文件

- `remote/remote-backend/alembic.ini`
- `remote/remote-backend/alembic/env.py`
- `remote/remote-backend/alembic/script.py.mako`（若需要）
- `remote/remote-backend/alembic/versions/<baseline>.py`
- `remote/remote-backend/README.md` 或对应后端开发文档

### 验收标准

- 可以在空库上执行 Alembic baseline upgrade 成功
- `alembic current` 能返回正确 revision
- 不要求此时删除旧 runner

### 风险

- baseline 定义如果与真实 ORM 漂移，会导致后续 revision 全乱

### 缓解

- baseline 建立后立即做 schema diff 校验
- 先在测试数据库验证，再碰默认数据库

## Day 5.2：把数据库初始化/测试入口切到 Alembic

### 目标

不是只“有 Alembic 文件”，而是让开发与测试路径真正开始使用 Alembic。

### 任务

1. 盘点当前数据库初始化入口
   - 看谁还在调用 `app/migrations/runner.py`
2. 把测试环境初始化优先切到 Alembic
3. 若有 CLI/bootstrap 脚本，新增统一迁移入口
4. 为 Alembic upgrade/downgrade 增加最小自动化测试
   - 至少覆盖：
     - 空库 upgrade
     - current revision
     - downgrade 到 base 或上一个稳定点

### 建议落点

- `remote/remote-backend/tests/`
  - 新增 migration contract / smoke tests

### 验收标准

- 测试或初始化流程不再依赖旧 runner 作为主路径
- Alembic upgrade 能驱动测试准备
- downgrade 至少有一条最小 smoke 验证

### 风险

- 测试基建和开发基建混切，容易出现“本地能跑、CI 不能跑”

### 缓解

- 先让测试环境切换，再处理开发命令文档与本地脚本

# Slice 11：密码哈希升级 / step-up security（Day 5.4 ~ Day 5.6）

## Day 5.4：只做 Argon2 平滑升级设计，不引入 step-up runtime

### 目标

把 Slice 11 收缩成**最小可交付安全升级**：

- 先做 Argon2
- 先做平滑迁移
- 先不做 TOTP / 二次密码 / step-up token 运行时

### 任务

1. 设计双栈校验模型
   - 旧 hash：继续可验证
   - 新 hash：默认使用 Argon2
2. 明确数据模型收口策略
   - end-user：是否继续使用现有 `password_algo`
   - admin-user：是否补 `password_algo` 字段
3. 明确升级触发机制
   - 推荐：**登录成功时 opportunistic rehash**
4. 明确失败回滚策略
   - Argon2 验证异常不能影响旧 PBKDF2 登录可用性

### 推荐决策

- **Day 5 只做密码哈希升级**
- **step-up security 只保留设计位，不进实现**

### 验收标准

- 有一份明确的哈希升级 ADR
- 明确 admin / end-user 是否共用同一套哈希元数据策略

---

## Day 5.5：落地双栈密码校验 + opportunistic rehash

### 目标

在不打断现有登录主路径的前提下，把默认新哈希切到 Argon2，并允许旧 PBKDF2 在用户下次成功登录时平滑升级。

### 任务

1. 扩展 `app/core/security.py`
   - 支持 Argon2 hash
   - 支持 PBKDF2 verify
   - 支持“是否需要 rehash”的判断
2. 扩展配置
   - Argon2 参数配置
   - 保留旧 `PASSWORD_HASH_ITERATIONS` 以兼容旧 hash
3. 统一运行时调用点
   - `remote/remote-backend/app/services/admin_service.py`
   - `remote/remote-backend/app/services/auth_service.py`
4. 登录成功后触发 opportunistic rehash
   - 仅当旧 hash 登录成功时重写为新 hash
5. 如有必要，为 `admin_users` 补 `password_algo`
   - 这一步若涉及 schema 变更，必须通过 Slice 10 的 Alembic 流程完成

### 测试任务

- PBKDF2 旧 hash 仍可登录
- 新注册/新写入默认用 Argon2
- 旧 hash 登录成功后会升级为 Argon2
- 错误密码不会误触发 rehash
- admin / end-user 两条登录链路都覆盖

### 验收标准

- 旧密码数据无需批量重置
- 默认新 hash 已切到 Argon2
- 登录主路径没有行为倒退

### 风险

- admin 与 end-user 两套密码存储结构不对称，容易一边升级一边遗忘

### 缓解

- 先统一元数据，再统一逻辑
- 两条登录链路必须一起建回归
