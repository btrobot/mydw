# 远端最小风险改造路线图

日期：2026-04-25

## 1. 目标

在**不先推翻现有后端**的前提下，按最低回归风险完成远端系统改造：

1. 先把远端后台从“可用但脆弱”提升到“可验证、可迭代”。
2. 再把远端前端从手写 DOM shell 平滑迁到 React Admin 应用。
3. 最后补生产化能力：限流、迁移工具、安全增强。

---

## 2. 为什么这条路线风险最低

### 现状判断

1. **远端前端是主要改造对象**
   - 当前 `remote-admin` 仍以字符串拼 HTML + DOM 事件绑定为主：`remote/remote-admin/src/app/App.ts:455`、`remote/remote-admin/src/main.ts:489`、`remote/remote-admin/src/main.ts:663-664`、`remote/remote-admin/src/main.ts:1254`
   - `remote/remote-admin/package.json` 未引入 React / Ant Design / React Query / Router 体系。

2. **远端后端架构方向基本可保留**
   - 业务已在 service 层沉淀：`remote/remote-backend/app/services/admin_service.py:55`
   - 高风险动作已集中：`remote/remote-backend/app/services/admin_service.py:223`、`:230`、`:284`、`:294`、`:301`、`:353`
   - RBAC 已存在：`remote/remote-backend/app/services/admin_authz.py:58-67`

3. **后端当前最大短板不是“架构错”，而是“没有保护网”**
   - `remote/remote-backend/tests/` 目前基本为空，仅有 `__init__.py`
   - 但管理端 API 已包含登录、封禁、解绑、重绑、撤销 session 等高风险动作：`remote/remote-backend/app/api/admin.py:64`、`:140`、`:152`、`:199`、`:211`、`:223`、`:261`

4. **生产化问题真实存在，但不该抢在最前面**
   - 登录限流还是内存实现：`remote/remote-backend/app/core/rate_limit.py:7`、`remote/remote-backend/app/api/admin.py:34`、`:67`
   - 迁移仍是自定义 runner：`remote/remote-backend/app/migrations/runner.py`
   - 密码哈希仍是 `pbkdf2_sha256`：`remote/remote-backend/app/core/security.py:15`、`:24`

### 路线原则

1. **先锁行为，再换壳。**
2. **先补后端测试，再动大规模前端重写。**
3. **前端按页迁移，不做后端接口大洗牌。**
4. **生产化增强分批做，不与前端重写绑死。**
5. **能复用本地端技术栈就复用。**

本地端现有栈已成熟，可作为远端前端默认对齐目标：`frontend/package.json:19-20`、`:36-42`、`:46`、`:51`、`:59`

---

## 3. 总体执行顺序

### Phase 0：冻结现状与改造边界（1 个小迭代）

**目标：** 在改代码前，把“什么不能变”说清楚。

#### 要做什么

1. 盘点远端后台现有页面与 API：
   - login
   - dashboard
   - users
   - devices
   - sessions
   - audit
2. 明确哪些是**必须保持兼容**的接口与返回 shape。
3. 把现有关键用户流整理成回归清单：
   - 管理员登录
   - 只读角色访问
   - revoke / restore user
   - unbind / disable / rebind device
   - revoke session
   - audit 筛选与分页
4. 冻结一版“现网行为基线”文档。

#### 产出物

- 一份远端改造基线清单
- 一份关键回归场景清单

#### 通过门槛

- 所有高风险动作都有回归条目
- 所有现有 admin API 都有归属页面/用途说明

---

### Phase 1：先补后端测试地板（最高优先级）

**目标：** 给后续前端重写和后端增强铺保护网。

#### 要做什么

1. 为 `AdminService` 建立核心单元测试：
   - `list_users`
   - `revoke_user`
   - `restore_user`
   - `unbind_device`
   - `disable_device`
   - `rebind_device`
   - `revoke_session`
2. 为 RBAC 建立权限测试：
   - `super_admin`
   - `auth_admin`
   - `support_readonly`
3. 为关键 API 建立最小集成测试：
   - `/login`
   - user/device/session 高风险动作
   - 常见 401/403/404/429 错误路径

#### 主要落点

- `remote/remote-backend/tests/`
- `remote/remote-backend/app/services/admin_service.py:149-353`
- `remote/remote-backend/app/services/admin_authz.py:58-67`
- `remote/remote-backend/app/api/admin.py:64-261`

#### 为什么先做

因为后续不管是前端重写、分页统一、限流替换还是 step-up security，都要依赖这层保护网；否则每次改动都只能“靠感觉”回归。

#### 通过门槛

- 高风险 service 动作都有测试
- 角色差异都有测试
- 登录与关键 destructive API 有最小集成覆盖
- 测试能稳定区分 401 / 403 / 404 / 429

---

### Phase 2：搭新前端骨架，但先不替换旧后台逻辑

**目标：** 先搭出可演进的 React 壳，不急着一次性切全量。

#### 建议栈

- React 18
- TypeScript
- Ant Design 5
- React Query
- React Router 6
- Vite

这与本地端现有体系一致：`frontend/package.json:36-42`、`:51`、`:59`

#### 要做什么

1. 新建远端前端 React 工程骨架。
2. 先接现有后端 API，不同步推动后端协议大改。
3. 先落基础设施：
   - app shell / layout
   - auth state
   - API client
   - route guards
   - error boundary
   - 通用 loading / empty / error 组件
4. 首批只迁最稳定页面：
   - login
   - dashboard
   - users list

#### 明确约束

- **默认不用 Refine。**
- 如要评估 Refine，也只作为备选方案，不要让它阻塞主线迁移。

#### 为什么不先上 Refine

因为当前后台并不只是 CRUD，还包含 dashboard、RBAC、审计、高风险动作确认；先用 plain React + AntD + Query 更稳，更贴近本地已有开发经验。

#### 通过门槛

- 新前端可独立运行
- 登录流打通
- 至少一个核心列表页可替代旧页面
- 通用状态处理不再依赖字符串模板重绑事件

---

### Phase 3：按“页面切片”迁移旧前端

**目标：** 逐页替换，避免一次性大爆炸切换。

#### 推荐切片顺序

1. login
2. dashboard
3. users
4. devices
5. sessions
6. audit

#### 每页迁移时必须一起处理的内容

1. 列表加载/空态/错误态
2. 筛选与分页
3. 详情或关键动作面板
4. 权限态差异
5. destructive action UX

#### 现有旧实现依据

- 页面整体仍由 `renderApp` 输出：`remote/remote-admin/src/app/App.ts:455`
- 页面交互主要靠 `innerHTML` 后再重绑事件：`remote/remote-admin/src/main.ts:489`、`:539-1046`
- 高危操作确认仍是 `window.confirm`：`remote/remote-admin/src/main.ts:1254`

#### 迁移策略

- 新旧前端可以短期并存
- 切一页，验一页
- 不在此阶段引入额外后端 schema 重构，除非该页迁移被明确阻塞

#### 通过门槛

- 每切完一页，都能通过该页回归场景
- 旧前端对应页面可以下线或留作临时 fallback

---

### Phase 4：做后端“低耦合生产化增强”

**目标：** 处理那些明显需要升级、但不应阻塞前端重写的能力。

#### 4.1 限流升级（优先级高）

**建议顺序：**

1. 先在 Nginx / API gateway 处理登录限流
2. 如需应用内细粒度策略，再补 Redis-backed limiter

#### 证据

- 当前是 `InMemoryRateLimiter`：`remote/remote-backend/app/core/rate_limit.py:7`
- 登录 API 直接依赖它：`remote/remote-backend/app/api/admin.py:34`、`:67`

#### 4.2 分页返回 shape 统一（优先级中高）

建议逐步统一到：

```json
{
  "items": [],
  "total": 0,
  "page": 1,
  "page_size": 20
}
```

这项应在**新前端已经接稳现有列表页后**推进，避免前后端同时大改。

#### 4.3 错误语义与权限态补齐（优先级中高）

把 401 / 403 / 404 / 429 的前后端处理方式固定下来，防止新前端和后端各自猜语义。

#### 通过门槛

- 登录限流不再依赖单进程内存状态
- 列表 API 返回 shape 至少在新前端接入范围内保持一致
- 常见错误码语义被文档化并有测试覆盖

---

### Phase 5：做后端“中期基础设施升级”

**目标：** 提升长期可维护性，但不抢主线节奏。

#### 5.1 迁移到 Alembic

#### 现状

- 当前仍是自定义迁移 runner：`remote/remote-backend/app/migrations/runner.py`

#### 策略

1. 先补测试，确保 schema 相关行为有保护
2. 再做一次性基线迁移接入 Alembic
3. 后续所有 schema 变更只走标准迁移

#### 5.2 密码哈希升级

#### 现状

- 当前使用 `pbkdf2_sha256`：`remote/remote-backend/app/core/security.py:15`、`:24`

#### 策略

1. 不急着在前端迁移期同时切密码方案
2. 等测试、登录流、限流稳定后，再评估 Argon2 迁移
3. 采用“新登录逐步升级 hash”或“下次改密升级”的平滑策略

#### 5.3 高危操作 step-up security

如果将来要做：

- TOTP
- 二次密码确认
- step-up token

则必须由**后端强制校验**，不能只做前端弹窗替换。

#### 通过门槛

- schema 迁移脱离自定义 runner
- 密码策略升级不影响存量登录
- 高危动作强确认具备后端约束

---

## 4. 不建议现在做的事

1. **不要先推翻远端后端分层重写。**
2. **不要把前端重写、分页统一、限流替换、Alembic、密码升级绑成一个大版本。**
3. **不要默认押注 Refine 作为主方案。**
4. **不要把真正的安全确认误判成纯前端交互问题。**
5. **不要在没有测试保护的情况下重排 admin service 逻辑。**

---

## 5. 可并行的最小化分工

### Lane A：后端测试保护网

负责：

- `remote/remote-backend/tests/`
- admin service / RBAC / API 关键路径测试

### Lane B：新前端基础骨架

负责：

- React + TS + AntD + Query + Router + Vite 基础工程
- login / dashboard / users 首批页面

### Lane C：协议与行为基线

负责：

- API shape 清单
- 错误码语义清单
- 页面迁移验收清单

这三条 lane 可以并行，但**前端全面切页必须等 Lane A 至少完成最小测试地板**。

---

## 6. 最小风险验收标准

满足以下条件，才算这轮改造真正“稳”：

1. 远端后台高风险动作有自动化测试保护。
2. 新前端已覆盖 login / users / devices / sessions / audit 的主流程。
3. 旧前端核心页面已可下线。
4. 登录限流不再依赖单机内存。
5. 分页和错误码语义在新前端范围内一致。
6. 迁移工具已切到标准方案，后续 schema 变更不再依赖自定义 runner。

---

## 7. 推荐落地顺序（一句话版）

> **先补后端测试 → 再搭 React 新前端 → 再按页迁移 → 再补限流/分页一致性 → 最后做 Alembic、密码策略和 step-up security。**

这条顺序的核心不是“最先进”，而是**每一步都能验证、都能回退、都不要求同时改太多层。**
