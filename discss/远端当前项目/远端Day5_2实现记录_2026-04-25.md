# 远端 Day 5.2 实现记录（切测试 / 初始化 / bootstrap 到 Alembic）

## 目标

在 **Day 5.1 已经建立 baseline** 的前提下，继续推进 Slice 10，但仍坚持最小风险原则：

- 让本地初始化入口默认走 Alembic
- 让 bootstrap 脚本的 `--migrate` 不再直接依赖 legacy runner
- 给测试补上对新入口的最小回归
- 暂时不删除 legacy runner 文件本身

## 本次改动

### 1. 新增 Alembic 运行时 helper

新增：

- `remote/remote-backend/app/migrations/alembic.py`

职责：

- 统一构建 Alembic config
- 统一读取 current revision
- 提供 `upgrade_head()` / `downgrade_to()` / `stamp_revision()`
- 提供 `ensure_database_on_head()`

其中 `ensure_database_on_head()` 的策略是：

1. 如果数据库已经有 Alembic revision，则直接 `upgrade head`
2. 如果数据库还没有 Alembic revision，但检测到 legacy runner 留下来的状态（如 `schema_migrations` 或现有业务表），先 `stamp baseline`
3. 然后再执行 `upgrade head`

这样可以覆盖两类最常见场景：

- 空库初始化
- 历史本地库 / 测试库从 legacy runner 平滑接管

### 2. 本地 migrate 入口切到脚本

新增：

- `remote/remote-backend/scripts/migrate.py`

提供统一入口：

- `python scripts/migrate.py upgrade head`
- `python scripts/migrate.py downgrade base`
- `python scripts/migrate.py current`
- `python scripts/migrate.py stamp 20260425_0001`
- `python scripts/migrate.py ensure-head`

这里的价值不是“再包一层”，而是把：

- sys.path 初始化
- settings/db cache reset
- Alembic 调用方式

统一到一个稳定入口，减少 README、测试、bootstrap 各自拼装调用的样板。

### 3. bootstrap_admin 的 `--migrate` 改走 Alembic

修改：

- `remote/remote-backend/scripts/bootstrap_admin.py`

变化：

- 之前：`--migrate` 调 `app.migrations.runner.upgrade()`
- 现在：`--migrate` 调 `ensure_database_on_head()`

这样做的好处：

- 空库可以直接升级到 head
- 旧 runner 初始化过的库可以先 stamp baseline，再继续进入 Alembic 轨道
- 不需要在 Day 5.2 就删除 legacy runner

### 4. README 默认命令切换

修改：

- `remote/remote-backend/README.md`

默认本地运行前步骤改为：

- `python scripts/migrate.py ensure-head`

同时保留：

- brand-new 空库：`upgrade head`
- 旧库接管：`stamp baseline` + `upgrade head`

### 5. 增加 Day 5.2 回归测试

新增：

- `remote/remote-backend/tests/migrations/test_alembic_entrypoints.py`

覆盖：

1. 空库执行 `ensure_database_on_head()` 可以建出完整 schema
2. legacy runner 初始化过的库可以被 `ensure_database_on_head()` 平滑接管
3. `bootstrap_admin.py --migrate` 会通过 Alembic 入口完成迁移并创建 admin
4. `scripts/migrate.py current` 能返回当前 revision

## 风险判断

### 已控制的风险

- **没有直接删掉 legacy runner**
  - 仍可作为 Day 5.3 之前的兼容保底
- **没有在 Day 5.2 引入新的 schema 变更**
  - 只切入口，不动数据模型
- **对旧库采用 stamp + upgrade**
  - 避免 baseline DDL 重放到已存在表上

### 仍然存在的风险

- `ensure_database_on_head()` 目前对“已有业务表但状态不完整”的库采用保守接管策略，默认认为它来自 legacy runner；如果有人手工造出异常半成品库，仍可能需要人工清理
- legacy runner 代码还在 repo 中，Day 5.3 前仍可能被人误用

## 结论

Day 5.2 的核心目标已经落地：

- **测试入口**：已有 Alembic 回归
- **初始化入口**：README 默认迁移入口切到 Alembic
- **bootstrap 入口**：`bootstrap_admin --migrate` 切到 Alembic

同时仍保持最小风险：

- 不切线上运行时
- 不删 legacy runner
- 不把 schema 演进和运行时切换混在同一刀里

下一步应进入 **Day 5.3**：

- 收口 legacy runner
- 清理旧文档与旧路径
- 明确后续 schema 变更必须经 Alembic revision
