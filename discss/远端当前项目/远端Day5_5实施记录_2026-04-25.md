# 远端 Day 5.5 实施记录（Argon2 平滑升级落地）

日期：2026-04-25

## 本次落地范围

- 仅升级**账户密码**哈希链路到 Argon2id。
- 保持 **refresh token** 继续走现有 PBKDF2 兼容路径。
- 不改前端合同，不引入 step-up runtime。

## 已完成改造

1. **拆分密码/refresh token helper**
   - `app/core/security.py` 不再把账户密码与 refresh token 共用同一组 helper。
   - 新增：
     - `hash_account_password`
     - `verify_account_password`
     - `hash_refresh_token`
     - `verify_refresh_token`

2. **账户密码默认写入 Argon2id**
   - 新增 Argon2 参数配置。
   - `UserCredential.password_algo` ORM 默认值切到 `argon2id`。
   - `AdminUser` 新增 `password_algo` 字段，ORM 默认值切到 `argon2id`。

3. **保留 PBKDF2 兼容登录**
   - 老的 `pbkdf2_sha256` 密码仍可登录。
   - 登录成功后会尝试 opportunistic rehash 到 Argon2id。
   - rehash 失败不会阻塞登录成功路径。

4. **refresh token 语义保持不变**
   - refresh token 仍使用 PBKDF2 兼容哈希。
   - repository 查询路径已切换到显式 `verify_refresh_token`，避免后续再次与账户密码耦合。

5. **Alembic 迁移补齐**
   - 新增 revision：`20260425_0002`
   - 为 `admin_users` 增加 `password_algo`
   - 对既有 admin 行回填 `pbkdf2_sha256`

## 剩余风险

- 生产环境还需要实际安装 `argon2-cffi` 并进入部署镜像/虚拟环境。
- 当前 opportunistic rehash 只覆盖登录成功路径；若长期不登录，旧 PBKDF2 密码会继续存在。
- 账户密码与 refresh token 已拆 helper，但 PBKDF2 底层实现仍共用同一内部编码格式；后续若要继续降耦，可再单独抽离 token hashing policy。
