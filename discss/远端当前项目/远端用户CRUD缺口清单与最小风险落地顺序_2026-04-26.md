# Remote Admin 用户 CRUD 缺口清单 + 最小风险落地顺序（2026-04-26）

## 1. 当前事实基线

### 1.1 Remote Admin 前端当前已落地的用户能力
当前 `remote/remote-admin` 的用户页只实现了以下能力：

- 用户列表查询
- 用户详情查询
- `revoke user`
- `restore user`
- 只读 / 可写角色控制
- destructive action 的 step-up verify
- 成功后列表 / 详情刷新

对应证据：

- `remote/remote-admin/src/pages/users/UsersPage.tsx`
- `remote/remote-admin/src/features/auth/auth-client.ts`

### 1.2 当前后端 admin 用户路由现状
当前 `remote/remote-backend` 的 admin 路由包含：

- `GET /admin/users`
- `GET /admin/users/{user_id}`
- `PATCH /admin/users/{user_id}`
- `POST /admin/users/{user_id}/revoke`
- `POST /admin/users/{user_id}/restore`

但**不包含**：

- `POST /admin/users`
- `DELETE /admin/users/{user_id}`

对应证据：

- `remote/remote-backend/app/api/admin.py`

### 1.3 规格与实现的落差
UI 规格里，Users 页其实已经写到了 edit form / Save Changes / Reset，并且把 `PATCH /admin/users/{id}` 列入了 backend surface。

但当前 React 实现并没有把这条更新能力接出来。

对应证据：

- `remote/remote-shared/docs/remote-admin-platform-ui-spec-v1.md`

---

## 2. 用户 CRUD 缺口清单

## Gap A：Create（新增用户）缺失

### 当前状态
- 前端没有新增入口
- 后端没有 `POST /admin/users`
- OpenAPI / runtime contract 里当前也没有 create user 的 admin surface

### 结论
这不是前端漏接，而是**产品合同当前未定义**。

### 风险
- 一旦补 create，会立刻牵涉：
  - 用户名唯一性 / 邮箱唯一性校验
  - 初始密码 / 邀请制 / 首次登录改密策略
  - 初始 license / entitlement 设定
  - tenant 归属
  - 审计记录
  - admin role 边界

### 建议
- 不要把 create 当成“顺手补 UI”
- 必须先做 contract 设计，再做后端，再做前端

---

## Gap B：Update（编辑用户）后端已具备，前端未落地

### 当前状态
后端已支持：

- `PATCH /admin/users/{user_id}`

当前 schema 可更新字段：

- `display_name`
- `license_status`
- `license_expires_at`
- `entitlements`

对应证据：

- `remote/remote-backend/app/api/admin.py`
- `remote/remote-backend/app/schemas/admin.py`

服务层还已经区分了：

- 哪些字段更新需要 step-up
- 哪些更新可以低风险直接改

对应证据：

- `remote/remote-backend/app/services/admin_service.py`
- `remote/remote-backend/tests/services/test_admin_step_up_runtime.py`

### 当前前端缺口
前端还没有：

- `updateAdminUser()` client
- Users 页 edit form
- Save / Reset
- 更新成功后的 detail/list refresh
- 对 PATCH 的最小 contract / React 回归入口

### 结论
这是当前最适合优先补的 CRUD 缺口。

### 风险
- 相比 create / delete，风险最低
- 因为：
  - 后端 contract 已存在
  - 服务层逻辑已存在
  - step-up 机制已存在
  - 用户页已有 list/detail/refresh 模式可复用

---

## Gap C：Delete（删除用户）缺失

### 当前状态
- 前端无 delete 按钮
- 后端无 `DELETE /admin/users/{user_id}`
- 当前产品语义使用的是：
  - `revoke`
  - `restore`

### 结论
这大概率是**有意不支持 hard delete**，不是遗漏。

### 为什么当前模型更偏 revoke / restore
- 更符合审计型后台
- 不破坏：
  - audit logs 追踪链路
  - 设备绑定历史
  - session 历史
  - target_user_id 的历史可追踪性
- 更适合 support / auth-admin 场景

### 风险
如果直接引入 delete，会牵涉：

- 数据保留策略
- 审计追踪完整性
- 外键 / 历史记录一致性
- “软删除”还是“硬删除”
- 删除后 restore 是否还存在

### 建议
- 短期内**不要把 delete 放进主线**
- 除非业务明确提出“彻底移除用户”的真实场景，并补齐 retention / audit / legal 约束

---

## 3. 最小风险落地顺序

## Phase 0：保持现状，不碰 create / delete

### 目标
- 承认当前 Remote Admin 本质上是“治理后台”，不是“用户主数据后台”
- 不把问题范围一下子拉成全量 CRUD 平台化改造

### 输出
- 产品口径统一：
  - 当前支持：review / update（待补前端）/ revoke / restore
  - 当前不支持：create / delete

---

## Phase 1：优先补 Update UI（最小风险，最高收益）

### 为什么先做这个
- 后端已具备
- 规格已定义
- 前端主页面已具备详情面板
- 与现有 users page 架构最贴近

### 建议切片

#### Slice 1：补前端 client
- 在 `remote/remote-admin/src/features/auth/auth-client.ts` 增加：
  - `updateAdminUser(accessToken, userId, payload, stepUpToken?)`
- 对齐后端 PATCH payload

#### Slice 2：补 Users 详情区 edit form
- 在 `UsersPage.tsx` 的 detail panel 中增加：
  - Display name
  - License status
  - License expires at
  - Entitlements
- 加：
  - Save changes
  - Reset

#### Slice 3：接 step-up 策略
- 与后端语义对齐：
  - 低风险字段（如 display_name）若不要求 step-up，则直接提交
  - 敏感字段（如 entitlements / license status）若返回 step-up required，则走现有 password verify 流程

#### Slice 4：补刷新策略
- PATCH 成功后：
  - refresh detail
  - refresh list
  - 保持当前 selected user

#### Slice 5：补最小回归
- contract test：
  - PATCH client body / header / step-up token header
- React 级回归：
  - Save 成功
  - step-up required -> modal 打开
  - verify 成功后 PATCH 成功
  - verify 失败 / token_expired 跳转

### Phase 1 验收标准
- 用户详情可以编辑并保存
- 不破坏现有 revoke / restore 流程
- 角色受限时，表单和按钮显式 disable
- step-up 行为与现有 destructive action 体验一致

---

## Phase 2：评估“Create user”是否真的需要

### 先回答的不是“怎么做”，而是“要不要做”
必须先确认：

1. 用户是由哪个系统创建？
   - Remote Admin 自己？
   - 上游 IAM / 运营后台？
   - 邀请制？
2. 初始密码如何处理？
3. 是否允许管理员直接设置 entitlements？
4. tenant_id 是否手填？
5. create 是否要求 step-up？
6. create 后是否必须记录 audit reason？

### 若决定做 create，建议顺序

#### Slice 2.1：先补 contract / schema / OpenAPI
- 明确 `POST /admin/users`
- 明确请求体与校验规则

#### Slice 2.2：补后端 service / audit / RBAC
- 权限控制
- 唯一性校验
- 审计
- step-up 策略

#### Slice 2.3：最后才补 React UI
- 新增按钮
- create drawer / modal
- 成功后定位到新用户详情

### 结论
- create 是一个**独立产品能力**
- 不建议和 update UI 混成一个小修复一起做

---

## Phase 3：原则上不做 Delete，除非业务明确要求

### 默认建议
- 继续坚持：
  - revoke / restore
  - audit-first
  - retain history

### 如果未来一定要做 delete，先明确三选一

1. **Hard delete**
   - 风险最高
2. **Soft delete**
   - 需要额外 deleted 状态与查询过滤策略
3. **Archive / deactivate**
   - 其实常常可以被当前 revoke 模型覆盖

### 推荐判断
- 大多数远端授权治理场景里，**revoke 已经足够**
- delete 往往不是 P1 功能，而是治理策略问题

---

## 4. 推荐优先级（建议直接照这个顺序排）

### P0
- 明确口径：当前不支持 create / delete
- 对外说明：当前 Users 页是治理型，不是全量 CRUD 主数据后台

### P1
- 补 Users Update 前端落地

### P2
- 为 Update 补 contract / React regression 护栏

### P3
- 评估是否真的需要 Create user

### P4
- 如业务必须，再设计 create contract

### P5
- 仅在合规 / retention / audit 规则清楚后，才讨论 delete

---

## 5. 最终建议（一句话版）

### 不要现在就追求“把 CRUD 一次补齐”
最小风险路径是：

1. **先补 Update UI**
2. **Create 单独立项**
3. **Delete 默认不做，继续坚持 revoke / restore**

---

## 6. 如果要直接进入实施，建议下一个执行 slice

### Day X.1：Users Update UI 最小落地

目标：
- 在不改现有 revoke / restore 主线的前提下，把 `PATCH /admin/users/{id}` 接到 React Users detail panel

最小范围：
- `auth-client.ts`
- `UsersPage.tsx`
- `react-contract.test.mjs`
- `step-up.react.test.mjs`（如涉及 step-up update）

验收：
- display_name 可改
- entitlements / license_status 变更可走 step-up
- 保存后 list + detail 刷新
- 只读角色不可编辑

这会是当前所有 CRUD 缺口里**风险最低、收益最大、与现有实现最连续**的一步。
