# 远端现状行为基线

日期：2026-04-27

## 1. 目标

这份基线文档只回答一个问题：

> 在远端改造正式开做前，当前系统有哪些页面、对应哪些 API、哪些角色差异和高风险动作必须被保留？

它是 Day 1 之后所有测试补齐、前端迁页、验收回归的共同参照。

---

## 2. 页面与主数据流

## login

- 前端入口：`remote/remote-admin/src/main.ts`
- 路由守卫：`resolveGuardedRoute('login', ...)`
- 核心 API：
  - `POST /admin/login` → `remote/remote-backend/app/api/admin.py:64`
  - `GET /admin/session` → `remote/remote-backend/app/api/admin.py:77`

### 当前关键行为

- 未登录时进入 login
- 登录成功后跳转 dashboard
- 登录失败显示稳定错误文案
- 本地存储 `remote_admin_access_token`

---

## dashboard

- 前端加载逻辑：`ensureDashboardMetricsLoaded()` in `remote/remote-admin/src/main.ts`
- 核心 API：
  - `GET /admin/metrics/summary` → `remote/remote-backend/app/api/admin.py:312`

### 当前关键行为

- 已登录用户进入 dashboard
- 有 loading / error / retry 三种主态
- dashboard 展示 metrics + recent failures + destructive actions

---

## users

- 前端加载逻辑：`ensureUsersLoaded()` / `loadUserDetail()` in `remote/remote-admin/src/main.ts`
- 核心 API：
  - `GET /admin/users` → `remote/remote-backend/app/api/admin.py:90`
  - `GET /admin/users/{user_id}` → `remote/remote-backend/app/api/admin.py:111`
  - `PATCH /admin/users/{user_id}` → `remote/remote-backend/app/api/admin.py:123`
  - `POST /admin/users/{user_id}/revoke` → `remote/remote-backend/app/api/admin.py:140`
  - `POST /admin/users/{user_id}/restore` → `remote/remote-backend/app/api/admin.py:152`

### 当前关键行为

- 支持 query / status / license_status 筛选
- 列表与详情联动
- 支持 revoke / restore
- readonly 角色不可执行 destructive action

---

  - `POST /admin/devices/{device_id}/unbind` → `remote/remote-backend/app/api/admin.py:199`

## 3. 角色差异基线

权限矩阵定义在：

- `remote/remote-backend/app/services/admin_authz.py:58-67`

## super_admin

- 具备 read + write 全权限

## auth_admin

- 当前与 `super_admin` 同权限

## support_readonly

- 仅有 read 权限：
  - session.read
  - users.read
  - devices.read
  - sessions.read
  - audit.read
  - metrics.read

### 当前前端表现要求

- readonly 角色可以看 users / devices / sessions / audit / metrics
- readonly 角色不能执行 revoke / restore / unbind / disable / rebind / session revoke

---

## 4. 高风险动作基线

以下动作全部视为 destructive / high-risk action：

1. `POST /admin/users/{user_id}/revoke`
2. `POST /admin/users/{user_id}/restore`
3. `POST /admin/devices/{device_id}/unbind`
4. `POST /admin/devices/{device_id}/disable`
5. `POST /admin/devices/{device_id}/rebind`
6. `POST /admin/sessions/{session_id}/revoke`

### 当前行为

- 前端使用 `window.confirm(...)` 做弱确认：
  - `remote/remote-admin/src/main.ts:1254`
- 真正权限控制仍以后端 RBAC 为准：
  - `remote/remote-backend/app/services/admin_authz.py`
  - `remote/remote-backend/app/services/admin_service.py`

## 5. 当前错误语义草图（记录现状）

## login

- `401 invalid_credentials`
- `403 forbidden`
- `429 too_many_requests`

对应位置：

- `remote/remote-backend/app/api/admin.py:64`
- `remote/remote-admin/src/main.ts` 中 `mapLoginError(...)`

## 受保护接口

- `401 token_expired`
- `403 forbidden`
- `404 not_found`（资源不存在）

这些语义在 users / devices / sessions 的 destructive action 中都存在。

## 6. Day 1 之后默认不改的边界

在完成第一轮测试保护网之前，下面内容默认不动：

1. 不做分页 shape 统一
2. 不做限流替换
3. 不做 Alembic 迁移
4. 不做密码哈希升级
5. 不做 step-up security

原因不是它们不重要，而是它们不属于“先把基线锁住”的 Day 1 范围。
