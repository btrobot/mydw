# 远端 Day 4.1 实现记录（关键 destructive API 最小集成测试入口）

## 本次 slice 目标

- 在 Day 4 已固定 `/admin/login` 的 `401 / 403 / 429` 语义之后，继续补齐**关键 destructive API** 的最小集成测试入口。
- 本次不改后端业务逻辑，不改接口协议，只把这些高风险路由的 API 层行为固定下来，便于后续继续做限流替换与错误语义收口。

## 本次实现

### 1. 新增 destructive API 集成测试

新增文件：

- `remote/remote-backend/tests/api/test_admin_destructive_api.py`

覆盖路由：

- `POST /admin/users/{user_id}/revoke`
- `POST /admin/users/{user_id}/restore`
- `POST /admin/devices/{device_id}/unbind`
- `POST /admin/devices/{device_id}/disable`
- `POST /admin/devices/{device_id}/rebind`
- `POST /admin/sessions/{session_id}/revoke`

### 2. 固定“成功转发”语义

对每个 destructive route，都验证：

- `Authorization: Bearer ...` 会被正确抽取成 access token
- path 参数会被正确传给 service
- `rebind_device` 的 body payload 会被正确转成 request model 再传入 service
- 成功时稳定返回：

```json
{ "success": true }
```

### 3. 固定“错误透传”语义

对每个 destructive route，都验证：

- **401**
  - 当前统一按 `token_expired` 语义透出
  - 缺失 bearer header 时，API 层会把空 token 传给 service，由 service 决定返回 401
- **403**
  - 当前统一按 `forbidden` 语义透出
- **404**
  - 当前统一按 `not_found` 语义透出
  - 并保留对应资源 id 的 details

也就是说，Day 4.1 的目标不是让这些 API 变复杂，而是让它们的**路由层行为可验证、可回归**。

## 与 429 的关系

本次没有给 destructive routes 新增 429，因为当前这些路由本身并**没有**独立限流器。  
Day 4 已经通过 `test_admin_login_api.py` 固定了 `/admin/login` 的 `429 too_many_requests` 入口。

因此当前错误语义基线可理解为：

- `/admin/login`：`401 / 403 / 429`
- destructive admin APIs：`401 / 403 / 404`

这正符合当前真实实现边界，避免为了“语义整齐”凭空发明不存在的 429 行为。

## 为什么这是“最小风险”

1. **只补测试，不改运行时逻辑**
   - 本 slice 没有修改 admin API 的业务实现
   - 只是让已有行为变成自动化可验证的合同

2. **先固定 API 层合同，再谈统一抽象**
   - 未来如果继续推进 401/403/404/429 的全局文档化或响应中间层统一，这些测试可以直接当回归基线

3. **覆盖高风险动作，但不扩大范围**
   - 只盯住真正的 destructive routes
   - 不顺手把 list/read/audit/query 路由也一起掺进来
