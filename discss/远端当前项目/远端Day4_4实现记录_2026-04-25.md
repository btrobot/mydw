# 远端 Day 4.4 实现记录（`/refresh` 429 合同测试与最小回归入口）

## 本次 slice 目标

- 继续沿着 Day 4 的“错误语义收口 + 最小回归入口”路线推进。
- 为 `/refresh` 补上：
  - **429 限流合同测试**
  - 一组最小但可复用的 API 回归入口

本次仍坚持最小风险原则：
- **不改运行时逻辑**
- **不改前端合同**
- 只把当前真实存在的 `/refresh` 路由行为固化为自动化测试

## 本次实现

### 1. 新增 `/refresh` API 测试文件

新增文件：
- `remote/remote-backend/tests/api/test_auth_refresh_api.py`

测试方式：
- 使用 `FakeAuthRefreshService`
- 覆盖 route 层行为，不扩展成 auth service 逻辑测试

### 2. 固定成功路径

补齐了 `/refresh` 的最小成功回归入口：

- `POST /refresh` 成功返回 `200`
- 返回 payload shape 固定
- `client_ip` 会从请求上下文中正确透传给 service
- `refresh_token / device_id / client_version` 会原样进入 service

这让 `/refresh` 不再只有“限流是否命中”的单点测试，而是也有最小的成功面基线。

### 3. 固定 `/refresh` 的 401 / 403 语义透传

本次补了两类最关键的 service error 透传语义：

- `401 token_expired`
- `403 revoked`

目的不是穷举所有 refresh 失败原因，而是先把：
- **认证失败语义**
- **已撤销语义**

这两类前端最敏感的合同固定下来。

### 4. 固定 `/refresh` 的 429 合同

新增回归点：
- 当 `refresh_rate_limiter` 桶已满时，route 会在调用 service 之前直接返回 `429`

并固定返回体：

```json
{
  "error_code": "too_many_requests",
  "message": "Too many refresh attempts, please retry later.",
  "details": {}
}
```

同时验证：
- 命中限流后 **不会继续调用** `service.refresh`

## 为什么这是“最小风险 Day 4.4”

1. **零运行时改动**
   - 没有改 `app/api/auth.py`
   - 没有改 `AuthService`
   - 没有改现有限流实现

2. **只补当前真实行为的回归入口**
   - 不是新增语义
   - 不是改 refresh contract
   - 只是把现状自动化固定下来

3. **与 Day 4 既有收口逻辑一致**
   - Day 4：`/admin/login` 合同
   - Day 4.1：destructive admin APIs
   - Day 4.2：login 限流替换
   - Day 4.3：admin read routes
   - Day 4.4：`/refresh` 429 + 最小 route contract
