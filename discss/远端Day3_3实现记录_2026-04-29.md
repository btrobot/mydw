# 远端 Day 3.3 实现记录（sessions 列表页迁移 + revoke refresh 模式）

## 本次落地范围

- 将 `/sessions` 从占位页切到真实 React 页面
- 接入 sessions 相关接口：
  - `GET /admin/sessions`
  - `POST /admin/sessions/{id}/revoke`
- 复用 users / devices 已验证过的 revoke 后 refresh 模式

## 具体实现

### 1. auth client

- 新增 sessions 类型与 helper：
  - `AdminSessionRecord`
  - `AdminSessionListResponse`
  - `AdminSessionsFilters`
  - `buildSessionsQuery()`
  - `getAdminSessions()`
  - `revokeAdminSession()`
  - `mapSessionActionError()`

### 2. sessions 页面

- 新增 `remote/remote-admin/src/pages/sessions/SessionsPage.tsx`
- 页面能力：
  - search / auth state / user / device filters
  - 左侧列表 + 右侧详情
  - revoke session
  - revoke 成功后 refresh 当前过滤结果，并重算选中项
- query / revoke 路径统一处理 `token_expired`

## 这次和 users / devices 的一个差异

sessions 当前后端**没有单独的 detail endpoint**，  
所以本页的 detail rail 直接来自当前 filtered list 中的选中 session。

这意味着 Day 3.3 复用的是同一个思路：

- mutation 后回刷后端真值
- 重算当前选中项
- 避免 detail rail 停留在旧状态

只是这里的“detail refresh”来源于 list refresh，而不是单独 detail query。

## 为什么这样做仍然是最小风险

因为没有新增不存在的后端契约，也没有为了“形态统一”硬造前端缓存层。  
当前 slice 直接服从现有后端接口边界，只把 React 路由迁移到可用状态。
