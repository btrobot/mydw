# 远端 Day 4.6 实现记录（self-service 关键 route 的最小 contract 回归入口）

## 本次 slice 目标

- 在 Day 4.5 已补齐 core auth 路由的最小 contract 回归后，继续把 **self-service** 关键路由补进同一条回归链。
- 这次优先关注：
  - `GET /self/me`
  - `GET /self/devices`
  - `GET /self/sessions`
  - `GET /self/activity`
  - `POST /self/sessions/{session_id}/revoke`

本次仍然坚持最小风险原则：
- **不改运行时逻辑**
- **不改前端合同**
- 只把现有 self-service route 的请求转发与错误语义固化为自动化测试

## 为什么这几个 route 是“关键 route”

因为它们基本覆盖了 self-service 的主要交互面：

1. `self/me`
   - 当前用户基础状态

2. `self/devices`
   - 设备列表

3. `self/sessions`
   - 会话列表

4. `self/activity`
   - 行为审计 / 活动流

5. `self/sessions/{session_id}/revoke`
   - 自助撤销会话的唯一 destructive 入口

这组接口足以作为 self-service 的最小 contract 回归面。

## 本次实现

### 1. 新增 self-service API 测试文件

新增文件：
- `remote/remote-backend/tests/api/test_self_service_api.py`

测试方式：
- 使用 `FakeSelfService`
- 只校验 route 层行为
- 不扩展成 `AuthService` 业务逻辑测试

### 2. 固定 self read routes 的请求转发

本次为以下 read routes 固定了：

- `GET /self/me`
- `GET /self/devices`
- `GET /self/sessions`
- `GET /self/activity`

固定内容包括：

#### bearer token 提取与转发

- `Authorization: Bearer access_token` 会被正确提取
- route 会把 access token 原样传给 service

#### query 参数转发

对列表类接口固定：

- `/self/devices`
  - `limit`
  - `offset`

- `/self/sessions`
  - `limit`
  - `offset`

- `/self/activity`
  - `limit`
  - `offset`

### 3. 固定 self read routes 的 401 / 403 语义

本次为上述 read routes 统一固定：

- `401 -> token_expired`
- `403 -> revoked`

并且保持当前 route 行为：
- 缺失 bearer token 时，route 会向 service 传空字符串 `""`
- 最终由 service 决定返回 `401`

### 4. 固定 self destructive route 的最小合同

本次为：

- `POST /self/sessions/{session_id}/revoke`

补齐了最小 contract 回归入口。

#### 成功路径

固定：
- access token 转发
- path 参数 `session_id` 转发
- 成功返回体：

```json
{
  "success": true,
  "session_id": "sess_2",
  "auth_state": "revoked",
  "already_revoked": false
}
```

#### 错误语义

固定：
- `401 -> token_expired`
- `403 -> revoked`
- `404 -> not_found`

其中 `404` 的 message 与 details 也跟当前实现保持一致：

```json
{
  "error_code": "not_found",
  "message": "Requested resource was not found.",
  "details": { "session_id": "sess_2" }
}
```

## 为什么这是“最小风险 Day 4.6”

1. **零运行时改动**
   - 没有改 `app/api/self_service.py`
   - 没有改 `AuthService`
   - 没有改 self-service 业务逻辑

2. **先固化真实 contract，再考虑抽象**
   - 先把 self-service 的 read / destructive 基线补齐
   - 未来如果再抽 contract helper，会更稳

3. **与 admin / auth 的 Day 4 路线一致**
   - 先补最小回归入口
   - 再谈统一 contract harness
