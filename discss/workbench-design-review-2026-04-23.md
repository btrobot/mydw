# 工作台相关页面设计复盘（客观评判）

## 评审范围

本次复盘基于以下实现：

- 路由入口：`frontend/src/App.tsx`
- 工作台：`frontend/src/features/creative/pages/CreativeWorkbench.tsx`
- 作品详情：`frontend/src/features/creative/pages/CreativeDetail.tsx`
- 相关组件：
  - `frontend/src/features/creative/components/CreativeEmptyState.tsx`
  - `frontend/src/features/creative/components/VersionPanel.tsx`

---

## 总体判断

### 产品 / 交互设计

整体方向是对的，而且明显优于传统“任务列表优先”的后台设计。

### 前端实现设计

目前可以工作，但复杂度已经偏高，开始进入“继续叠功能会越来越脆”的阶段。

如果打分：

- 信息架构 / 产品设计：**8/10**
- 前端实现可维护性：**5.5/10**

---

## 做得好的地方

### 1. 主业务对象选对了：以“作品”为中心，而不是以“任务”为中心

从路由层面就能看到这个判断：

- `/creative/workbench`
- `/creative/:id`
- `/task/:id` 只是执行侧诊断 / 追踪入口

这意味着系统明确区分了：

- 日常操作主体是 **作品**
- 任务是执行侧副产物
- AIClip、审核、发布池都围绕作品聚合

这是一个成熟且正确的产品判断。

---

### 2. Workbench 的定位清楚：它是“分诊台”，不是“大而全控制台”

当前工作台承担的是：

- 总览统计
- 高频视角 preset
- 搜索 / 筛选 / 排序
- 进入详情
- 次级打开运行诊断

这很像一个运营分诊台，而不是试图把所有系统信息都堆在首屏。

优点包括：

- summary 卡片在上，列表在下
- 高频视角单独成一排
- 运行诊断不是主入口，而是次级入口抽屉

说明主流程和辅助流程是有意区分的。

---

### 3. URL 状态可分享、可恢复，对运营后台是加分项

Workbench 明确把以下状态写进 URL：

- `keyword`
- `status`
- `poolState`
- `preset`
- `sort`
- `page`
- `pageSize`

详情页又保留了 `returnTo` 返回路径。

这带来的价值非常实际：

- 刷新不丢状态
- 从详情返回能回到原始工作台视图
- 方便 E2E 测试
- 内部协作可复制链接

这属于后台系统里成熟且正确的设计。

---

### 4. 领域边界表达得比较清楚

从详情页和版本面板可以看出，页面在努力区分以下几个概念：

- 作品定义
- 版本结果
- 审核结论
- 发布包冻结值
- 执行任务
- 发布池候选项

这比很多“后端怎么返回，前端就怎么堆”的页面要好得多。

至少从产品语言和页面结构上，已经在避免把这些对象混成一团。

---

### 5. 空态 / 错态 / loading 都有显式设计

不是只做 happy path，而是对下面几种状态都做了专门处理：

- Workbench loading / error / empty
- Detail loading / error / empty

对内部运营后台来说，这点很重要。接口异常时，用户至少能知道发生了什么。

---

## 主要问题

### HIGH 1. Workbench 的状态源过多，设计上已经脆了

目前 Workbench 同时存在三套状态：

1. URL search params
2. ProTable 内部 form / request 状态
3. React Query 外部数据状态

这类结构最容易出现的问题就是状态不同步。近期已经出现过的 preset 相关 bug，本质上就是这个问题的直接表现。

#### 为什么危险

因为以后每增加一个状态维度，比如：

- 更多 preset
- tab
- 更多排序
- 批量操作
- 诊断筛选

都会继续放大同步复杂度。

#### 客观评价

现在不是“不能用”，但已经出现明显的架构摩擦信号，不适合继续无节制往上加状态逻辑。

---

### HIGH 2. 当前是在“借用 ProTable”，但没有真正顺着它的状态模型走

现在的实现里：

- 真正的数据请求来自 `useCreatives(...)`
- `ProTable.request(...)` 更像是一个状态桥接器，而不是实际请求入口

结果是：

- 表面上像 ProTable 驱动
- 实际上是 React Query 驱动
- 中间再夹一层 ProTable 自己的 form / request 生命周期

这会导致实现理解成本高，也更容易出现细碎但烦人的一致性 bug。

---

### HIGH 3. CreativeDetail 页承担的职责太多，已经接近“超级页面”

详情页同时覆盖了：

- 作品基本信息编辑
- 素材编排
- profile / product / material 选择
- 保存
- 提交合成
- 审核入口
- 版本历史
- 发布池记录
- 高级诊断
- AIClip 抽屉
- 跳转 task diagnostics

这说明它还没有完全失控，但已经到了应该警惕的程度。

#### 风险

对用户：

- 信息量大
- 学习成本高
- 页面很容易让非熟练用户感到压迫

对开发：

- 改一个区域容易影响另一个区域
- 页面逻辑持续膨胀
- 难以明确“这页的主任务到底是什么”

---

### MEDIUM 4. 诊断设计思路正确，但当前偏“解释型”，不够“行动型”

现在的诊断视图在说明模型边界方面做得很好，例如会说明：

- 任务不是作品定义
- Package 不是版本本身
- 运行诊断只是执行侧观察

但它目前更偏向于：

- 信息说明
- 结构展示
- 对账展示

而不是优先告诉用户：

- 当前最该处理什么
- 为什么被阻塞
- 下一步应该去哪里

这意味着它对熟悉系统的人很有价值，但对日常运营用户来说，阅读成本还是偏高。

---

### MEDIUM 5. 页面在“领域概念清晰”和“认知负担偏高”之间摇摆

这不是纯缺点，而是当前设计的真实 tradeoff：

优点：

- 没有把模型做假
- 领域概念清晰

代价：

- 页面需要较多解释文案
- 用户必须理解更多内部机制

如果目标用户是熟练运营 / 审核 / 诊断角色，这个取向是合理的。  
如果未来用户中包含更多轻度用户，这套设计会显得偏重。

---

### LOW 6. Workbench 首屏主次还可以继续收紧

当前 Workbench 首屏已经包含：

- 入口模式说明
- 运行诊断入口
- summary
- 搜索 / 筛选
- 高频视角
- 列表
- sort
- toolbar

目前还没有乱，但如果继续往首屏叠更多控制项，就很容易走向“控制中心化”。

---

## 最值得肯定的设计决策

如果只选一个最好的决定，我会给这个：

> 把“作品”定义为主对象，把任务 / 诊断 / 发布池降为围绕作品的辅助视角。

这是会长期影响系统质量的正确产品判断。

---

## 最该警惕的实现问题

如果只选一个最该警惕的问题，我会给这个：

> Workbench 的状态管理已经出现架构性摩擦。

现在还在可修范围内，但已经开始依赖：

- 手动 `setFieldsValue`
- URL 同步
- `ProTable.request` 桥接
- 通过 `key` 触发 remount

来兜一致性。

这说明它该收敛了，而不是继续横向加功能。

---

## 最终结论

一句话总结：

> 这套“工作台 + 作品详情 + 诊断抽屉 + 任务追踪”的产品设计是成熟且有方向感的；但前端实现层已经明显比产品结构更脆，后续如果继续加功能，维护成本会快速上升。

### 审核意见

- 产品设计：**偏正面，值得继续沿这个方向打磨**
- 实现设计：**建议进入一次结构收敛阶段，而不是继续横向堆功能**

---

## 建议的下一步

### P1

Workbench 不要再让 URL / ProTable / React Query 三方互相推状态。  
应收敛成一个单一状态源。

### P2

把 `CreativeDetail` 拆成更明确的页面内分区，甚至子路由 / Tab：

- 创作输入
- 版本与审核
- 发布与诊断

### P3

把“解释性文案”逐步转成“行动建议”：

- 当前状态
- 阻塞原因
- 下一步入口

---

## 补充判断

从当前实现看，这个项目最可惜的地方不是“产品方向错了”，而是：

> 产品结构比前端状态实现更成熟。

也就是说，方向其实是对的，真正需要控制的是实现复杂度，而不是重新推翻页面定位。
