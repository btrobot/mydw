# 远端 Day 4.2 实现记录（登录限流替换：SQLite 持久化，前端合同不变）

## 本次 slice 目标

- 落地 **登录限流替换方案**，优先解决 `InMemoryRateLimiter` 的进程内失效问题。
- **尽量不改前端合同**：
  - `/login` 继续返回既有的 `429 too_many_requests`
  - `/admin/login` 继续返回既有的 `429 too_many_requests`
  - 既有 `401 / 403` 语义不变
- 控制改造范围，只替换 **登录入口** 的限流基础设施，不顺手扩大到全部鉴权接口。

## 方案评估

本次优先评估过三类方案：

### 方案 A：网关层限流（Nginx / API Gateway）

优点：
- 天然跨进程
- 对应用层侵入小

问题：
- 当前仓库内没有现成网关接入与联调基线
- 本 slice 需要同时补齐本地开发、测试、CI 的可验证路径，落地成本偏高
- 很容易把“登录语义收口”问题外溢成“部署架构改造”

结论：
- **不作为本次最小风险方案**

### 方案 B：Redis-backed limiter

优点：
- 语义清晰，扩展性好
- 更适合多实例场景

问题：
- 需要新增基础设施依赖
- 当前仓库约束是 **无显式请求不新增依赖**
- 本 slice 的主要目标是先把“进程重启即失忆”的问题替掉，而不是引入新的运维面

结论：
- **保留为后续演进方向，不作为本次首选**

### 方案 C：SQLite 持久化限流（本次采用）

优点：
- 使用 Python 标准库 `sqlite3`，**不新增依赖**
- 可直接落在当前 `data/` 目录，符合现有 backend 本地运行方式
- 能覆盖最关键的风险点：**跨进程 / 重启后仍能保留窗口内限流状态**
- 对 API 合同零改动，只替换底层存储

风险控制：
- 仅用于 `login` / `admin login`
- 保留内存限流兜底，避免 SQLite 异常直接把登录打挂
- `refresh` 仍保留现状，避免本 slice 范围膨胀

结论：
- **这是当前仓库约束下的最小风险替换路径**

## 本次实现

### 1. 新增 SQLite 持久化限流器

更新：
- `remote/remote-backend/app/core/rate_limit.py`

新增：
- `SqliteRateLimiter`
- `create_rate_limiter(...)`

实现要点：
- 使用独立 SQLite 文件存储窗口内尝试记录
- 以 `scope + key` 作为限流桶维度
- 每次判定前清理窗口外旧记录
- 命中阈值后返回 `False`
- SQLite 出错时回退到 `InMemoryRateLimiter`，优先保证登录可用性

### 2. 为登录限流增加独立配置

更新：
- `remote/remote-backend/app/core/config.py`

新增配置：
- `LOGIN_RATE_LIMIT_BACKEND`
- `LOGIN_RATE_LIMIT_SQLITE_PATH`

默认值：
- backend 默认 `sqlite`
- 路径默认落到 `data/login_rate_limits.sqlite3`

这样做的目的是：
- 让这次替换保持可配置、可回退
- 不需要修改前端，也不需要改动主业务数据库 schema

### 3. 仅替换 `/login` 与 `/admin/login`

更新：
- `remote/remote-backend/app/api/auth.py`
- `remote/remote-backend/app/api/admin.py`

结果：
- `login_rate_limiter` 改为持久化 limiter
- `admin_login_rate_limiter` 改为持久化 limiter
- `refresh_rate_limiter` **暂时不动**

这是有意的范围控制：
- 用户要求是“登录限流替换”
- `refresh` 的行为与压测模式不同，单独评估更稳

## 测试补强

### 1. 核心限流器测试

覆盖：

```json
  "message": "Too many login attempts, please retry later.",
  "details": {}
}
```

### 3. 既有 admin login 测试继续作为回归基线

保留：
- `remote/remote-backend/tests/api/test_admin_login_api.py`

用于确认：
- `/admin/login` 的 `429` 语义没有被本次替换改坏

## 为什么这是“最小风险”落地

1. **只换底层，不换合同**
   - 前端看到的状态码、`error_code`、文案都不变

2. **只处理最关键入口**
   - 先覆盖 `/login` 与 `/admin/login`
   - 不顺手扩散到所有 rate limit 使用点

3. **不引入新基础设施**
   - 没有 Redis
   - 没有网关编排
   - 没有主库 schema 改造

4. **保留兜底**
   - SQLite 不可用时仍可回退内存限流
   - 优先保证服务可用性

## 当前边界

- 这次 **没有** 统一替换 `refresh` 限流
- 这次 **没有** 引入网关层限流
- 这次 **没有** 做多实例 / 分布式级别的强一致限流承诺
- 这次的目标是：先把“单进程内存桶”升级为“本地持久化桶”
