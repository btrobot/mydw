# 远端后续增强 backlog（按优先级）

日期：2026-04-26

## 说明

本清单不再属于“主线未完成项”，而是主线完成后的增强 backlog。

优先级分为：

- **P0：建议尽快做**
- **P1：下一轮优先**
- **P2：有收益但不紧急**

---

## P0：建议尽快做

### 1. 输出 remote-admin 测试覆盖说明 / 维护文档

**目标**

- 让后续维护者知道：
  - 哪些链路已有 contract test
  - 哪些链路已有 browser regression
  - 哪些动作只靠代表性回归覆盖

**收益**

- 降低后续修改时的认知成本
- 防止误以为“没测到”或“已经全测了”

**建议产出**

- 一页覆盖矩阵说明
- 一页新增页面/接口时应该补什么测试的 checklist

### 2. 为 step-up 浏览器回归补维护入口说明

**目标**

- 说明 `test:react-step-up` 的用途、运行方式、串行执行原因

**收益**

- 避免后续维护者擅自改回并发，导致不稳定
- 降低浏览器测试的上手成本

### 3. 收敛当前分散的状态文档入口

**目标**

- 把“状态看板 / 完成度总结 / backlog / 测试矩阵”之间的关系整理清楚

**收益**

- 对汇报、交接、复盘都更友好

---

## P1：下一轮优先

### 4. 细化 step-up 浏览器级 failure 矩阵

**当前现状**

- users / devices / sessions 已有代表动作的 success / failure / token_expired 回归
- restore / unbind / rebind 目前只补了最小差异动作回归

**增强方向**

- 对 restore / unbind / rebind 继续补：
  - verify failure
  - token_expired
  - 可选的 forbidden / too_many_requests 文案回归

**收益**

- 对共享 step-up 流程形成更完整的 UI 级护栏

### 5. 评估 remote-admin 测试依赖内聚

**当前现状**

- 浏览器级测试复用了 workspace 内已有 Playwright

**增强方向**

- 评估是否需要把浏览器测试依赖显式收敛到 `remote/remote-admin`

**收益**

- 子项目独立性更强
- CI / 本地维护更直观

**注意**

- 若引入新依赖，要评估是否值得增加维护成本

### 6. 为新的 destructive admin route 建“接线模板”

**目标**

- 总结出统一模式：
  - `useAdminStepUp`
  - `createAdminAuthHeaders`
  - mutation refresh
  - browser regression 最小模板

**收益**

- 后续新页面/新动作接入 step-up 时更快、更稳

---

## P2：有收益但不紧急

### 7. 构建体积 / chunk 拆分优化

**当前现状**

- `vite build` 仍有 chunk size warning

**增强方向**

- 评估按页面或依赖做 `manualChunks`
- 适度做懒加载

**收益**

- 改善构建产物结构
- 为后续页面增量扩展留余量

### 8. 补更细粒度的 visual / UX 回归

**增强方向**

- 对关键管理页增加更明确的交互/文案/状态一致性检查

**收益**

- 提升非功能性稳定度

### 9. 继续收敛测试样板

**增强方向**

- 将现有 browser harness 再抽象一层复用工具

**收益**

- 减少重复 mock 代码
- 降低后续新增浏览器回归成本

---

## 推荐执行顺序

如果只做一轮后续增强，建议顺序：

1. 测试覆盖说明文档
2. step-up 浏览器回归维护说明
3. 细化 step-up failure 矩阵
4. 评估 remote-admin 测试依赖内聚
5. 构建体积优化

---

## 一句话建议

后续不要再按“补主功能”思路推进，而应转成：

> **优先做文档化、测试矩阵透明化、step-up 回归加固和工程化收口。**
