# UI Gap Audit 与优先级表

> 日期：2026-04-24  
> 对应目标稿：`discss/ui-gap-page-expectations-2026-04-24.md`  
> 前置讨论：`discss/ui-gap-productization-plan-2026-04-24.md`

---

## 1. 文档目的

本文件把“页面与期待之间的差距”整理成一份可执行的审计清单，用于：

1. 明确当前主要问题在哪里
2. 给出优先级，而不是平铺并列 backlog
3. 为后续 PRD / slice 拆分提供入口

一句话：

> **不是再重复说“差距大”，而是把差距变成有优先级的行动项。**

---

## 2. 当前总体判断

整体上，当前前端的主要矛盾不是“完全没方向”，而是：

- 主业务方向已经对了（Creative-first）
- 但界面仍保留较强的过渡态 / 诊断台痕迹
- 页面层级、默认信息密度、可管理性、文案统一性还没收口

因此，后续最合理的策略不是大重写，而是：

> **围绕已成立的主业务方向，做渐进式产品化收口。**

---

## 3. 差距总表

| 页面/区域 | 期待形态 | 当前主要差距 | 优先级 |
| --- | --- | --- | --- |
| CreativeWorkbench | 日常分诊台 / 管理台 | 仍偏工程态；管理能力需继续收口 | P0 |
| CreativeDetail | 作品定义与审核中心 | 职责过载；业务层与诊断层混放 | P0 |
| 文案与页面四态 | 统一、稳定、面向中文用户 | 中英混搭；坏文案；四态不统一 | P0 |
| AIClip | 产品工作流页 | 更像工具面板；低层参数感偏重 | P1 |
| Dashboard | 高层概览页 | 仍容易带“解释型入口”痕迹 | P1 |
| Task 层 | 执行/追踪层 | 仍需持续压住其“反客为主”的倾向 | P1 |
| Settings/Auth/Profile/Schedule | 配置与治理层 | 需要统一策略页表达和状态反馈 | P1 |
| 响应式/布局密度 | 非最大化窗口仍稳定可用 | 多处布局密度与窄窗口表现不足 | P2 |

---

## 4. P0：必须优先收口的问题

## 4.1 P0-A Workbench 可管理性与主入口质感

### 问题

虽然 Workbench 已经不是简单卡片墙，但距离“真正高效的日常工作台”仍有差距。

### 当前差距

- 还需要继续强化高频视角
- 默认业务层里仍带有一定诊断气味
- 作品量增长后，管理效率风险仍在

### 收口目标

- 用户能快速找出待处理作品
- 常用筛选和排序不需要重复手工组合
- 页面默认只强调“今天该处理什么”

### 推荐动作

1. 再次审视 Workbench 首屏信息层级
2. 确认 preset / 筛选 / 排序是否真的贴近日常使用
3. 持续压缩默认可见的工程诊断信息
4. 把“运行诊断”牢固放在次级入口

---

## 4.2 P0-B CreativeDetail 的职责过载

### 问题

CreativeDetail 目前承担的信息和动作太多，已经接近超级页面。

### 当前差距

- 作品定义信息
- 版本审核信息
- 发布相关信息
- 运行/诊断信息

这些内容虽然都有关联，但不应该默认处于同一层级。

### 收口目标

- 首屏先回答：
  - 当前作品定义是否完整
  - 当前版本状态是什么
  - 是否可以进入下一步
- 诊断信息通过二级区域/Drawer/Tab 进入

### 推荐动作

1. 明确 Detail 首屏只保留业务主语义
2. 把 shadow diff / scheduler / raw diagnostics 下沉
3. 重排主 CTA，避免按钮层级互相抢位置

---

## 4.3 P0-C 文案与四态统一

### 问题

即使功能逻辑是对的，若文案、状态反馈、空态/错态不统一，页面仍会显得“不像完整产品”。

### 当前差距

- 中英混搭
- 坏文案/异常字符
- 成功、空、错、加载的表达方式不一致

### 收口目标

- 形成统一中文主文案策略
- 统一 loading / empty / error / success 四态模板
- 让页面表达风格稳定下来

### 推荐动作

1. 先做一轮文案 inventory
2. 锁定语言策略
3. 提炼页面四态模式

---

## 5. P1：第二阶段重点

## 5.1 P1-A AIClip 产品化

### 当前差距

- 更像工程工具面板
- 输入、过程、输出三者关系不够显性
- 高级参数与主流程层级容易混

### 目标

- 让 AIClip 更像一条明确的创作工作流
- 把高级参数降级为次级能力

---

## 5.2 P1-B Dashboard / Task / 系统治理页的角色稳定化

### 当前差距

- Dashboard 还容易承担解释型职责
- Task 层仍需防止重新成为业务中心
- Settings/Auth/Profile/Schedule 的产品表达还可继续统一

### 目标

- Dashboard 只做概览和导流
- Task 明确是执行层
- 配置页像策略面，而不是参数散表

---

## 6. P2：第三阶段优化项

## 6.1 响应式与布局密度

### 当前差距

- Electron 非最大化窗口下，部分页面的信息密度与布局弹性仍不理想

### 目标

- 让工作台、详情页、Drawer、统计卡在较窄窗口下仍清晰可用

### 说明

这项重要，但更适合放在前两轮主语义与层级收口之后。

---

## 7. 不建议的执行方式

以下做法不建议采用：

1. 一次性大改所有页面
2. 一边做 UI，一边顺手大改领域模型
3. 不设优先级，所有问题并列推进
4. 只修“看起来最显眼的地方”，不解决页面职责和层级问题

---

## 8. 建议执行顺序

推荐顺序如下：

### 第一轮

- Workbench 可管理性与默认业务层收口
- CreativeDetail 业务层 / 诊断层分层
- 文案与四态统一

### 第二轮

- AIClip 产品化
- Dashboard / Task / Settings / Auth 等角色稳定化

### 第三轮

- 响应式与布局密度专项优化

---

## 9. 如何把这份审计转成执行计划

推荐转化方式：

1. 从 P0 中选一条作为单独主线
2. 写一份 focused PRD
3. 再写对应 test spec
4. 最后按 2~4 个 slice 执行

推荐优先从下面两条二选一开始：

- **主线 A：CreativeDetail 分层收口**
- **主线 B：文案与四态统一**

如果你的感受是“页面虽然能用，但整体产品感不够”，那通常更建议先做：

> **主线 A + 主线 B**

因为它们最直接影响“看起来像不像成品”。

---

## 10. 一句话收尾

这份 Gap Audit 的核心结论是：

> **当前 UI 的问题已经不再是“功能有没有”，而是“产品层面的收口有没有完成”。**
