# 远端 Day 3.2 实现记录（devices 列表页迁移 + mutation refresh 模式复用）

## 本次落地范围

- 将 `/devices` 从占位页切到真实 React 页面
- 接入 devices 相关接口：
  - `GET /admin/devices`
  - `GET /admin/devices/{id}`
  - `POST /admin/devices/{id}/unbind`
  - `POST /admin/devices/{id}/disable`
  - `POST /admin/devices/{id}/rebind`
- 复用 Day 3.1 在 users 页跑通的“query/detail/mutation refresh”模式

## 具体实现

- 新增 devices 类型与 helper：
  - `AdminDeviceRecord`
  - `AdminDeviceListResponse`
  - `AdminDevicesFilters`
  - `AdminDeviceRebindRequest`
  - `buildDevicesQuery()`
  - `getAdminDevices()`
  - `getAdminDeviceDetail()`
  - `unbindAdminDevice()`

  - `mapDeviceActionError()`

### 2. devices 页面

- 新增 `remote/remote-admin/src/pages/devices/DevicesPage.tsx`
- 页面能力：
  - search / device status / bound user 过滤
  - 左侧列表 + 右侧详情
  - unbind / disable / rebind 操作
  - mutation 成功后执行：
    1. refresh 列表
    2. 重算选中 device
    3. 若仍有选中项，再主动 refresh detail
- 对 `token_expired` 在 query / detail / mutation 路径统一回退登录页

### 3. 路由

- `routes/router.tsx`
  - `/devices` 由 `PendingPage` 切换到 `DevicesPage`
  - 占位页说明同步更新为“users + devices 已迁移”

## 为什么继续沿用 users 模式

因为 devices 页和 users 页有相同风险：

- mutation 发生后，列表和详情可能不同步；
- 当前筛选条件下，被操作对象可能从结果集消失；
- token 过期如果只在 query 路径处理，mutation 仍会漏掉边界。

所以这次不重新发明另一套状态逻辑，而是直接复用：

- list query
- detail query
- mutation success 后 list-then-detail refresh
- mutation / query 统一 auth-expired redirect
