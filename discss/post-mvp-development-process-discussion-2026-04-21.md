# 从 0 到 MVP 之后，开发流程如何继续往前走

> Date: 2026-04-21  
> Type: Discussion / 非正式分析沉淀  
> Status: Active  
> Purpose: 记录你当前的困惑、我的问题总结，以及对后续开发流程的建议

---

## 1. 你的原始问题

你描述的现状大致是：

- 项目从 0 开始时，有一个初始 PRD
- 基于 PRD，分析了核心业务流程、边界、数据库设计草稿、接口初稿
- 然后做了系统规划设计和实施步骤
- 每个步骤先 `ralplan`
- 再切分 PR
- 再用 `ralph` 实现 PR
- 最终完成了 MVP

你现在的困惑主要有三类：

1. **MVP 过程中陆续输出了很多文档，后面如何把这些文档体系化，并在后续开发中真正起到规范和指引作用？**
2. **前面边界设计只是初稿，后面不断被修正，前期规范逐渐过时，这种情况应该如何处理？**
3. **MVP 之后到底应该往哪个方向走？如何一步一步继续推进，而不是陷入“到处都能做、但不知道先做什么”的状态？**

---

## 2. 我对你问题的总结

我认为你现在遇到的，不是“不会做开发”，而是一个很典型的 **MVP 后治理问题**：

- **前期的问题是：怎么把东西做出来**
- **后期的问题是：怎么让系统持续可演进**

也就是说，你已经跨过了“从 0 到 1”的混沌阶段，现在进入的是“从 1 到 N”的治理阶段。  
这个阶段的核心难点，不再是单个功能怎么实现，而是：

- 哪些文档还是活的
- 哪些规范已经失效
- 哪些设计是当前真相
- 下一步该押注哪条主线
- 如何让开发节奏持续，而不是每做一段就重新迷路

所以你的困惑，本质上不是执行力问题，而是 **“缺少一套 MVP 之后的收口 + 续航机制”**。

---

## 3. 根因分析

### 3.1 从 0 到 MVP 时，文档天然是“推进型”的

前期文档的主要作用是：

- 帮助形成共识
- 帮助拆解问题
- 帮助开始施工

所以那时候产生的很多文档，本质上是：

- 讨论稿
- 规划稿
- 假设稿
- 临时边界稿

这些文档在那个阶段非常有价值，但它们的职责不是“长期真相”，而是“帮助推进”。

### 3.2 MVP 之后，问题从“有没有文档”变成“谁说了算”

文档一多，真正的问题不是缺文档，而是：

- 文档层级不清
- 有些是 proposal，有些是 current truth，但外观看起来差不多
- 同一主题有多份文档，互相覆盖
- 老文档还在，新文档也在，团队不知道该信哪份

所以真正要解决的，是 **文档 authority（权威性）** 问题，而不是单纯继续补文档。

### 3.3 边界被修正不是失败，而是正常演进

前期边界设计被修正，完全正常。

因为：

- 初始边界是在信息不完整时做的
- MVP 的目的之一，就是暴露真实边界
- 真实系统一旦跑起来，很多“设计上的正确”会被“运行时的真实约束”修正

所以问题不是“前期设计改了，怎么办”，而是：

> 如何让“被修正”成为一种可管理的机制，而不是让系统逐渐堆满过时规范。

### 3.4 MVP 后最大的风险不是做错，而是没有主线

MVP 后通常会同时出现很多事情都值得做：

- 补文档
- 补测试
- 修架构
- 提升 UX
- 提升自动化
- 推进下一个大功能

如果没有一个明确主线，就会出现：

- 每件事都对
- 但做起来很散
- 做完一轮之后系统还是不稳定
- 团队一直忙，但没有明显阶段感

所以 MVP 后最重要的不是“继续多做”，而是 **重新选择主线**。

## 4. 文档体系应该怎么收口

我建议你把文档明确分成 4 层。

### 4.1 第一层：Current / Canonical（当前真相层）

这一层回答：

- 现在系统是什么样
- 现在谁说了算
- 现在开发应该遵守什么

这类文档应该尽量少，但必须稳定：

- 当前架构基线
- 当前运行时真相
- 域模型真相
- 页面规格真相
- 验证基线
- 当前 backlog / 下一阶段入口

特点：

- 数量少
- 可直接指导开发
- 允许被更新，但不允许长期过时

### 4.2 第二层：Working / Planning（工作规划层）

这一层回答：

- 下一步准备怎么做
- 当前阶段准备拆哪些 PR
- 哪些方案还在收敛

比如：

- `.omx/plans/` 中仍然活跃的 PRD / test spec / PR sequence
- 当前阶段的执行计划

特点：

- 生命周期短
- 一旦被正式文档吸收，就应该归档或下沉
- 不应该长期和 canonical docs 竞争

### 4.3 第三层：Reports / Closeout（验收与收口层）

这一层回答：

- 某阶段做完了什么
- 实际交付了什么
- 还有哪些残留风险

比如：

- closeout summary
- acceptance report
- release gate

特点：

- 它不是 current truth
- 但它是“为什么 current truth 变成这样”的证据

### 4.4 第四层：Archive / Historical（历史层）

这一层回答：

- 以前怎么想过
- 为什么后来不这么做了
- 历史上下文是什么

特点：

- 保留历史价值
- 不参与当前开发决策
- 只用于追溯，不用于指导日常实现

## 5. 前期规范过时了，应该怎么办

核心原则不是“删掉旧文档”，而是：

> **允许旧文档存在，但必须让它失去当前指导权。**

我建议用下面这套机制。

### 5.1 给文档显式标状态

每份重要文档都至少有：

- `Status: Proposed`
- `Status: Active`
- `Status: Historical`
- `Status: Stale`
- `Status: Archived`

这样读的人一眼就知道：

- 这份是方案草稿
- 还是当前真相
- 还是历史参考

### 5.2 一个主题只允许一个“当前权威入口”

例如一个主题最多只能有一个：

- 当前商品域真相
- 当前任务域真相
- 当前页面规格真相

其他文档可以存在，但都必须：

- 指向这个入口
- 或者说明自己只是历史上下文

### 5.3 不要求“所有旧文档都更新”，但要求“入口文档必须正确”

这点很重要。

如果你试图把所有老文档都修到完全一致，会非常累，也不现实。  
更合理的是：

- 先确保入口文档正确
- 再把高可见度、易误导的旧文档标 stale / redirect
- 最后逐步归档低价值旧文档

也就是说：

- **先保入口**
- **再控误导**
- **最后清历史**

### 5.4 每次阶段收口时做一次“文档吸收”

建议把这个动作变成固定流程：

1. 该阶段规划完成实现
2. 验证通过
3. 把实现结果吸收到正式文档
4. 把阶段 planning 文档移到 archive
5. 更新 retention / authority / checklist

这样文档不会一直膨胀。

### 5.5 用“文档真相测试”保护高价值事实

对特别容易漂移的关键点，可以像现在这样用测试兜住，例如：

- 创建接口字段是否已改变
- 页面规格是否还在描述旧流程
- 权威入口是否指向正确文档

这类测试的价值很高，因为它能防止“代码改了，文档没改”。

## 6. MVP 之后该往哪个方向走

我建议你不要把问题理解成：

> “后面还有很多事，到底先做哪件？”

而是理解成：

> “MVP 之后，下一阶段的主线是什么？”

### 6.1 主线选择要看 3 个维度

每个候选方向都用这 3 个维度看：

#### A. 业务价值

它能不能明显提升：

- 用户可用性
- 交付能力
- 核心流程打通程度

#### B. 系统杠杆

它是不是“做完以后会让后面很多事情都更顺”的那种工作？

比如：

- 文档真相收口
- 验证基线稳定
- 核心工作流稳定
- 数据边界收敛

#### C. 风险 / 不确定性

如果现在不做，会不会导致后面：

- 继续返工
- 继续边界漂移
- 继续难以规划

### 6.2 MVP 后常见的四类主线

通常会从下面 4 类里选一条作为下一阶段主线：

#### 方向 1：真相收口 / 基线治理

目标：

- 把系统当前真实状态收拢清楚
- 为下一阶段开发打基础

适用场景：

- 文档多、口径乱
- 边界漂移严重
- 继续开发前需要先“盘一盘”

这其实就是你现在已经在做的事情。

#### 方向 2：核心工作流稳定化

目标：

- 让 MVP 的主流程真正稳定可用

适用场景：

- 功能看起来有了
- 但成功率、可维护性、可验证性还不够

#### 方向 3：体验与效率收口

目标：

- 把前端 UX、操作闭环、错误反馈、维护成本收紧

适用场景：

- 系统主流程基本可用
- 但“能用”和“好用”之间差距很大

#### 方向 4：下一阶段产品能力扩展

目标：

- 在稳定基线上，正式开启第二阶段业务能力建设

适用场景：

- current truth 已经比较稳
- baseline 已经可依赖
- 团队可以放心扩展主功能

### 6.3 你现在最适合的，不是直接跳大功能，而是“收口后再起跑”

结合你的描述，我的判断是：

你现在最需要的不是立刻做更多新功能，  
而是把这件事完成：

> **先把 MVP 的真实状态收清楚，形成一个可依赖的当前基线，然后再从这个基线启动下一阶段。**

换句话说，当前最合理的顺序是：

1. **收口**
2. **定主线**
3. **再进入下一阶段开发**

## 7. 我建议的“下一阶段推进方法”

我建议你把 MVP 之后的推进节奏固定成一个循环。

## 7.1 阶段 0：收口

目标：

- 当前真相清楚
- 当前边界清楚
- 当前验证基线清楚

产出：

- current architecture / runtime truth
- verification baseline
- authority matrix
- next-phase kickoff
- active backlog

## 7.2 阶段 1：选主线

只回答一个问题：

> 接下来 4~8 周，我们最重要的一条主线是什么？

注意：

- 只能选一条主线
- 其他都进 backlog
- 不要同时推进 3 条大主线

## 7.3 阶段 2：做下一阶段启动包

为这条主线准备：

- next-phase PRD
- next-phase test spec
- execution breakdown / PR sequence

这样进入执行前，你就知道：

- 为什么做
- 先做什么
- 怎样算完成

## 7.4 阶段 3：按 PR 序列推进

每个 PR 继续用你已经验证有效的方法：

- `ralplan`
- 切 PR
- `ralph` 实现
- 验证
- closeout

## 7.5 阶段 4：阶段收口

每做完一个阶段，固定执行：

- 吸收正式文档
- 归档阶段 planning
- 更新 baseline / backlog
- 写 closeout report

然后再进入下一轮。

## 8. 一个更适合你的认知切换

你现在最需要做的一个认知切换是：

### 从：

- PRD 驱动
- 设计驱动
- 任务驱动

### 切到：

- **基线驱动**
- **主线驱动**
- **阶段收口驱动**

也就是说，MVP 之后最重要的不再是“想到什么就继续做什么”，而是：

1. 当前基线是什么
2. 下一阶段主线是什么
3. 这条主线的完成边界是什么
4. 做完后如何沉淀成新的 current truth

## 9. 对你当前处境的结论

如果我把你的问题压缩成一句话，我会这样总结：

> 你已经证明了“能把项目从 0 做到 MVP”，现在卡住的是“如何把 MVP 变成可持续演进的产品基线”。

而这个问题的解法不是继续盲目前进，而是建立下面这套机制：

- 文档分层
- 权威入口
- 过时规范降级
- 阶段收口
- 主线优先
- 启动包先行

---

## 10. 我给你的直接建议

如果只保留最实用的 6 条建议，我会给你这 6 条：

1. **不要再把所有文档都当成同一层级。** 先划分 current / working / reports / archive。
2. **每个主题只保留一个当前权威入口。**
3. **旧规范过时是正常现象，不要纠结“为什么变了”，而要建立 stale / archive 机制。**
4. **MVP 后不要同时开很多大方向，只选一条下一阶段主线。**
5. **每个阶段都必须有 closeout，把 planning 吸收到正式文档。**
6. **把“继续开发”改造成“基于当前基线启动下一阶段”。**

---

## 11. 一个适合你现在的实际动作顺序

如果明天就要继续推进，我建议按这个顺序：

1. 确认当前 canonical docs 是否够用  
2. 把仍然会误导人的旧文档标 stale / 归档  
3. 把 open issues 压缩成 next-phase backlog  
4. 只选一个 next-phase 主线  
5. 为这条主线做 PRD / test spec / PR breakdown  
6. 再进入执行

---

## 12. 最后一句

你现在的困惑，其实说明项目已经到了一个更成熟的阶段。  
从“怎么做出来”走到“怎么让它持续发展”，本身就是进步。

所以不用把现在的混乱理解成失败，更准确的理解应该是：

> 你已经到了需要建立“项目治理能力”的时候了。

而一旦治理能力建立起来，后面的开发会比前面的 MVP 阶段轻松很多。
