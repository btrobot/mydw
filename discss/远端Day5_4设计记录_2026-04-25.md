# 远端 Day 5.4 设计记录（Argon2 平滑升级）

## 本轮结论

Day 5.4 只做设计，不改运行时。

最终建议已经固化到：

- `discss/specs/remote-password-hash-argon2-adr-v1-2026-04-25.md`

## 一句话决策

采用：

- **账户密码双栈校验**
- **Argon2id 作为默认新写入算法**
- **PBKDF2 旧密码继续可登录**
- **登录成功后 opportunistic rehash**
- **refresh token 暂不跟着改**
- **step-up runtime 延后到 Day 5.6 仅出设计稿**

## 关键风险结论

最大的隐藏风险不是“Argon2 本身”，而是：

- 当前 `hash_password()` / `verify_password()` 同时被账户密码和 refresh token 复用

所以 Day 5.5 真正第一步不应该是“直接改默认算法”，而应该是：

1. 先把账户密码 helper 和 refresh token helper 拆开
2. 再切账户密码默认算法到 Argon2id

## 对 admin / end-user 的最小收口

- end-user：继续复用现有 `user_credentials.password_algo`
- admin-user：补 `admin_users.password_algo`

本轮建议 **只把 `password_algo` 作为 Day 5 必选 schema 变更**；
`password_updated_at` 不是本轮最小必需项，可后移。

## Day 5.5 推荐落地顺序

1. helper 拆分（先不改行为）
2. 配置补 Argon2 参数
3. Alembic 补 `admin_users.password_algo`
4. 双栈 verify + opportunistic rehash
5. 新 seed / bootstrap 默认写 Argon2

## 验收口径

- 新密码默认 Argon2id
- 旧 PBKDF2 不强制重置
- admin / end-user 均支持平滑升级
- refresh token 合同不漂移
- 不改前端合同
- 不实现 step-up runtime
