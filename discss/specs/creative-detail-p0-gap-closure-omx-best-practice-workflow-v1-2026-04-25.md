# 《CreativeDetail P0 缺口补齐：OMX 最佳实践流程 v1》

> 日期：2026-04-25  
> 适用范围：CreativeDetail / 当前剩余 P0 缺口补齐  
> 状态：Draft

---

## 1. 先说结论

如果下面三件事还没完成：

1. 商品选择入口仍残留 Select 心智  
2. 封面默认采用 / 视频默认全选规则未闭环  
3. 回到编辑 / 确认结果 / 发布 / 重新生成 的版本边界证据未补强  

那么按 OMX 最佳实践，**不应该把它当“已经收口，只差备注”处理**，而应该：

> **把 P0 重新拉回“缺口补齐流程”，按“规则收敛 -> 切片实现 -> 自动化证明 -> 独立验收”四段走完。**

也就是说：

> **不是继续写总结，而是进入一个小型的 P0-R1 收尾执行闭环。**

---

## 2. OMX 下正确的处理方式

推荐流程不是直接继续改代码，而是下面这条链路：

### Phase 1：重新定性缺口

先把这 3 类问题明确标记为：

- **P0 必补缺口**
- 不是“可选优化”
- 也不是“先记到 P1 再说”

如果业务确认它们仍属于 P0，则后续所有动作都围绕：

> **把 P0 草案中的承诺，补成真实通过。**

---

### Phase 2：先做 `$ralplan`，不直接编码

在 OMX 里，最佳实践是先跑一次：

> **`$ralplan` / 共识计划**

目标不是重写整个 PRD，而是生成一个：

- **P0 缺口补齐计划**
- **验收增量计划**

这个计划至少回答 4 件事：

1. 这 3 类缺口分别归属哪些 AC / TC  
2. 哪些是实现缺口，哪些是证据缺口  
3. 要拆成几个 slice  
4. 每个 slice 的完成定义是什么

---

### Phase 3：把缺口拆成独立 slice

不要把 3 类问题混成一个“大修复”。

最佳实践是拆成最少 4 个 slice：

#### Slice R1：商品选择入口去 Select 心智

目标：

- 首屏与编辑链路都不再把普通 Select 当主入口
- 改成“入口按钮 / 已选摘要 / 展开查看 / 更换商品”

验收重点：

- AC-11
- AC-12
- AC-13

#### Slice R2：默认值与用户控制权闭环

目标：

- 封面：首次默认采用 + 手改后脱钩
- 视频：首次默认全选 + 后续不静默重置

验收重点：

- AC-18
- AC-19
- TC-03
- TC-04

#### Slice R3：版本边界动作闭环

目标：

- 回到编辑不建版本
- 确认结果不建版本
- 发布不建版本
- 重新生成才建版本

验收重点：

- AC-23 ~ AC-26
- TC-06 ~ TC-08

#### Slice R4：总验收证明补齐

目标：

- 把前 3 个 slice 的通过证据汇总进新的总验收文档
- 不再写“部分通过”，而是给出最终口径

---

## 3. 每个 slice 在 OMX 里的标准动作

每个 slice 都按同一模板走，不要跳步。

### Step A：先写 slice plan

落盘 1 份文档：

- `...plan-v1-日期.md`

内容只写：

1. 本 slice 解决什么  
2. 对应哪些 AC / TC  
3. 改哪些文件  
4. 不改什么  
5. 怎么验证

---

### Step B：先补测试，再改实现

这类问题不是纯视觉问题，而是：

- 交互规则
- 状态机规则
- 版本边界规则

所以最佳实践是：

> **先补失败测试或至少先定义可执行测试，再改代码。**

测试层建议：

1. **E2E**：用户主链路  
2. **view-model / hook**：默认值、脱钩、版本边界  
3. **组件交互**：区内入口与操作

---

### Step C：再做实现

实现阶段建议角色分工：

- **`architect`**：状态机 / 版本边界 / 投影契约
- **`executor`**：具体前后端改造
- **`test-engineer`**：自动化测试补齐

如果 3 个 slice 相互独立，可用：

> **`$team`**

如果当前只有一个主执行者连续推进，则用：

> **`$ralph`**

选择原则：

- **有多个独立写入面**：`$team`
- **一个人连续收尾更稳**：`$ralph`

对当前 CreativeDetail，这一轮更推荐：

> **`$ralph` 主持 + 必要时局部委派 executor / test-engineer / verifier**

因为它更像“收尾闭环”，不是新功能大并行。

---

### Step D：做 closeout，而不是直接说完成

每个 slice 完成后必须再落盘 1 份：

- `...closeout-v1-日期.md`

closeout 要写清楚：

1. 改了哪些文件  
2. 对应哪些 AC / TC  
3. 测了什么  
4. 还剩什么风险

---

### Step E：独立验证

最佳实践不是“写代码的人自己说通过了”，而是：

> **让 `verifier` 或至少独立视角做一次复核。**

特别是版本边界这种问题，必须要求：

- 代码证据
- E2E 证据
- 文档口径一致

三者同时成立，才叫真正通过。

---

## 4. OMX 里的推荐角色分工

## 4.1 规划层

### `$ralplan`

负责：

- 把 3 类缺口变成明确 slice
- 把 AC / TC 映射补齐
- 给出执行顺序

### `architect`

负责：

- 状态机、页面模式、版本边界
- 判断哪些动作该改 creative draft，哪些该改 version

---

## 4.2 执行层

### `executor`

负责：

- 页面交互改造
- view-model / 状态改造
- 后端契约或接口改造

### `test-engineer`

负责：

- E2E 主链路
- hook / 状态单测
- 边界 case 补测

---

## 4.3 验收层

### `verifier`

负责：

- 对照 PRD / 覆盖矩阵 / AC / TC 复核
- 判断是“已通过”还是“只是看起来差不多”

---

## 5. 当前这 3 类问题的推荐执行顺序

顺序不要乱。

最佳实践顺序是：

### 先做 R3：版本边界

原因：

- 它决定页面模式与动作合法性
- 会影响 CTA、锁定态、重新生成、回到编辑

也就是先把：

> **什么动作会不会建版本**

这件事钉死。

---

### 再做 R2：默认值与脱钩

原因：

- 这是当前入选区真值逻辑的一部分
- 会影响商品区 / 自由素材区 / 入选区联动

先把“真值规则”做对，再调整入口。

---

### 最后做 R1：商品选择入口升级

原因：

- 它更偏交互承载层
- 应建立在前两者规则稳定之后

否则容易出现：

- 入口重做了一遍
- 里面的默认值与状态机规则还是错的

---

### 最后做 R4：总验收

只有前 3 个 slice 都过了，P0 总收口才有意义。

---

## 6. 什么叫“符合最佳实践地完成”

要同时满足下面 5 条，才算真的完成：

### 1. 规则完成

不是口头规则，而是代码里真的实现了。

### 2. 交互完成

不是文档画过，而是页面上真实可操作。

### 3. 自动化证明完成

至少有对应的：

- E2E
- 必要的状态 / hook 测试

### 4. 文档口径完成

PRD、覆盖矩阵、验收草案、closeout 口径一致。

### 5. 独立复核完成

不是作者自己觉得可以，而是 `verifier` 也能给出通过结论。

---

## 7. 对当前 CreativeDetail 的直接建议

如果严格按 OMX 最佳实践，我建议下一步不是再写泛总结，而是直接进入下面流程：

### 第一步

产出：

> 《CreativeDetail P0 缺口补齐计划 v1》

内容：

- R1 / R2 / R3 / R4 四个 slice
- 每个 slice 对应 AC / TC
- 每个 slice 的验证方案

### 第二步

按顺序执行：

1. R3 版本边界
2. R2 默认值与脱钩
3. R1 商品选择入口
4. R4 总验收

### 第三步

每个 slice 都做：

- plan
- 实现
- 测试
- closeout

### 第四步

最后再做：

> **P0-9 Revised 总验收**

那时如果还有“部分通过”，就说明 P0 还没封。

## 8. 一句话收口

一句话：

> **在 OMX 里，像你指出的这类“关键业务规则还没闭环”的情况，最佳实践不是继续写收口总结，而是把它重新纳入一个小型执行闭环：`$ralplan` 定缺口 -> slice 化实施 -> 自动化证明 -> `verifier` 独立验收。**

再压缩一句：

> **先补规则，再补交互，再补证据，最后才允许收口。**
