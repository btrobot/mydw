# 《取消审核后，CreativeDetail 何时才应该创建新版本 v1》

> 日期：2026-04-24  
> 适用范围：单用户 CreativeDetail / 版本机制 / 发布机制  
> 目的：回答“一个作品可以有多个版本、多次发布；在取消审核操作之后，什么时候才可以创建新的版本？”  
> 状态：Draft

---

## 1. 一句话结论

> **取消审核之后，“创建新版本”的时机不应该变成任意时刻。**  
> **新版本仍然应该只在“开始一轮新的结果生成”时创建。**

也就是：

- **保存草稿**：不创建版本
- **仅修改发布参数**：不创建版本
- **点击提交生成 / 重新生成**：创建新版本

所以真正的版本边界不是“审核通过”，而是：

> **内容冻结并进入新一轮生成**

---

## 2. 先把 3 个动作分开

这个问题最容易混乱的原因，是下面 3 件事常被混成一件事：

### 2.1 编辑作品

用户在改：

- 商品定义
- 文案
- 封面
- 输入素材
- 时长

这是在改 **Creative 草稿定义**。  
这一步**不应该直接创建版本**。

---

### 2.2 生成结果

用户点击：

- 提交生成
- 重新生成

这意味着：

> “我认可当前定义已经足够稳定，值得产出一版结果”

这一步才应该形成：

- `CreativeVersion Vn`

---

### 2.3 发布结果

用户拿某个 version 去发：

- 一次发布
- 多次发布
- 不同渠道发布
- 不同时间再发

这一步通常应该生成的是：

- publish record / release record / package record

而**不是**新的 creative version。

---

## 3. 最核心原则：版本记录“内容变化”，不是记录“每次操作”

最佳实践里，`Version` 只应该在下面这种场景创建：

> **内容定义发生了新的冻结，并准备生成一版新的结果**

而不应该在下面这些场景创建：

- 普通保存
- 页面自动保存
- 打开详情页
- 看结果
- 单纯确认结果
- 单纯再次发布同一结果

一句话：

> **版本是内容迭代单位，不是操作流水单位。**

---

## 4. 取消审核之后，推荐的版本创建规则

## 4.1 规则一：草稿编辑阶段永远不建版本

适用状态：

- `DRAFT`
- `READY`

在这个阶段，用户可以反复改定义。  
无论改多少次，都还是同一个“当前草稿头部”。

### 原因

如果每次保存都建版本，会带来 3 个问题：

1. 版本爆炸
2. 用户无法理解哪个版本是真正结果版
3. 审核/发布会失去明确边界

所以：

> **编辑不建版本，只维护当前作品定义。**

---

## 4.2 规则二：点击“提交生成 / 重新生成”时才创建新版本

这是推荐的唯一主入口。

即：

```text
当前草稿定义稳定
  -> 用户点击提交生成
  -> 系统冻结输入
  -> 创建新版本 Vn
  -> 进入 GENERATING
```

这时版本代表的是：

> **“基于这一刻的定义，发起的一次结果产出”**

---

## 4.3 规则三：取消审核，不取消“结果确认前的稳定窗口”

就算取消审核动作，也不代表用户可以在结果刚出来后立刻对当前版本继续乱改。

最佳实践仍然是：

- 结果出来后，当前版本是稳定结果
- 如果用户不满意，可以“回到编辑”
- 但回到编辑改的是 **Creative 草稿头**
- 下一次点击生成时，才形成 **V+1**

所以：

> **“回到编辑”不等于“立即创建新版本”**  
> **“再次提交生成”才等于“创建新版本”**

---

## 5. 一个作品多版本、多次发布时，正确关系是什么

## 5.1 一个作品可以有多个版本

例如：

- V1：首轮生成结果
- V2：优化文案后重新生成
- V3：换封面和视频组合后再生成

这些都属于：

> **内容有变化，因此应创建新版本**

---

## 5.2 一个版本也可以被多次发布

例如：

- 同一版结果先手动发布一次
- 过两天重新发一次
- 换一个渠道再发一次
- 换一个账号再发一次

如果发布的还是同一份内容结果，那么：

> **不应该创建新 version**

因为内容没变，只是发布动作变了。

---

## 5.3 所以版本和发布不是 1:1

正确理解应该是：

```text
Creative 1
  -> Version 1
      -> Publish A
      -> Publish B
  -> Version 2
      -> Publish C
  -> Version 3
      -> Publish D
      -> Publish E
```

也就是说：

- **Version 是内容结果轴**
- **Publish 是分发执行轴**

---

## 6. 那么“什么时候才可以创建新的版本？”

如果按单用户简化模型，答案是：

## 6.1 可以“开始新一轮版本编辑”的时机

可以从这些稳定态开始新一轮：

- `RESULT_PENDING_CONFIRM`
- `RESULT_CONFIRMED`
- `PUBLISHED`
- `FAILED`
- 当然也包括还没出结果前的 `DRAFT / READY`

但要注意：

> **这里说的是“可以开始下一轮内容修改”**  
> **不是说系统此刻立刻生成 version 记录**

---

## 6.2 真正“落库创建新 version”的时机

推荐只在以下动作发生时创建：

- `提交生成`
- `重新生成`

也就是：

| 动作 | 是否创建 version |
|---|---|
| 保存草稿 | 否 |
| 回到编辑 | 否 |
| 撤回修改 | 否 |
| 确认结果 | 否 |
| 去发布 | 否 |
| 再次发布同一结果 | 否 |
| 提交生成 | 是 |
| 重新生成 | 是 |

---

## 7. 哪些状态下不应该允许创建新版本

## 7.1 生成中

状态：

- `GENERATING`

不建议允许直接再建新版本。

### 原因

如果当前版本还在跑，又立刻创建下一版本，会让用户搞不清：

- 当前结果对应哪个定义
- 哪个版本正在跑
- 当前页到底在看哪一版

所以最佳实践是：

> **生成中禁止直接创建新 version**

如果一定要支持高级能力，也建议做成：

- “保存为待编辑草稿”
- 但**不分配新 version_no**
- 直到用户下一次真正提交生成时，才创建新版本

---

## 7.2 发布中

状态：

- `PUBLISHING`

通常也不建议直接同时创建新版本。

### 原因

发布中的核心目标是：

- 保证当前发布执行闭环

如果这时又立刻开新版本，会把“当前发的是哪一版”搞复杂。

更稳妥的是：

- 允许用户回到作品继续编辑草稿
- 但**不要立即创建新 version**
- 等他真正提交新一轮生成时再创建

---

## 8. 最推荐的单用户规则

我给你一个最实用的版本创建规则：

## 规则 A：只有“生成动作”能创建新版本

即：

- 提交生成
- 重新生成

---

## 规则 B：只有在“非运行中状态”才能发起新一轮版本

推荐允许发起新一轮的状态：

- `DRAFT`
- `READY`
- `RESULT_PENDING_CONFIRM`
- `RESULT_CONFIRMED`
- `PUBLISHED`
- `FAILED`

不推荐：

- `GENERATING`
- `PUBLISHING`

---

## 规则 C：同一版本可多次发布，但内容不变就不升版本

这是非常关键的业务边界。

### 需要新版本的情况

- 改了文案
- 改了封面
- 改了输入素材
- 改了商品表达
- 改了时长目标
- 重新生成想拿新结果

### 不需要新版本的情况

- 同一结果重复发布
- 改发布时间
- 改发布账号
- 改发布渠道
- 重新入池但内容没变

---

## 9. 如果取消审核，推荐状态流怎么配合版本机制

推荐简化后仍这样工作：

```text
DRAFT
  -> READY
  -> 提交生成
  -> 创建 Version N
  -> GENERATING
  -> RESULT_PENDING_CONFIRM
```

然后：

### 路径 1：接受结果

```text
RESULT_PENDING_CONFIRM
  -> RESULT_CONFIRMED
  -> PUBLISHED
```

这里不产生新 version。

---

### 路径 2：不接受结果，回去修改

```text
RESULT_PENDING_CONFIRM
  -> 回到编辑
  -> DRAFT / READY
  -> 再次提交生成
  -> 创建 Version N+1
```

新版本是在**再次提交生成**时创建，不是在“回到编辑”时创建。

---

### 路径 3：不改定义，直接再生成一次

```text
RESULT_PENDING_CONFIRM
  -> 重新生成
  -> 创建 Version N+1
```

因为虽然定义没改，但你希望产出一版新的结果，这依然应当是新 version。

---

## 10. 页面交互上应该怎么表达

为了不让用户混乱，页面上最好把两个层次说清楚：

## 10.1 “继续编辑”

语义：

- 回去修改作品定义
- 还没有形成新版本

按钮文案建议：

- `回到编辑`
- `继续修改`

不要写：

- `创建新版本`

因为这一步还没真的创建 version。

---

## 10.2 “生成新版本”

语义：

- 以当前定义重新产出一版结果

按钮文案建议：

- `提交生成`
- `重新生成新版本`

这一步才是真正的 version creation。

---

## 11. 最终建议

如果你问我：

> 取消审核后，什么时候才可以创建新的版本？

我给你的最佳实践答案是：

### 11.1 业务上

只有当用户决定：

> **“我要基于当前定义再产出一版新结果”**

时，才应该创建新版本。

---

### 11.2 系统上

真正的创建时机应绑定到：

- `submit-composition`
- 或等价的“重新生成”动作

---

### 11.3 页面上

应区分：

- `回到编辑`（不建版本）
- `重新生成新版本`（建版本）

---

## 12. 最终结论

> **取消审核，不等于随时建版本。**

对于一个可以有多个版本、也可以多次发布的作品，最稳的规则是：

> **只有当内容进入新一轮生成时，才创建新的版本。**

因此：

- 编辑阶段：不建版本
- 结果确认阶段：可回编辑，但不建版本
- 发布阶段：同一结果可多次发布，但不建版本
- 只有“提交生成 / 重新生成”时：才创建 `V+1`

这套规则最清晰，也最不容易把“内容版本”和“发布动作”混在一起。
