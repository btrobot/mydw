# CreativeDetail 页面规范样板 v1

> 版本：v1  
> 页面路由：`/creative/:id`  
> 页面类型：编辑定义页  
> 目标用户：创作运营 / 内容编辑 / 审核前置整理人员  
> 关联页面：`/creative/workbench` → `CreativeDetail` → `AIClip / 审核 / 执行任务`  
> 状态：Draft

---

## 1. 页面一句话定位

> 这是一个用于**定义新作品**的编辑定义页，用户在这里明确作品的商品上下文、作品当前真值、候选池与当前入选媒体，并在准备完成后提交进入后续合成与审核流程。

---

## 2. 页面职责

## 2.1 负责什么

- 定义作品基础信息与业务 brief
- 建立作品关联商品与主题商品
- 明确作品当前真值：当前商品名、当前文案、当前封面
- 管理候选池与当前入选媒体
- 在满足条件时提交合成，进入版本结果阶段

## 2.2 不负责什么

- 不默认承担执行任务排障主界面职责
- 不默认承担发布侧候选、cutover 对账、shadow diff 的首屏展示职责
- 不让历史版本、发布承接和高级诊断压过“定义作品”主流程

> 说明：版本、审核、发布、诊断仍然存在，但必须服从“定义作品”这一主职责。

---

## 3. 用户核心任务

## 3.1 用户来到页面最想完成的事

1. 定义一个新的作品
2. 补齐这个作品的商品上下文与创作真值
3. 选定当前真正参与合成的媒体
4. 在准备完成后提交合成

## 3.2 30 秒内必须完成的核心动作

> 用户进入页面后，30 秒内最应该完成或看清楚的动作：

- 看清“这个页面的第一任务是定义作品”
- 看清当前作品还缺哪些关键输入
- 进入主创作区开始补齐商品、文案、候选和入选媒体

## 3.3 成功定义

当用户离开本页时，理想结果是：

- 作品的业务定义已经明确且可保存
- 用户知道作品是否已经达到“可提交合成”的状态

## 5. 首屏原则

## 5.1 首屏必须回答的问题

用户只看首屏时，必须立刻知道：

1. 这是什么作品 / 当前在定义哪条作品
2. 这个作品是否已具备提交合成的条件
3. 下一步应该先补什么，而不是先去看诊断

## 5.2 首屏必须出现的内容

- 页面标题与返回路径
- 轻量业务概览：作品编号、状态、合成准备状态、最近更新时间
- 主创作区入口：作品 brief、商品上下文、候选池、当前入选媒体
- 主 CTA：保存创作定义 / 提交合成

## 5.3 首屏绝不能抢占主语义的内容

- scheduler mode / shadow diff / kill switch 等工程诊断信息
- 发布池候选、package 冻结值、cutover 对账原始证据
- 需要通过长段说明文案才能理解的工程语义

## 6. 信息架构

## 6.1 页面区块（从上到下）

| 序号 | 区块名称 | 层级 | 作用 | 默认展示 | 备注 |
|------|----------|------|------|----------|------|
| 1 | 页头 | 主 | 标识作品、提供返回与主操作 | 是 | `PageContainer` |
| 2 | 业务概览 | 主 | 让用户快速理解作品当前状态 | 是 | 轻量，不喧宾夺主 |
| 3 | 主创作区 | 主 | 定义作品 | 是 | 页面核心 |
| 4 | 当前版本 / 审核摘要 | 次 | 辅助决策，不压过主创作区 | 是 | 已有版本时显示 |
| 5 | 版本时间线 | 次 | 回看历史版本 | 是 | 放在主创作区之后 |
| 6 | 高级诊断层 | 高级 | 承接任务、发布、对账、raw diagnostics | 否 | Drawer |
| 7 | 审核抽屉 | 高级 | 审核当前版本 | 否 | Drawer |
| 8 | AIClip 抽屉 | 高级 | 进入后续创作/合成工作流 | 否 | Drawer |

## 6.2 默认业务层 vs 高级层

### 默认业务层展示

- 作品业务概览
- 创作 brief
- 商品上下文（关联商品 / 主题商品）
- 当前商品名、当前文案、当前封面
- 候选池
- 当前入选媒体
- 当前版本摘要与审核结论（若存在）

### 高级层展示

- 执行任务诊断
- 发布侧能力与调度诊断
- 发布池候选记录
- Cutover 对账
- shadow diff 原始证据

### 高级层入口

- 入口形式：页头按钮 + Drawer
- 入口文案：`查看高级诊断`

## 7. 主次操作设计

## 7.1 主 CTA

| 操作 | 文案 | 位置 | 触发条件 | 成功反馈 |
|------|------|------|----------|----------|
| 保存创作定义 | 保存创作定义 | 主创作区右上 | 表单可保存 | 提示保存成功并刷新数据 |
| 提交合成 | 提交合成 / 提交作品合成 | 主创作区右上 | `eligibility_status === READY_TO_COMPOSE` | 提示已提交并进入后续状态 |

## 7.2 次 CTA

| 操作 | 文案 | 位置 | 触发条件 | 说明 |
|------|------|------|----------|------|
| 进入 AIClip | 进入 AIClip | 页头 | 已有当前版本 | 属于后续流程，不应抢主创作区位置 |
| 审核当前版本 | 审核当前版本 | 页头 | 已有当前版本 | 是次级决策动作 |
| 查看高级诊断 | 查看高级诊断 | 页头 | 始终可见 | 进入高级层，不属于主定义流 |

## 7.3 危险操作

| 操作 | 风险 | 是否二次确认 | 文案策略 |
|------|------|--------------|----------|
| 移除候选项 / 非主商品关联 | 影响作品定义 | 否（轻量可撤销更优） | 用“移除”而非“删除作品” |
| 移除主商品 | 破坏主上下文 | 是（若支持） | 强提示会影响商品名/封面默认来源 |

## 8. 页面状态

| 状态 | 触发条件 | UI 表现 | 用户可执行动作 |
|------|----------|---------|----------------|
| loading | 初次加载且无 creative 数据 | 全页居中 Spin | 等待 |
| error | 详情请求失败 | Result 错误页 | 重试加载 / 返回工作台 |
| empty | creative 不存在或无权限 | Empty | 返回 |
| normal | creative 数据已加载 | 正常页面结构 | 编辑 / 保存 / 提交 |
| submitting | 保存或提交中 | 按钮 loading / 禁用 | 等待 |
| partial-error | 诊断请求失败但主体成功 | 仅诊断区提示 warning | 继续定义作品 / 稍后重试诊断 |

### 状态原则

- 主体定义区成功加载时，不应因诊断失败而整体报错
- 诊断问题应局部化，不得中断主定义流程
- 空态和错误态必须给出返回路径

## 9. 交互流程

## 9.1 主流程

```text
用户从 Workbench 进入作品详情
→ 首屏看到作品状态与主创作区
→ 编辑标题 / 商品上下文 / 当前真值 / 候选池 / 当前入选媒体
→ 点击“保存创作定义”
→ 页面保存并刷新
→ 当状态满足 READY_TO_COMPOSE
→ 点击“提交合成”
→ 生成执行任务或进入下一阶段
```

## 9.2 次流程

```text
用户在已有当前版本时
→ 查看版本结果摘要
→ 打开审核抽屉
→ 或进入 AIClip
→ 或查看版本时间线
```

## 9.3 异常流程

```text
用户保存或提交
→ 表单校验失败 / API 失败
→ 显示对应错误
→ 保留当前上下文
→ 用户继续修正后重试
```

## 10. 数据与 API

## 10.1 页面初始化数据

| 数据 | 来源 | 必需 | 失败影响 |
|------|------|------|----------|
| creative 详情 | `useCreative(creativeId)` | 是 | 页面无法进入正常态 |
| 素材池：商品/视频/文案/封面/音频 | `useProducts / useVideos / useCopywritings / useCovers / useAudios` | 是 | 主创作区部分能力不可用 |
| profile 列表 | `useProfiles` | 否（但重要） | 无法完整选择合成配置 |
| 版本/审核/发布诊断衍生数据 | 页面 view-model | 否（对主定义流） | 仅影响次级决策与高级层 |

## 10.2 操作 API

| 前端动作 | HTTP 方法 | API | 请求参数 | 成功后刷新什么 |
|----------|-----------|-----|----------|----------------|
| 保存创作定义 | 更新 | `useUpdateCreative` | `buildCreativeAuthoringPayload(values)` | creative + diagnostics |
| 提交合成 | 提交 | `useSubmitCreativeComposition` | 当前 creative id | creative + diagnostics |
| 重新拉取详情 | 查询 | `creativeQuery.refetch()` | creative id | creative |
| 重试诊断 | 查询 | `retryDiagnostics / refetchDiagnostics` | creative id | diagnostics |

## 10.3 页面间通信

| 方向 | 方式 | 数据 |
|------|------|------|
| 进入本页 | route params / navigation state | `creativeId`、`returnTo`、可能的任务优先上下文 |
| 离开本页 | 返回路径恢复 | workbench query state / returnTo |

## 11. 组件组合

| 区块 | 组件 | 类型 | 说明 |
|------|------|------|------|
| 页头外壳 | `PageContainer` | 页面级 | 页面容器 |
| 业务概览 | `ProDescriptions` + `Card` | 页面级 | 轻量摘要 |
| 主创作区 | `Form` + 多个 `Card` / `Form.List` | 页面级 | 页面核心 |
| 版本历史 | `VersionPanel` | 业务组件 | 历史版本回看 |
| 审核 | `CheckDrawer` | 业务组件 | 版本审核 |
| AIClip | `AIClipWorkflowPanel` | 业务组件 | 后续处理 |
| 高级诊断 | `DiagnosticsActionPanel` + `Drawer` | 业务组件 | 下沉的高级层 |

### 复用要求

- 候选池区块、当前入选媒体区块、业务概览区块未来可进一步抽组件
- 主创作区的状态源应保持在页面级 view-model，不要下沉到纯展示组件
- 高级诊断组件不得反向主导页面首屏结构

## 12. 文案与反馈原则

## 12.1 语言策略

- 主语言：中文
- 保留英文术语范围：AIClip、必要技术名词
- 禁止出现：工程占位词、兼容 carrier 细节直接暴露给普通用户

## 12.2 提示策略

- 成功提示：短句，告诉用户保存/提交结果
- 失败提示：说明失败动作，不泛化成系统故障
- 空态提示：说明缺什么、下一步怎么补
- 权限提示：明确“记录不存在或无权限”

---

## 13. 视觉与布局原则

- 页面宽度策略：主创作区优先占据视觉中心
- 窄窗口策略：主区块纵向堆叠，不用固定多列思维压缩
- 固定底栏：当前可不强制，但若动作继续增多可考虑
- Drawer 宽度策略：高级诊断和 AIClip 允许宽 Drawer
- 主区块与次区块视觉优先级：主创作区 > 版本/审核摘要 > 版本历史 > 高级诊断

## 14. 边界与风险

## 14.1 已知边界

- 当前页面仍然共存“定义作品”和“版本/审核/发布承接”信息
- 当前选中媒体仍依赖兼容 carrier 投影

## 14.2 已知风险

- 如果继续往首屏叠加发布与诊断信息，页面会重新滑回“总控页”
- 如果新建态与成熟态不分层，定义作品的主路径会被稀释

## 14.3 本页禁止混入

- 默认展示 raw shadow diff JSON
- 让任务诊断成为首屏核心任务
- 用过长解释文案替代清晰的区块和层级
