# CreativeDetail A+B 交互规范 v1

> 日期：2026-04-24  
> 适用页面：`/creative/:id` / `CreativeDetail`  
> 范围：A 当前定义区 + B 当前入选区  
> 前置文档：`discss/specs/creative-detail-business-flow-v1-2026-04-24.md`  
> 状态：Draft

---

## 1. 目标

本规范只解决 CreativeDetail 最核心的两个区块：

- **A 当前定义区**：这个作品现在被定义成什么
- **B 当前入选区**：这个作品现在准备拿什么去生成

这两个区块共同承担页面主任务：

> **让用户把作品定义清楚，并确认当前送去生成的输入集合。**

---

## 2. A+B 的页面地位

## 2.1 为什么先做 A+B

因为用户当前最缺的不是“更多信息”，而是：

- 什么是当前业务真值
- 什么是当前实际入选
- 两者之间的关系

如果 A+B 不清楚，页面就会继续退化成：

- 候选堆叠页
- 结果回看页
- 总控诊断页

## 2.2 A 与 B 的关系

- **A 当前定义区**回答：作品想做成什么
- **B 当前入选区**回答：这次实际拿什么去做

两者必须前后相邻，且语义连续。

---

## 3. A 当前定义区规范

## 3.1 一句话定位

> A 区不是“填写表单”，而是“确认作品当前真值”。

## 3.2 区块目标

用户在 A 区完成后，应能回答：

1. 这个作品围绕哪个商品
2. 当前外显商品名是什么
3. 当前主文案是什么
4. 当前封面是什么
5. 目标时长是什么

## 3.3 信息结构

建议拆成 4 个子卡片，按顺序排列：

### A1 作品基础信息

字段：

- 作品标题
- Profile
- 作品状态（只读）

作用：

- 标识当前编辑对象
- 给出基础上下文

### A2 主题商品定义

字段：

- 主商品
- 关联商品列表
- 当前商品名快照

作用：

- 明确作品围绕谁展开
- 区分“主商品”与“可关联商品”

交互要求：

- 主商品必须有明显主选态
- 切换主商品后，相关快照字段要明确提示会受影响

### A3 当前表达定义

字段：

- 当前主文案
- 当前封面
- 目标时长

作用：

- 明确作品当前的表达真值

交互要求：

- 当前主文案与候选文案要区分“已采用 / 仅可选”
- 当前封面需要有明确的当前态预览
- 目标时长必须使用业务语言，不要写成技术参数配置

### A4 生成准备状态

字段 / 信息：

- 是否可提交生成
- 缺失项列表
- 每个缺失项的跳转锚点

作用：

- 把 readiness 从抽象状态变成可操作清单

交互要求：

- 缺什么就写什么
- 每条缺失项可定位到 A / B / C 对应区域

## 3.4 布局要求

- A 区整体应位于首屏上半区
- 信息优先纵向展开，不要为了压缩高度强行多列
- A2 / A3 是视觉重点，A1 次之，A4 应靠近主 CTA

## 3.5 主操作

主操作：

- 保存作品定义
- 提交生成

辅助操作：

- 恢复最近一次保存
- 查看影响说明（例如切换主商品会影响什么）

## 3.6 状态要求

- 未完成：展示缺失项，不阻断继续编辑
- 可提交：明确显示“已具备生成条件”
- 保存中：局部 loading，不锁死全页
- 保存失败：错误就地显示，不清空用户输入

---

## 4. B 当前入选区规范

## 4.1 一句话定位

> B 区不是“展示候选结果”，而是“确认这次真正参与生成的输入集合”。

## 4.2 区块目标

用户在 B 区完成后，应能回答：

1. 这次到底选了哪些输入
2. 哪些输入正在启用
3. 输入之间有没有顺序 / 主次差异
4. 如果现在提交生成，送出的就是不是这一组内容

## 4.3 信息结构

建议拆成 3 个子卡片：

### B1 当前入选总览

展示：

- 已入选视频数
- 已入选音频数
- 当前主文案
- 当前封面
- 当前主商品

作用：

- 让用户 5 秒内看清当前送出的输入组合

### B2 入选内容清单

按类型分组展示：

- 视频
- 音频
- 文案
- 封面
- 商品关系

每个入选项至少展示：

- 名称 / 摘要
- 来源
- 当前状态（启用 / 停用）
- 角色标签（主素材 / 辅助素材 / 当前文案 / 当前封面）

交互要求：

- 同类内容允许排序时，要直观表达顺序
- 若暂不支持排序，也要明确告诉用户“当前顺序规则”
- 停用和移除要区分

### B3 入选调整动作

动作：

- 从候选池加入
- 从当前入选移除
- 设为主文案
- 设为当前封面
- 设为主商品 / 主关联

作用：

- 让 B 区成为“定稿区”，而不是只读区

## 4.4 B 区的关键交互语义

必须清楚区分四个概念：

1. 候选
2. 已入选
3. 当前采用
4. 已停用但保留记录

推荐状态文案：

- 候选中
- 已入选
- 当前采用
- 已停用

禁止使用让用户分不清层级的词，例如：

- 已关联
- 已配置
- 已存在

这些词太泛，不能表达当前作品语义。

## 4.5 布局要求

- B 区紧跟 A 区之后
- 先总览，后明细，最后动作
- 入选项卡片要明显强于候选项视觉层级
- 当前采用项要有高亮，不要只靠文本描述

## 4.6 状态要求

- 空状态：明确提示“当前还没有可参与生成的输入”
- 部分完成：告诉用户已有多少项入选，还缺哪些关键项
- 已完成：展示“当前可用于生成”的确定性提示
- 变更中：就地反馈，不跳页

---

## 5. A+B 之间的联动规则

## 5.1 数据联动

- A 区主商品变化，会影响 B 区商品关系显示
- A 区当前主文案变化，应同步反映到 B 区“当前采用”标签
- A 区当前封面变化，应同步反映到 B 区封面入选状态
- B 区移除某个“当前采用”项时，A 区必须立即暴露缺失

## 5.2 视觉联动

- A 区定义的是“当前真值”
- B 区展示的是“真值在输入集合上的落点”

所以建议：

- A 区强调“当前”
- B 区强调“入选 + 当前采用”

## 5.3 CTA 联动

“提交生成”按钮的可用性应受 A+B 共同约束：

- A 未定义清楚，不能提交
- B 没有足够输入，不能提交
- 不满足条件时，按钮旁必须给出缺失原因入口

---

## 6. 首屏建议骨架

```text
页头
└─ 作品标题 / 返回 / 状态 / 主 CTA

轻量业务概览
└─ 编号 / 更新时间 / readiness / 当前版本状态

A 当前定义区
├─ A1 基础信息
├─ A2 主题商品定义
├─ A3 当前表达定义
└─ A4 生成准备状态

B 当前入选区
├─ B1 当前入选总览
├─ B2 入选内容清单
└─ B3 入选调整动作
```

---

## 7. 对实现层的直接约束

1. A 与 B 必须使用明确标题，不要混成一个“大表单”
2. B 区不能只复用候选池样式，否则“当前入选”会失焦
3. readiness 不能只显示枚举值，必须显示可操作缺失项
4. “当前文案 / 当前封面 / 主商品”必须有唯一当前态
5. 版本、审核、诊断不得插入 A 与 B 中间

---

## 8. 最小验收标准

满足以下条件，才算 A+B 交互骨架成立：

1. 用户进入页面后，能在 30 秒内看懂“当前定义”和“当前入选”的区别
2. 用户能直接看出当前主商品、当前文案、当前封面、目标时长
3. 用户能直接看出这次将参与生成的输入集合
4. 用户能直接看到还缺什么才能提交生成
5. 页面首屏不会被版本、发布、诊断信息抢走主语义

---

## 9. 下一步建议

在 A+B 落稳后，后续设计顺序建议为：

1. C 候选池区规范
2. A/B/C 的状态映射与表单模型
3. D 版本与审核区降级承接
4. E 发布与诊断区高级入口化
