# 远端 Day 3.1 实现记录（users revoke/restore + 刷新策略）

## 本次补齐内容

- 在 React 版 `/users` 页面接入：
  - `POST /admin/users/{id}/revoke`
  - `POST /admin/users/{id}/restore`
- 明确落地 action 成功后的刷新策略：
  - 先刷新当前 users 列表；
  - 再根据刷新后的结果决定保留当前选中用户、切换到列表首项，或清空 detail；
  - 若仍有选中项，则主动刷新 detail query，避免只刷列表不刷详情。

## 具体实现

### 1. auth client

- `remote/remote-admin/src/features/auth/auth-client.ts`
  - 新增 `AdminActionResponse`
  - 新增 `revokeAdminUser()`
  - 新增 `restoreAdminUser()`

### 2. users 页面

- `remote/remote-admin/src/pages/users/UsersPage.tsx`
  - 新增 revoke / restore mutation
  - 新增 action 成功 / 失败反馈
  - 新增 `refreshUsersState()`：
    - refresh 当前列表
    - 依据刷新后的结果重新确定 `selectedUserId`
    - 对新的选中项主动拉取 detail
  - 对 mutation 后续 refresh 失败做兜底提示
  - 对 `token_expired` 继续统一走登出 + 回登录页

## 这一步为什么重要

Day 3 只是把 users 页从占位页切成“可浏览”页面；  
Day 3.1 才真正把它推进到“可操作、可回读、状态一致”的最小完整切片。

如果只补 action，不补刷新策略，会出现这些问题：

- 列表状态更新了，但 detail 还是旧的；
- 当前筛选条件下用户被移出结果集后，右侧还停留旧对象；
- 操作成功后用户不知道页面上的数据是不是最新。

所以这次的重点不是“按钮接上了”，而是“按钮接上后页面状态还能自洽”。
