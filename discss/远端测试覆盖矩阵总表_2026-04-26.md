## 六、当前覆盖结论

### 已经比较扎实的部分

1. **关键 auth / destructive contract**
2. **remote-admin 主页面迁移结果**
3. **admin step-up 主链路**
4. **代表性 destructive action 的浏览器交互**

### 目前仍是“代表性覆盖”的部分

1. users restore 的 failure / token_expired
2. devices unbind / rebind 的 failure / token_expired
3. 更细粒度的 UI 文案、布局、视觉一致性回归

这不意味着没测，而是说明这些分支当前主要依赖：

- 已有共享 step-up 主链路覆盖
- 差异动作专属 endpoint / payload / 状态断言覆盖

## 七、推荐维护原则

后续新增管理端 destructive 动作时，至少应补三层中的两层：

### 最低要求

1. **contract/helper 级**
   - header
   - query/body
   - 错误映射

2. **浏览器级**
   - modal 是否打开
   - verify 是否成功接线
   - destructive 请求是否带 token
   - 页面是否刷新到预期状态

### 如果动作有独特 payload 或独特状态变化

则必须额外补：

- 差异 payload 断言
- 差异状态断言

---

## 八、一句话总结

当前测试矩阵已经能支撑这样的判断：

> 远端主线不是“只有功能可用”，而是已经具备后端 contract、前端 helper、页面迁移和关键 destructive 浏览器链路四层护栏。  
> 后续工作重点应放在覆盖说明透明化和少量差异分支补强，而不是重新补主干。
