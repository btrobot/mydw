# 远端 Day 4.5 实现记录（`/logout` 与 `/me` 的最小 route contract 回归入口）

## 本次 slice 目标

- 在 Day 4.4 已经为 `/refresh` 补齐最小 route contract 回归入口后，继续把 auth 主链路补完整。
- 为以下两个接口增加最小但稳定的 route-level 回归测试：
  - `POST /logout`
  - `GET /me`

本次仍坚持最小风险原则：
- **不改运行时逻辑**
- **不改前端合同**
- 只把当前真实存在的 route 行为固化成自动化测试

### 2. 固定 `/logout` 的最小合同

本次为 `POST /logout` 固定了：

#### 成功路径
- 返回 `200`
- 返回体固定为：

```json
{ "success": true }
```

- route 会把：
  - `refresh_token`
  - `device_id`

原样转发给 `service.logout`

#### 错误语义
- `401 token_expired`
- `403 revoked`

也就是说，现在 `/logout` 至少已经有一组能覆盖：
- 成功
- token 失效
- 已撤销

的最小 API 回归入口。

### 3. 固定 `/me` 的最小合同

本次为 `GET /me` 固定了：

#### 成功路径
- 返回 `200`
- 返回当前 `MeResponse` payload shape
- bearer token 会从：

```http
Authorization: Bearer access_token
```

中正确提取，并转发给 `service.me`

#### 错误语义
- `401 token_expired`
- `403 revoked`

并额外固定：
- 当请求没有 bearer token 时，route 会向 service 传入空字符串 `""`
- 由 service 决定返回 `401`

这与此前 admin destructive / admin read 的 contract 处理方式保持一致：  
**route 负责提取与转发，service 负责最终语义判定。**

## 为什么这是“最小风险 Day 4.5”

1. **零运行时改动**
   - 没改 `app/api/auth.py`
   - 没改 `AuthService`
   - 没改任何限流或鉴权逻辑

2. **把 auth 主链路补成连续回归面**
   - Day 4.2：`/login`
   - Day 4.4：`/refresh`
   - Day 4.5：`/logout` + `/me`

3. **只补 route contract，不扩大到 service 内部实现**
   - 不混入数据库行为
   - 不混入 token 生命周期细节
   - 保持 slice 可回滚、可审查
