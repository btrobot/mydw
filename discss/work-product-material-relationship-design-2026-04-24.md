# 作品 / 商品 / 素材 关系设计分析（基于“四件套输出”反推）

## 1. 背景：先看我们最终到底要产出什么

当前我们真正要交付给外部的，其实不是“商品记录”本身，也不是“素材集合”本身，而是一套可发布结果：

1. 封面
2. 视频（可能由多个视频片段 + 音频编排而成）
3. 商品名称
4. 文案

所以，领域模型不应该从“系统里有哪些表/有哪些素材类型”出发，而应该从“谁对这四件套负责”出发。

我对这个问题的核心判断是：

- **商品不是作品**
- **素材不是作品**
- **作品才是对四件套最终负责的业务对象**

商品、素材，都是作品的上游输入或依赖，不应该和作品处于同一语义层级。

---

## 2. 从“四件套输出”反推：创建作品时，必须定义哪些东西

如果把作品理解为“一个待生成 / 待审核 / 待发布的成片单元”，那么在“定义作品”时，至少要明确下面几类真值：

### 2.1 主体商品是谁

作品必须知道自己在讲哪一个商品。

这意味着作品层至少需要：

- 关联的主体商品 `subject product`
- 商品名称快照 `product name snapshot`

这里建议：

- **作品关联商品 ID**
- 同时**在作品上冻结一份商品名称快照**

原因是：

- 后续商品库里的名称可能被修正、重抓、重新解析
- 但某个作品在某次创作里到底想展示什么名称，应由作品自己负责

换句话说：

- 商品库保存“来源事实”
- 作品保存“本次创作采用的展示真值”

### 2.2 视频输出目标是什么

最终视频不是“素材列表”本身，而是一次编排结果。

所以作品创建时，至少应定义：

- 目标时长
- 选用哪些视频素材作为候选输入
- 是否需要音频
- 是否允许后续再增补素材

这里不一定要求一开始就把所有镜头都编排完，但至少要确定：

- 这支作品的视频来源主要来自哪里
- 这支作品要做成什么样的成片目标

### 2.3 封面策略是什么

封面是最终输出的一部分，不应只是“某个 cover 素材碰巧被选中”。

建议作品层明确一个 **cover strategy**：

- 使用商品导入的默认封面
- 从素材库中手动选择某张封面
- 后续单独上传/维护

这样做的意义是：

- 用户创建作品时，不一定马上精修封面
- 但系统知道封面将如何得出
- 后续审核 / 发布时也能清楚追溯封面来源

### 2.4 文案策略是什么

文案不是普通素材，它和“商品名称”一样，属于最终输出真值。

所以作品创建时，应至少确定：

- 文案来源策略
  - 直接使用已有主文案
  - 基于商品介绍 / 话题调用大模型生成
  - 先生成草稿，人工再编辑确认
- 当前作品的主文案是什么

因此，**文案必须在作品层有一个明确字段/真值位**，而不应只存在于商品库或素材库里。

---

## 3. 商品 / 作品 / 素材 三者的推荐关系

## 3.1 商品：上游来源对象

商品更适合作为：

- 外部链接导入的来源对象
- 商品事实信息的归档对象
- 可复用素材的挂载对象

它应该保存：

- 来源链接（例如得物链接）
- 抓取到的商品标题
- 商品介绍原文
- 抓取到的话题
- 抓取到的视频 / 封面等原始资产

也就是说，**商品是 source package / source aggregate**。

它解决的是：

- 这个外部商品页是什么
- 我们从这个页上拿到了哪些源信息

但它不解决：

- 本次作品最终叫什么名字
- 本次作品最终用哪段文案
- 本次作品最终封面/视频怎么组合

这些都应该由作品负责。

## 3.2 素材：可复用资产对象

素材更适合作为：

- 视频素材
- 封面素材
- 音频素材
- 候选文案素材
- 话题 / 标签 / 提示参考

素材库解决的是“可复用性”和“可管理性”：

- 同一个商品导入的视频，可以被多个作品复用
- 某张封面可以被多个作品引用
- 某段音频不依赖具体商品

因此素材的职责是：

- 归档
- 复用
- 选择
- 编排输入

而不是承担最终业务真值。

## 3.3 作品：面向输出的业务对象

作品应被定义为：

> 围绕某个主体商品，组织视频 / 封面 / 文案等输入，并生成最终可审核可发布四件套的业务单元。

作品至少应承担：

- 主体商品是谁
- 最终展示商品名
- 当前主文案
- 当前封面选择
- 当前视频编排结果
- 当前版本/审核/发布状态

所以关系应该是：

- **商品 = 来源**
- **素材 = 资产**
- **作品 = 输出责任主体**

## 4. 得物链接导入，在新模型里的正确定位

你提到的关键难点非常准确：

- 为了简化用户输入，系统允许直接导入得物商品页链接
- 这个链接里已经能拿到：
  - 封面
  - 视频
  - 商品名称
  - 商品介绍（里面有话题）

这意味着，“导入商品链接”这个动作非常像“新建作品”。

但我建议语义上仍然要分清：

## 4.1 导入链接，不等于作品已经定义完成

导入链接本质上是：

1. 创建 / 更新一个商品来源对象
2. 解析并沉淀一批商品相关素材
3. 用这些信息去**初始化一个作品草稿**

所以更合理的交互语义是：

### 模式 A：只导入商品

- 用户输入得物链接
- 系统只创建商品来源记录
- 同时沉淀相关素材
- 不自动建作品

适合“先建库，再创作”。

### 模式 B：导入商品并创建作品草稿

- 用户输入得物链接
- 系统创建/复用商品来源对象
- 自动抽取封面/视频/标题/介绍/话题
- 自动生成一个“作品草稿”

草稿里预填：

- 主体商品
- 商品名称快照
- 默认封面候选
- 默认视频输入候选
- 话题参考
- 文案草稿生成入口，或直接生成初稿

这才是最符合你说的“为了简化用户输入”的方案。

结论是：

- **商品链接导入是一个 bootstrap 动作**
- **它可以触发作品草稿初始化**
- **但不能把“商品对象”直接等同于“作品对象”**

## 5. 文案与话题：到底应该算“素材”吗

这里是这次设计里最值得单独拿出来说的点。

## 5.1 文案不应只被当作素材

文案和视频/封面不一样。

视频/封面通常是“被选择的资产”；
但文案在业务上更接近“作品的表达结论”。

所以我建议区分两层：

### 候选文案

可以存在素材库里，作为可复用输入，例如：

- 商品页解析出的标题文本
- 商品介绍摘要
- 某次大模型生成的候选文案
- 人工沉淀的模板文案

这些都可以算“文案素材”。

### 主文案 / 最终文案

必须存在作品层 / 版本层 / 发布包层，作为输出真值。

也就是说：

- **文案可以有素材形态**
- **但最终必须在作品层被冻结为作品真值**

## 5.2 话题更像“参考信号”，不应继续与视频同权

话题和视频、音频、封面最大的区别在于：

- 它通常不是直接被拼进成片里的资产
- 它更像生成文案时的提示词
- 或者发布时的标签/辅助元数据

所以我建议把话题从“核心创作素材”里降级，重新定位为：

1. **文案生成输入**
2. **推荐标签 / 检索标签**
3. **发布元数据候选**

而不是继续与：

- 视频
- 音频
- 封面

并列成同等地位的创作素材。

一句话说：

- **文案是输出真值**
- **话题是生成文案的参考信号**

这两者不能再混在同一层里处理。

## 6. 推荐的领域分层

为了让系统后续更稳定，我建议把概念拆成下面几层。

## 6.1 ProductSource（商品来源）

职责：

- 保存外部商品链接
- 保存抓取到的原始商品事实
- 保存来源级元数据

字段可包括：

- source_url
- source_platform
- source_title
- source_description
- parsed_topics
- import_status
- last_synced_at

## 6.2 MaterialAsset（素材资产）

职责：

- 保存可复用资产

素材类型建议包括：

- video
- cover
- audio
- copy_candidate

对于话题，不建议继续硬塞进和视频并列的 asset 类型；更适合作为：

- product metadata
- work prompt inputs
- publish tags

## 6.3 CreativeWork（作品）

职责：

- 定义一次创作任务 / 一条输出责任链

字段建议包括：

- subject_product_source_id
- product_name_snapshot
- target_duration
- cover_strategy
- selected_cover_asset_id
- main_copy_strategy
- main_copy_text
- status

以及：

- 当前采用哪些视频/音频输入
- 当前作品是否已生成版本
- 当前是否进入审核/发布流程

## 6.4 CreativeVersion（作品版本）

职责：

- 保存每次实际生成结果

字段建议包括：

- rendered_video_path
- actual_duration
- final_cover_asset_id / final_cover_path
- final_product_name
- final_copy_text
- generation_params_snapshot

这样可以保证：

- 作品是长期对象
- 版本是阶段性产出

## 6.5 PublishPackage（发布包）

职责：

- 保存真正对外发布时冻结的四件套

字段建议包括：

- frozen_cover
- frozen_video
- frozen_product_name
- frozen_copy_text
- publish_channel_snapshot

发布包的意义是：

- 即使作品后续还会修改，某次已发布的内容仍然可回溯

## 7. 推荐交互设计

## 7.1 入口一：从工作台直接新建作品

适合已经有商品/素材沉淀的场景。

流程：

1. 点击“新建作品”
2. 先选主体商品
3. 系统带出该商品相关素材
4. 用户定义：
   - 商品显示名
   - 目标时长
   - 封面策略
   - 文案策略
5. 进入作品详情页继续编排视频/音频/文案

这个路径适合熟练用户。

## 7.2 入口二：粘贴得物链接，一步初始化作品

适合高频、低门槛场景。

流程建议：

### 第一步：输入链接

- 粘贴得物链接

### 第二步：系统解析预览

系统展示：

- 商品名称
- 抓到的封面
- 抓到的视频
- 抓到的话题
- 抓到的商品介绍摘要

### 第三步：选择初始化策略

给用户两个主选项：

- 仅导入商品
- 导入商品并创建作品草稿

如果选“创建作品草稿”，再补两个轻量选项：

- 默认封面是否采用商品封面
- 是否立即根据话题/介绍生成文案初稿

### 第四步：落到作品详情页

此时作品草稿已经带着：

- 主体商品
- 商品名快照
- 默认视频输入
- 默认封面候选
- 文案草稿/待生成状态

这样用户会觉得系统“很省输入”，但底层模型仍然清晰。

## 7.3 作品详情页的推荐结构

作品详情建议按“输出真值优先”组织，而不是按“素材类型堆栈”组织：

### 区块 1：输出定义

- 商品名称
- 主文案
- 封面
- 目标时长

### 区块 2：输入来源

- 主体商品
- 已选视频素材
- 已选音频素材
- 参考话题
- 来源介绍

### 区块 3：版本 / 生成 / 审核 / 发布

- 生成版本列表
- 当前诊断
- 审核状态
- 发布包预览

这样可以让用户始终知道：

- 上面看到的是“我要交付什么”
- 下面才是“我拿什么来生成它”

## 8. 推荐的管理维护方式

## 8.1 商品库管什么

商品库应该负责：

- 外部来源链接
- 商品事实信息
- 解析记录
- 来源级素材挂载

不应该让商品库承担：

- 作品最终文案
- 作品最终封面
- 作品最终名称

## 8.2 素材库管什么

素材库应该负责：

- 原始视频
- 原始封面
- 原始音频
- 候选文案
- 资产去重 / 复用 / 引用

素材库是资源层，不是业务决策层。

## 8.3 作品库管什么

作品库应该负责：

- 哪个商品被创作
- 当前作品四件套真值
- 当前版本
- 审核状态
- 发布状态

作品库是核心业务面。

---

## 9. 推荐的边界与避免事项

## 9.1 不要把“商品导入”直接做成“作品详情”

否则后面会出现很多混乱：

- 改商品信息会不会影响历史作品？
- 一个商品能不能做多个作品？
- 商品介绍和主文案谁是准的？

这些问题一旦混起来，后续维护成本会很高。

## 9.2 不要把“话题”继续和视频/音频并列成同权核心素材

因为它们的职责明显不同。

如果继续并列：

- UI 上会越来越别扭
- 用户会不清楚话题到底是“拿来剪”的，还是“拿来写”的
- 领域层会越来越难维护

## 9.3 不要把“主文案”只挂在商品层

因为商品介绍只是来源内容，不是作品表达结论。

作品可能：

- 同商品不同场景写不同文案
- 同商品同素材但不同投放口径写不同文案

所以文案必须在作品层成为第一公民。

## 9.4 不要把封面完全降为普通素材选择结果

封面虽然来源于素材，但它在输出上和商品名称/文案一样，属于显式交付项。

因此作品层应明确：

- 当前封面策略
- 当前选中的封面

而不是仅靠“素材列表里某个 item 恰好 role=cover”来间接表达。

---

## 10. 一个更顺手的心智模型

我建议后续产品和交互都围绕下面这句话来统一：

> **商品负责提供来源，素材负责提供资产，作品负责产出四件套。**

再展开就是：

- **商品**：我从哪里来
- **素材**：我手上有什么可用
- **作品**：我最终要交付什么

如果以后要让系统更省输入，就优化：

- “如何从商品快速初始化作品”

而不是让：

- 商品对象
- 素材对象
- 作品对象

三者继续互相代替。

---

## 11. 最终建议（结论版）

如果按你的目标来收敛，我的建议是：

1. **作品继续作为核心业务对象**
2. **商品定位为来源对象 / 导入对象**
3. **素材定位为可复用资产对象**
4. **文案在作品层成为真值，在素材层只保留候选形态**
5. **话题从“核心素材”降级为“文案生成输入 + 发布标签候选”**
6. **得物链接导入做成“商品导入 + 作品草稿初始化”的 bootstrap 流程**
7. **作品详情页优先展示四件套真值，再展示输入素材与来源信息**

如果一句话总结：

> **不要再从“系统里有哪些输入项”定义作品，而要从“作品最终要交付哪四件东西”来反推作品模型。**
